## 什么是“红灯测试”？

**红灯测试**通常来自 TDD 的 **Red-Green-Refactor** 工作流：

|阶段|含义|目的|
|---|---|---|
|**Red / 红灯**|先写一个会失败的测试|证明当前功能确实还不存在，或 bug 确实能被复现|
|**Green / 绿灯**|写最少的实现代码，让测试通过|证明功能被修好或实现了|
|**Refactor / 重构**|在测试仍然通过的前提下整理代码|改善结构，不改变行为|

在 TDD 里，**red 表示 failing test，green 表示 passing tests**。也就是说，“红灯测试”不是坏事，而是一个**预期失败的测试**，用于锁定目标行为。([The GDS Way](https://gds-way.digital.cabinet-office.gov.uk/standards/test-driven-development.html?utm_source=chatgpt.com "Test-driven development (TDD) - The GDS Way"))

---

## 为什么 Codex 经常提到“红灯测试”？

因为 Codex 写项目时，经常会按这个节奏工作：

```text
1. 先写测试：证明现在代码不满足需求
2. 运行测试：测试失败，进入红灯
3. 修改业务代码：让测试通过
4. 再运行测试：绿灯
5. 清理代码：保证测试继续绿灯
```

这对 AI 编程尤其重要。

因为没有“红灯测试”，Codex 很容易直接改代码，然后声称“已完成”。但你很难判断：

- 它是不是真的理解了需求；
    
- 它是不是只改了表面；
    
- 它有没有破坏旧功能；
    
- 它的改动是否可回归验证。
    

有了红灯测试，Codex 至少要先证明：

> 当前代码在这个需求/bug 上确实失败。

然后再用代码改动把失败变成通过。

## 举个 Java 后端例子

假设你的项目有一个 `WikiService`，需求是：

> 创建 wiki 时，标题不能为空。

Codex 可能会先写一个红灯测试：

```java
@Test
void createWiki_shouldRejectBlankTitle() {
    CreateWikiRequest request = new CreateWikiRequest();
    request.setTitle("   ");
    request.setContent("some content");

    assertThrows(BusinessException.class, () -> {
        wikiService.createWiki(request);
    });
}
```

如果当前代码没有校验标题为空，那么这个测试会失败：

```text
Expected BusinessException to be thrown, but nothing was thrown.
```

这就是**红灯**。

然后 Codex 再修改业务代码：

```java
public Wiki createWiki(CreateWikiRequest request) {
    if (request.getTitle() == null || request.getTitle().isBlank()) {
        throw new BusinessException("Wiki title must not be blank");
    }

    // 原有创建逻辑
}
```

再次运行测试，通过：

```text
Tests passed
```

这就是**绿灯**。

## 红灯测试的核心价值

### 1. 证明测试不是摆设

如果一个测试一写出来就通过，可能说明：

- 功能本来就已经存在；
    
- 测试写错了；
    
- 断言太弱；
    
- 没有测到真正的行为；
    
- mock 把真实问题绕过去了。
    

红灯测试要求它**先失败一次**，这样才能证明它真的能捕获问题。

---

### 2. 把需求变成“可执行规格”

比如这句话：

> 用户未登录时不能创建文章。

这只是自然语言需求。

但测试可以变成可执行规格：

```java
@Test
void createArticle_shouldRejectAnonymousUser() {
    assertThrows(UnauthorizedException.class, () -> {
        articleService.createArticle(null, request);
    });
}
```

以后任何人改代码，只要破坏这个规则，测试就会失败。

---

### 3. 防止 Codex 过度发挥

AI 写代码常见问题是：  
**需求只要求修一个点，它顺手改一堆结构。**

红灯测试可以把目标收窄：

```text
只要让这个失败测试通过，并保证旧测试不挂。
```

这会迫使 Codex 围绕具体行为修改，而不是大范围“重构式发挥”。

---

## 不是所有失败测试都叫合格的红灯测试

合格的红灯测试应该满足几个条件：

|标准|说明|
|---|---|
|**失败原因明确**|应该因为目标行为未实现而失败，不是因为环境、依赖、数据库连接失败|
|**测试目标单一**|一个测试最好只验证一个业务行为|
|**可重复运行**|本地、CI、Codex 环境里结果应稳定|
|**断言清晰**|不只是“不报错”，而是明确验证返回值、异常、状态变化|
|**尽量测公共接口**|不要过度绑定私有方法和实现细节|

比如这个就不是好红灯测试：

```java
@Test
void testSomething() {
    service.doSomething();
}
```

它没有断言，不知道到底测什么。

更好的写法是：

```java
@Test
void createWiki_shouldSetStatusToDraftByDefault() {
    Wiki wiki = wikiService.createWiki(request);

    assertEquals(WikiStatus.DRAFT, wiki.getStatus());
}
```

## 在 Codex prompt 里怎么使用这个概念？

你可以这样要求 Codex：

```text
请采用 Red-Green-Refactor 的方式处理这个 issue：

1. 先补充一个能复现当前问题的红灯测试；
2. 运行测试，确认该测试在当前代码下失败，且失败原因符合预期；
3. 再修改生产代码，让该测试通过；
4. 最后运行相关测试和全量测试，确认没有破坏已有功能；
5. 在回复中说明：
   - 新增了哪个红灯测试；
   - 它最初为什么失败；
   - 修改了哪些代码使其通过；
   - 最终执行了哪些测试命令。
```

## 简单总结

**红灯测试 = 先写一个预期失败的测试。**

它的意义不是“测试挂了”，而是：

> 用一个失败测试证明问题真实存在，再通过代码修改让它变绿。

在 Codex 场景下，它尤其有价值，因为它能约束 AI 不要凭感觉改代码，而是围绕**可验证行为**完成开发。
