**类装饰器**是Python里一个承上启下的关键概念，它把"装饰器"和"类"这两个你刚学的东西结合起来了。

用Java思维来拆解——**类装饰器本质上就是一个"可以接收参数并返回增强版对象"的工厂**。

---

## 1. 一句话定义（对照Java思维）

**类装饰器**：一个**可调用的类**，它接收一个函数或类作为参数，返回一个增强后的版本。

- **Java对照**：类似于**动态代理（Proxy）** + **工厂模式（Factory）** 的结合体。
- **核心区别**：Java的代理需要实现接口、写InvocationHandler，Python的类装饰器只需一个`__call__`方法。

---

## 2. 先看函数装饰器（温习一下）

```python
# 函数装饰器：用函数来实现
def timer(func):
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        print(f"耗时：{time.time() - start}s")
        return result
    return wrapper

@timer
def train_model():
    time.sleep(1)
    
# 等价于：train_model = timer(train_model)
```

**Java对照**：类似于方法拦截器（AOP），但Python用闭包实现，更轻量。

---

## 3. 类装饰器：把函数装饰器翻译成类

```python
class TimerDecorator:
    """类装饰器：统计函数执行时间"""
    def __init__(self, func):
        # 接收被装饰的函数
        self.func = func
    
    def __call__(self, *args, **kwargs):
        # 让实例可以像函数一样被调用
        start = time.time()
        result = self.func(*args, **kwargs)
        print(f"耗时：{time.time() - start}s")
        return result

@TimerDecorator
def train_model():
    time.sleep(1)

# 等价于：train_model = TimerDecorator(train_model)
# 现在 train_model 是 TimerDecorator 的实例
train_model()  # 调用实例的 __call__ 方法
```

**执行流程拆解**：
1. Python解析到`@TimerDecorator`时，执行`TimerDecorator(train_model)` → 创建`TimerDecorator`实例。
2. 实例的`self.func`保存了原始的`train_model`函数。
3. 当你调用`train_model()`时，实际调用的是实例的`__call__`方法。
4. `__call__`里先计时，再执行`self.func()`，最后打印耗时。

**Java对照**：这就像你写了一个实现`InvocationHandler`的类，然后在`invoke`方法里做增强逻辑。

## 4. 带参数的类装饰器（大模型项目必备）

**场景**：你需要一个带配置的装饰器，比如"重试3次"或"只允许GPU执行"。

```python
class RetryDecorator:
    """带参数的类装饰器：自动重试"""
    def __init__(self, max_retries=3):
        # 这里接收装饰器的参数
        self.max_retries = max_retries
    
    def __call__(self, func):
        # 这里接收被装饰的函数
        def wrapper(*args, **kwargs):
            for attempt in range(self.max_retries):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    print(f"第{attempt+1}次失败：{e}")
                    if attempt == self.max_retries - 1:
                        raise
            return None
        return wrapper

# 使用：@RetryDecorator(max_retries=5)  # 带参数
@RetryDecorator(max_retries=5)
def call_api(prompt):
    # 模拟调用大模型API可能失败
    import random
    if random.random() < 0.7:
        raise ConnectionError("API超时")
    return "模型返回结果"

# 等价于：
# call_api = RetryDecorator(max_retries=5)(call_api)
```

**执行流程（对照Java的工厂模式）**：
1. `@RetryDecorator(max_retries=5)` 先执行 `RetryDecorator.__init__(max_retries=5)` → 创建配置好的实例。
2. 然后这个实例被调用 `__call__(call_api)` → 返回`wrapper`函数。
3. 最终`call_api`被替换为`wrapper`。

**Java对照**：这就像你写一个`RetryProxyFactory`，构造函数接收`maxRetries`，`createProxy()`方法返回增强后的代理对象。

## 5. AI项目里的真实场景：类装饰器实战

### 场景1：自动选择设备（CPU/GPU）

```python
class DeviceDecorator:
    """自动把张量移到指定设备"""
    def __init__(self, device="cuda"):
        self.device = device
    
    def __call__(self, func):
        def wrapper(*args, **kwargs):
            # 把所有参数中的Tensor移到指定设备
            new_args = []
            for arg in args:
                if hasattr(arg, "to"):
                    new_args.append(arg.to(self.device))
                else:
                    new_args.append(arg)
            
            result = func(*new_args, **kwargs)
            # 返回值也移到指定设备
            if hasattr(result, "to"):
                return result.to(self.device)
            return result
        return wrapper

@DeviceDecorator("cuda")
def process_tensor(tensor):
    return tensor * 2

# 调用时会自动移到GPU
t = torch.tensor([1,2,3])
result = process_tensor(t)  # 自动在GPU上计算
```

### 场景2：记录训练日志

```python
class LogDecorator:
    """自动记录函数的输入输出"""
    def __init__(self, log_file="training.log"):
        self.log_file = log_file
    
    def __call__(self, func):
        def wrapper(*args, **kwargs):
            with open(self.log_file, "a") as f:
                f.write(f"调用函数：{func.__name__}\n")
                f.write(f"参数：{args}, {kwargs}\n")
            
            result = func(*args, **kwargs)
            
            with open(self.log_file, "a") as f:
                f.write(f"返回值：{result}\n")
                f.write("-" * 50 + "\n")
            return result
        return wrapper

@LogDecorator("train.log")
def train_epoch(epoch, lr=0.001):
    return {"loss": 0.5, "acc": 0.9}
```

## 6. 类装饰器 vs 函数装饰器（选哪个？）

| 维度 | 函数装饰器 | 类装饰器 |
| :--- | :--- | :--- |
| **简洁性** | ✅ 代码少，适合简单场景 | ❌ 代码多，但结构清晰 |
| **状态管理** | ❌ 需要用`nonlocal`或闭包变量 | ✅ 直接用`self.xxx`存状态 |
| **参数传递** | 需要两层嵌套函数 | 通过`__init__`和`__call__`分离 |
| **可读性** | 嵌套多了难懂 | ✅ 结构扁平，易扩展 |
| **适用场景** | 无状态的增强（如计时） | 有状态的增强（如重试计数、缓存） |

**实战建议**：
- **简单场景**（如计时、打印日志）→ 用函数装饰器。
- **复杂场景**（如需要配置、需要保存状态）→ 用类装饰器。
- **大模型项目里**：类装饰器更常见，因为你需要传递模型配置、设备、随机种子等一堆参数。

## 7. 转行必踩的坑（Java思维作祟）

### 坑一：以为装饰器是"继承"的替代
```python
# ❌ 错误：想通过装饰器"继承"方法
@TimerDecorator
class MyModel:
    def forward(self):
        pass
# 这不会让MyModel有计时功能，只是把MyModel这个类本身传给了TimerDecorator

# ✅ 正确：装饰器作用于类的某个方法
class MyModel:
    @TimerDecorator
    def forward(self):
        pass
```

### 坑二：忘了类装饰器返回的是实例，不是函数
```python
@TimerDecorator
def train():
    pass

print(type(train))  # <class '__main__.TimerDecorator'> 不是function！
# 所以 train.__name__ 会报错（TimerDecorator没有这个属性）
```

**解决方案**：在类装饰器里用`functools.wraps`保留原函数信息：
```python
from functools import wraps

class TimerDecorator:
    def __init__(self, func):
        self.func = func
        wraps(func)(self)  # 复制原函数的元信息
    
    def __call__(self, *args, **kwargs):
        # ...
```

### 坑三：装饰器执行时机搞混
```python
@RetryDecorator(max_retries=5)  # ← 这行在import时立即执行！
def call_api():
    pass
```

**关键认知**：装饰器的`__init__`和`__call__`在**定义函数时**就会执行，不是调用函数时才执行！这跟Java的注解完全不同（Java注解在编译期或运行时才处理）。

## 8. 终极对比：Python类装饰器 vs Java注解

| 维度 | Python 类装饰器 | Java 注解 |
| :--- | :--- | :--- |
| **执行时机** | **运行时**：定义函数时立即执行 | **编译期/运行时**：通过反射或APT处理 |
| **修改能力** | ✅ 直接修改函数行为（替换原函数） | ❌ 本身不修改代码，需要配合处理器 |
| **传递参数** | 通过`__init__`传递 | 通过注解属性传递 |
| **复杂度** | 简单，几行代码搞定 | 需要定义注解、处理器、反射等一堆东西 |
| **灵活性** | ✅ 极高，可以任意修改 | ❌ 受限于注解处理器框架 |

**结论**：Python的装饰器比Java的注解**强大得多**，因为它直接操作函数对象，而Java注解只是"元数据"，需要额外工具才能发挥作用。

## 🎯 给你的实战建议

1. **先用函数装饰器**：简单场景（计时、日志）用函数装饰器足够了。
2. **需要状态时切到类装饰器**：比如要统计调用次数、维护缓存、配置重试策略。
3. **看源码时认准 `__call__`**：遇到 `@ClassName` 的写法，去类里找 `__call__` 方法，那就是核心逻辑。
4. **别忘了 `functools.wraps`**：否则调试时看到的是装饰器类名，不是原函数名。

## 📝 快速自测

下面这个类装饰器实现了什么功能？

```python
class CacheDecorator:
    def __init__(self, max_size=128):
        self.cache = {}
        self.max_size = max_size
    
    def __call__(self, func):
        def wrapper(*args, **kwargs):
            key = (args, tuple(kwargs.items()))
            if key in self.cache:
                print("命中缓存")
                return self.cache[key]
            result = func(*args, **kwargs)
            if len(self.cache) >= self.max_size:
                self.cache.pop(next(iter(self.cache)))
            self.cache[key] = result
            return result
        return wrapper
```

**答案**：这是一个带LRU（最近最少使用）淘汰策略的缓存装饰器，用于缓存函数的计算结果，避免重复计算。
