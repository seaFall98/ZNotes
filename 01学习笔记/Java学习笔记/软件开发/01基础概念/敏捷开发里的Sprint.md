## 结论

**Sprint 是敏捷开发里的一种短周期迭代。**

可以简单理解为：

> 团队在一个固定时间盒里，集中完成一批明确目标的开发、测试、交付工作。

常见周期是：

- 1 周
    
- 2 周
    
- 3 周
    
- 4 周
    

实际项目里最常见的是 **2 周一个 Sprint**。

---

## 1. Sprint 和敏捷开发是什么关系？

Sprint 是 **Scrum 敏捷框架**里的核心概念。

敏捷开发是一个大方向，强调：

- 小步快跑
    
- 快速反馈
    
- 持续交付
    
- 拥抱变化
    
- 不要一开始把所有需求都设计死
    

而 **Scrum** 是敏捷开发的一种具体落地方法。

在 Scrum 里，一个项目不是一次性做完，而是拆成很多个 Sprint：

```text
Product Backlog
      ↓
Sprint 1：做一批功能
      ↓
Sprint 2：继续做一批功能
      ↓
Sprint 3：继续迭代
      ↓
持续交付、持续改进
```

所以关系是：

```text
敏捷开发 Agile
  └── Scrum
        └── Sprint
```

---

## 2. Sprint 里具体做什么？

一个 Sprint 通常包含这些环节：

|阶段|做什么|
|---|---|
|Sprint Planning|计划本轮要做哪些需求|
|Development|开发功能|
|Testing|测试、修 bug|
|Daily Scrum|每日站会，同步进度和阻塞|
|Sprint Review|演示本轮成果|
|Sprint Retrospective|复盘本轮过程，改进协作方式|

比如一个博客项目，两周 Sprint 可以这样安排：

```text
Sprint 目标：完成文章发布功能

本轮任务：
1. 后端文章 CRUD 接口
2. 前端文章编辑页
3. Markdown 编辑器接入
4. 草稿 / 发布状态
5. 基础测试
6. 本地联调验收
```

到 Sprint 结束时，不是只写了一堆代码，而是应该交付一个**可演示、可验证、尽量可上线的增量功能**。

---

## 3. Sprint 的关键点：时间盒

Sprint 不是“做到完为止”，而是有固定周期的。

例如：

```text
Sprint 周期：2 周
开始时间：周一
结束时间：下下周五
```

这期间团队尽量保持目标稳定，不频繁插需求。

如果中途总是插新需求，Sprint 就会失去意义，因为团队无法判断：

- 本轮目标是否完成
    
- 计划是否合理
    
- 估算是否准确
    
- 交付节奏是否稳定
    

---

## 4. Sprint 和普通开发计划有什么区别？

普通开发计划常见是：

```text
这个功能尽快做完。
```

Sprint 更强调：

```text
在接下来的两周内，我们承诺完成这批明确的任务，并在结束时交付一个可验证结果。
```

差异如下：

|对比项|普通开发计划|Sprint|
|---|---|---|
|时间|不一定固定|固定时间盒|
|目标|可能模糊|Sprint Goal 明确|
|任务范围|容易不断变化|Sprint 内尽量稳定|
|交付结果|可能只是代码|应该是可验证增量|
|反馈机制|不固定|Review + Retrospective|

---

## 5. 放到你的 AI Coding 工作流里怎么理解？

你现在让 Codex / Claude Code 做项目，也可以借鉴 Sprint 思路。

但可以缩小成 **AI Coding Batch / Mini Sprint**。

例如：

```text
Mini Sprint：文章模块后端重构

目标：
把文章模块从 JDBC 直连数据库重构为 MyBatis 或 MyBatis-Plus，并保证原有接口行为不变。

范围：
1. 评估现有 JDBC 代码
2. 设计 Mapper / Repository 分层
3. 替换数据库访问实现
4. 补充必要测试
5. 更新架构文档
6. 输出重构复盘

验收：
1. 后端单测通过
2. 接口联调通过
3. 原有功能不回退
4. 不出现假实现
```

这比直接说“优化一下后端代码”更适合 AI 执行。

---

## 6. 一句话总结

**Sprint 是敏捷开发中固定周期的迭代开发单元。**

它的核心不是“赶进度”，而是：

> 用固定时间、明确目标、可验证交付、持续复盘的方式，把复杂项目拆成一轮一轮可控的工程推进过程。

对你现在的 AI Coding 项目来说，Sprint 思路很有价值，可以用来管理 Codex / Claude Code 的开发节奏、验收边界和复盘文档。
