[Event_Storming](https://zh.wikipedia.org/wiki/%E4%BA%8B%E4%BB%B6%E9%A2%A8%E6%9A%B4)

# 一句话理解事件风暴

**事件风暴 Event Storming，就是用“领域事件”反推业务流程、业务规则、系统边界和领域模型的一种 DDD 协作建模方法。**

它不是画 UML，也不是先设计数据库，而是把开发、产品、业务专家拉到一起，用便利贴把业务中“已经发生的事实”贴出来，再逐步推导出：

```text
谁触发了什么命令
命令改变了哪个聚合
聚合产生了什么领域事件
事件又引发了哪些后续流程
哪些地方属于同一个限界上下文
```

Wiki 里说得比较核心：事件风暴是在 DDD 语境下由 Alberto Brandolini 提出，用研讨方式快速发现特定领域中发生的事情；它强调让开发者和领域专家一起讨论学习，并把业务流程表达成一连串领域事件。([维基百科](https://zh.wikipedia.org/wiki/%E4%BA%8B%E4%BB%B6%E9%A2%A8%E6%9A%B4 "事件風暴 - 维基百科，自由的百科全书"))

---

# 1. 为什么叫“事件风暴”？

重点在两个词：

```text
事件：业务中已经发生的事实
风暴：像头脑风暴一样，快速、大量、协作式地贴出来
```

比如电商下单流程里，不要一上来就说：

```text
我要设计 order 表、order_item 表、pay_order 表
```

而是先说：

```text
订单已创建
库存已锁定
优惠券已核销
支付已发起
支付已成功
订单已支付
发货单已生成
用户已收到通知
```

这些都是**已经发生的业务事实**，也就是领域事件。

DDD 里最容易犯的错误是：  
**从数据库表开始建模。**

事件风暴正好反过来：  
**从业务事实开始建模。**

---

# 2. 事件风暴在 DDD 里到底解决什么问题？

DDD 的难点不是写代码，而是这几个问题：

```text
业务边界在哪里？
核心流程是什么？
哪些对象应该成为聚合？
哪些规则属于领域模型？
哪些动作只是应用层编排？
哪些系统是外部依赖？
哪些模型名词需要统一？
```

事件风暴就是用来把这些问题“摊开在墙上”。

Wiki 里提到，事件风暴可以作为业务流程建模和需求工程工具，过程通常不需要电脑，只需要便利贴和一面大墙。([维基百科](https://zh.wikipedia.org/wiki/%E4%BA%8B%E4%BB%B6%E9%A2%A8%E6%9A%B4 "事件風暴 - 维基百科，自由的百科全书"))

这点很关键。

它不是一个技术框架，而是一个**协作建模方法**。

---

# 3. 事件风暴的核心元素

Wiki 里列了几个核心元素：领域事件、人员 Actor、命令 Command、聚合 Aggregate、View、业务流程、外部系统等。([维基百科](https://zh.wikipedia.org/wiki/%E4%BA%8B%E4%BB%B6%E9%A2%A8%E6%9A%B4 "事件風暴 - 维基百科，自由的百科全书"))

放到 DDD 里，可以这样理解：

|元素|DDD 含义|示例|
|---|---|---|
|领域事件 Domain Event|业务中已经发生的事实|`OrderCreated`、`PaymentSucceeded`|
|命令 Command|用户或系统想要做的动作|`CreateOrderCommand`、`PayOrderCommand`|
|Actor|发起动作的人或系统|用户、管理员、定时任务、消息消费者|
|View|用户操作入口|结算页、订单详情页、后台审核页|
|聚合 Aggregate|执行业务规则并保证一致性的领域对象|`Order`、`Payment`、`Inventory`|
|外部系统 External System|非本领域控制的系统|支付平台、物流系统、短信平台|
|Policy / Process|事件发生后的业务策略或流程编排|支付成功后发优惠券、发送通知|

# 4. 标准推导链路：Event → Command → Aggregate

Wiki 里给出的基本步骤是：

```text
第一步：找到所有领域事件
第二步：找到导致事件发生的命令
第三步：识别命令执行和事件发生对应的聚合
```

也就是：

```text
Command → Aggregate → Domain Event
```

例如：

```text
用户点击“提交订单”
        ↓
CreateOrderCommand
        ↓
Order 聚合执行 create()
        ↓
OrderCreatedEvent
```

再比如：

```text
用户点击“支付”
        ↓
PayOrderCommand
        ↓
Order 聚合执行 pay()
        ↓
OrderPaidEvent
```

这条链路非常 DDD。

因为它逼你思考：

```text
这个命令到底应该由哪个聚合处理？
这个聚合内部有什么业务规则？
命令成功后会产生什么业务事实？
```

而不是直接写：

```java
order.setStatus(PAID);
orderMapper.update(order);
```

# 5. 用电商订单系统走一遍事件风暴

假设我们要建模一个“用户下单并支付”的业务。

## 第一步：先贴领域事件

只关心“发生了什么”：

```text
商品已加入购物车
订单已创建
库存已锁定
优惠券已使用
支付单已创建
支付已成功
订单已支付
库存已扣减
发货单已创建
用户已收到支付成功通知
```

注意命名方式：  
领域事件通常用**过去式**，因为它表示已经发生的事实。Wiki 里也提到，领域事件如果用英文书写，一般用过去式表示。([维基百科](https://zh.wikipedia.org/wiki/%E4%BA%8B%E4%BB%B6%E9%A2%A8%E6%9A%B4 "事件風暴 - 维基百科，自由的百科全书"))

---

## 第二步：反推命令

每个事件前面通常有一个命令：

```text
CreateOrderCommand  →  OrderCreated
LockInventoryCommand → InventoryLocked
UseCouponCommand → CouponUsed
CreatePaymentCommand → PaymentCreated
ConfirmPaymentCommand → PaymentSucceeded
MarkOrderPaidCommand → OrderPaid
```

命令是“想做什么”：

```text
创建订单
锁定库存
使用优惠券
发起支付
确认支付
标记订单已支付
```

事件是“已经发生了什么”：

```text
订单已创建
库存已锁定
优惠券已使用
支付已成功
订单已支付
```

这两个不要混。

---

## 第三步：识别聚合

继续问：

```text
哪个领域对象负责处理这个命令？
哪个对象需要保证业务一致性？
```

得到：

```text
Order 聚合
- 处理 CreateOrderCommand
- 产生 OrderCreatedEvent
- 处理 MarkOrderPaidCommand
- 产生 OrderPaidEvent

Inventory 聚合
- 处理 LockInventoryCommand
- 产生 InventoryLockedEvent

Coupon 聚合
- 处理 UseCouponCommand
- 产生 CouponUsedEvent

Payment 聚合
- 处理 CreatePaymentCommand
- 产生 PaymentCreatedEvent
- 处理 ConfirmPaymentCommand
- 产生 PaymentSucceededEvent
```

这时你就开始看到 DDD 的模型了。

不是：

```text
order 表
payment 表
inventory 表
coupon 表
```

而是：

```text
Order 聚合
Payment 聚合
Inventory 聚合
Coupon 聚合
```

# 6. 事件风暴如何帮助识别限界上下文？

这才是事件风暴最有价值的地方。

一开始你可能把所有事件贴成一条大流程：

```text
订单已创建
库存已锁定
优惠券已使用
支付已成功
订单已支付
发货单已创建
```

但贴着贴着你会发现：

```text
订单相关语言是一组
库存相关语言是一组
支付相关语言是一组
营销优惠相关语言是一组
物流发货相关语言是一组
```

于是自然拆出限界上下文：

```text
Order Context
Inventory Context
Payment Context
Promotion Context
Logistics Context
```

这就是 DDD 里的一个核心判断：

> **限界上下文不是按数据库表拆的，而是按业务语言、业务规则、模型边界拆的。**

例如：

|限界上下文|关心的事件|核心模型|
|---|---|---|
|订单上下文|`OrderCreated`、`OrderPaid`、`OrderCancelled`|`Order`|
|支付上下文|`PaymentCreated`、`PaymentSucceeded`、`PaymentFailed`|`Payment`|
|库存上下文|`InventoryLocked`、`InventoryDeducted`、`InventoryReleased`|`Stock` / `Inventory`|
|营销上下文|`CouponUsed`、`CouponReleased`|`Coupon`|
|物流上下文|`ShipmentCreated`、`PackageDelivered`|`Shipment`|

事件风暴能让你从流程中发现边界。

# 7. 事件风暴和领域事件的关系

你之前问过“领域事件”，这里可以接上。

**领域事件是事件风暴的核心材料。**

但二者不是一个东西：

|概念|含义|
|---|---|
|领域事件|DDD 中的一个建模概念，表示领域中已经发生的重要事实|
|事件风暴|一种协作建模方法，用大量领域事件来探索业务流程和模型边界|

换句话说：

```text
领域事件是砖
事件风暴是用砖搭业务地图的方法
```

在代码里，领域事件可能长这样：

```java
public record OrderPaidEvent(
    OrderId orderId,
    UserId userId,
    Money paidAmount,
    LocalDateTime paidAt
) {}
```

但在事件风暴现场，它一开始只是橘色便利贴上的一句话：

```text
订单已支付
```

先有业务语言，再有代码模型。

# 8. 事件风暴和聚合设计的关系

事件风暴对聚合设计特别有帮助。

因为聚合不是拍脑袋设计出来的，而是从这些问题里推出来的：

```text
这个命令修改了什么对象？
这个对象要保护哪些不变量？
哪些数据必须在一个事务里保持一致？
哪个对象是业务规则入口？
事件应该由谁产生？
```

例如：

```text
PayOrderCommand → Order.pay() → OrderPaidEvent
```

说明 `Order` 很可能是一个聚合根。

`Order.pay()` 内部可以保护规则：

```java
public void pay(PaymentId paymentId, Money amount) {
    if (this.status != OrderStatus.CREATED) {
        throw new BusinessException("只有待支付订单可以支付");
    }

    if (!this.totalAmount.equals(amount)) {
        throw new BusinessException("支付金额不一致");
    }

    this.paymentId = paymentId;
    this.status = OrderStatus.PAID;
    this.registerEvent(new OrderPaidEvent(this.id, paymentId, amount));
}
```

这就是事件风暴对代码落地的帮助：

```text
便利贴上的事件
    ↓
领域模型的方法
    ↓
领域事件对象
    ↓
应用层编排
    ↓
消息发布 / 最终一致性
```

# 9. 事件风暴和应用层编排的关系

很多初学者会误会：

> 事件风暴贴出来的流程，是不是都要写进领域模型？

不是。

事件风暴会暴露出两类逻辑：

## 一类是领域规则

应该放进聚合或领域服务：

```text
订单金额必须大于 0
已取消订单不能支付
支付金额必须等于订单金额
库存不足不能锁定
优惠券过期不能使用
```

这些是业务规则。

## 另一类是流程编排

应该放在应用层、Saga、Process Manager 或消息消费者里：

```text
订单创建后锁库存
库存锁定后创建支付单
支付成功后标记订单已支付
订单已支付后通知物流
订单已支付后发送短信
```

这类不是单个聚合内部规则，而是跨上下文流程。

在代码里可能是：

```java
public class PayOrderUseCase {

    public void handle(PayOrderCommand command) {
        Payment payment = paymentService.pay(command);
        
        Order order = orderRepository.get(command.orderId());
        order.markPaid(payment.getId(), payment.getAmount());
        
        orderRepository.save(order);
        eventPublisher.publish(order.domainEvents());
    }
}
```

或者事件驱动：

```text
PaymentSucceededEvent
        ↓
OrderPaymentEventHandler
        ↓
MarkOrderPaidCommand
        ↓
Order.markPaid()
        ↓
OrderPaidEvent
```

# 10. 事件风暴和 MVC 重构 DDD 的关系

如果你现在有一个 MVC 项目，代码大概率是这样的：

```java
OrderController
    ↓
OrderService
    ↓
OrderMapper
    ↓
order 表
```

`OrderService` 里堆满：

```text
校验库存
校验优惠券
计算金额
创建订单
调用支付
发短信
改状态
写日志
```

事件风暴可以帮你把这坨逻辑拆开。

原来的 MVC 逻辑：

```text
submitOrder()
```

通过事件风暴拆成：

```text
Command:
- CreateOrderCommand
- LockInventoryCommand
- UseCouponCommand
- CreatePaymentCommand

Aggregate:
- Order
- Inventory
- Coupon
- Payment

Domain Event:
- OrderCreatedEvent
- InventoryLockedEvent
- CouponUsedEvent
- PaymentCreatedEvent
```

然后代码结构就会从：

```text
controller
service
mapper
entity
```

逐步变成：

```text
interfaces
  OrderController

application
  CreateOrderUseCase
  PayOrderUseCase

domain
  model
    Order
    OrderItem
    OrderStatus
  event
    OrderCreatedEvent
    OrderPaidEvent
  repository
    OrderRepository

infrastructure
  persistence
    OrderPO
    OrderMapper
    OrderRepositoryImpl
  external
    PaymentClient
    InventoryClient
```

这就是你前面学 DDD 工程模型时反复强调的：

```text
Controller 不直接写业务规则
Application 负责编排
Domain 负责业务规则
Infrastructure 负责数据库和外部系统
```

# 11. 事件风暴的颜色不是重点，语义才是重点

Wiki 里提到领域事件、命令、聚合等会用不同颜色便利贴表示，例如领域事件常用橘色便利贴。([维基百科](https://zh.wikipedia.org/wiki/%E4%BA%8B%E4%BB%B6%E9%A2%A8%E6%9A%B4 "事件風暴 - 维基百科，自由的百科全书"))

但颜色只是辅助。

真正重要的是这套语义关系：

```text
Actor 在 View 上发起 Command
Command 作用于 Aggregate
Aggregate 执行业务规则
Aggregate 产生 Domain Event
Domain Event 触发后续 Policy / Process
Policy 可能调用外部系统或发起新的 Command
```

可以抽象成：

```text
Actor
  ↓
View
  ↓
Command
  ↓
Aggregate
  ↓
Domain Event
  ↓
Policy / Process
  ↓
Command / External System
```

这张图背下来，事件风暴基本就入门了。

# 12. 一个完整小例子：订单支付

```text
用户
  ↓
订单详情页
  ↓
PayOrderCommand
  ↓
Order 聚合
  ↓
OrderPaidEvent
  ↓
后续策略：
    - 通知库存上下文扣减库存
    - 通知物流上下文创建发货单
    - 通知营销上下文发放积分
    - 通知消息系统发送短信
```

对应代码思路：

```java
public class Order {

    private OrderId id;
    private OrderStatus status;
    private Money totalAmount;
    private final List<DomainEvent> domainEvents = new ArrayList<>();

    public void pay(PaymentId paymentId, Money paidAmount) {
        if (status != OrderStatus.CREATED) {
            throw new BusinessException("订单状态不允许支付");
        }

        if (!totalAmount.equals(paidAmount)) {
            throw new BusinessException("支付金额不正确");
        }

        this.status = OrderStatus.PAID;
        this.domainEvents.add(new OrderPaidEvent(id, paymentId, paidAmount));
    }
}
```

这里有几个 DDD 要点：

```text
Order 是聚合根
pay() 是领域行为
状态变更不是裸 setStatus()
OrderPaidEvent 是领域事件
后续动作不要全部塞进 Order 里
```

# 13. 事件风暴最容易踩的坑

## 坑 1：把事件写成命令

错误：

```text
创建订单
支付订单
取消订单
```

这些是命令，不是事件。

正确：

```text
订单已创建
订单已支付
订单已取消
```

---

## 坑 2：把技术动作当领域事件

错误：

```text
数据库已更新
接口已调用
缓存已删除
MQ 消息已发送
```

这些是技术事件，不是领域事件。

正确：

```text
订单已支付
库存已扣减
优惠券已核销
发货单已创建
```

领域事件必须是业务人员听得懂的语言。

---

## 坑 3：一上来就设计表

事件风暴阶段不要急着说：

```text
order 表字段有哪些？
payment 表怎么设计？
要不要加索引？
```

这些是后面的事情。

事件风暴阶段先解决：

```text
业务流程是什么？
业务事实是什么？
谁触发了动作？
哪个聚合承担规则？
边界在哪里？
```

---

## 坑 4：把事件风暴当成流程图

事件风暴不是普通流程图。

普通流程图关注：

```text
步骤 A → 步骤 B → 步骤 C
```

事件风暴更关注：

```text
事实、命令、规则、聚合、上下文、外部系统、业务语言
```

它的目标不是把流程画漂亮，而是让团队形成共同理解。

Wiki 里也提到，事件风暴后的结果可以展示商业流程，但更重要的是参与者心中建立的知识。([维基百科](https://zh.wikipedia.org/wiki/%E4%BA%8B%E4%BB%B6%E9%A2%A8%E6%9A%B4 "事件風暴 - 维基百科，自由的百科全书"))

这句话很关键。

事件风暴的核心产物不是墙上的便利贴，而是团队对业务模型达成的共识。

# 14. 事件风暴的最终产出是什么？

一次好的事件风暴，至少应该产出这些东西：

```text
1. 关键业务流程
2. 领域事件清单
3. 命令清单
4. 聚合候选清单
5. 限界上下文候选边界
6. 外部系统依赖
7. 业务规则和异常场景
8. 跨上下文协作方式
9. 统一语言词汇表
10. 后续代码建模方向
```

对于 Java 后端落地，进一步可以转成：

```text
Command 对象
Application UseCase
Aggregate Root
Entity
Value Object
Domain Event
Domain Service
Repository Interface
Infrastructure Adapter
Message Handler
```

# 15. 最后给你一个判断标准

学 DDD 时，事件风暴的价值可以这样判断：

> **如果你不知道怎么拆领域模型，就先做事件风暴。**

因为它会逼你从业务语言出发，而不是从技术分层出发。

更准确地说：

```text
事件风暴不是 DDD 的全部，
但它是从业务理解走向 DDD 建模的一座桥。
```

它把抽象的 DDD 概念串起来：

```text
领域事件 → 命令 → 聚合 → 业务规则 → 限界上下文 → 上下文协作 → 代码结构
```

对于你现在学习的路径，可以把事件风暴理解成：

```text
DDD 概念理论：
    实体、值对象、聚合、领域服务、领域事件、限界上下文

事件风暴：
    用一套协作方法，把这些概念从真实业务里“挖出来”

DDD 工程实现：
    把挖出来的模型落到 Java 代码结构里
```
