
[xfg基础版](https://bugstack.cn/md/road-map/connection-pool.html)
# 结论先行：连接池不是“加速 SQL 的工具”，而是“控制数据库连接资源的基础设施”

数据库连接池的核心价值不是让慢 SQL 变快，而是：

1. **复用昂贵的数据库连接**
    
2. **限制应用对数据库的并发连接数**
    
3. **把连接创建、检测、回收、等待、超时这些问题标准化**
    
4. **让数据库连接从“随用随建”变成“受控资源”**
    

在 Java / Spring Boot / MySQL 项目里，连接池是数据库访问链路中的**必选基础设施**。Spring Boot 的 JDBC starter 默认使用 HikariCP 作为连接池实现，HikariCP 官方也将其定位为轻量、高性能、生产可用的 JDBC 连接池。([Home](https://docs.spring.io/spring-boot/docs/2.4.0-M1/reference/htmlsingle/ "Spring Boot Reference Documentation"))

# 一、先建立整体认知

## 1. 什么是数据库连接池？

数据库连接池就是在应用启动或运行过程中，提前创建并维护一组数据库连接。

业务线程需要访问数据库时：

```text
业务线程
  ↓
DataSource.getConnection()
  ↓
从连接池借一个 Connection
  ↓
执行 SQL
  ↓
connection.close()
  ↓
连接归还连接池，而不是真正关闭物理连接
```

本质上，连接池管理的是一批**可复用的物理数据库连接**。

---

## 2. 为什么不能每次请求都新建数据库连接？

因为数据库连接不是一个普通 Java 对象。

一次数据库连接创建通常涉及：

```text
TCP 三次握手
  ↓
MySQL 协议握手
  ↓
账号密码认证
  ↓
权限校验
  ↓
初始化 session 状态
  ↓
可能还有 SSL/TLS 握手
  ↓
MySQL 服务端分配线程/会话资源
```

如果每个 HTTP 请求都创建一个数据库连接，会出现几个问题：

|问题|后果|
|---|---|
|连接创建成本高|接口 RT 增加|
|高频创建和销毁连接|MySQL 线程、内存、网络资源抖动|
|无连接数上限|应用可能把 MySQL 打爆|
|高并发下大量线程同时建连|雪崩式压力|
|缺少统一监控|生产排查困难|

---

## 3. 数据库连接的创建成本包括哪些？

以 Java 连接 MySQL 为例，成本主要包括：

|成本类型|说明|
|---|---|
|网络成本|TCP 连接建立、网络往返|
|协议成本|MySQL 客户端与服务端握手|
|认证成本|用户名、密码、权限校验|
|服务端资源|MySQL 为连接分配线程、内存、会话状态|
|客户端资源|JDBC Driver 创建对象、维护 socket|
|安全成本|SSL/TLS 连接会更重|
|初始化成本|设置字符集、时区、事务隔离级别、autoCommit 等|

所以，连接池解决的是：

> **不要把连接创建成本放在每次业务请求路径上。**

---

## 4. 连接池解决了哪些核心问题？

|核心问题|连接池怎么解决|
|---|---|
|连接创建成本高|复用连接|
|连接数量不可控|maximumPoolSize 限流|
|高并发抢连接|等待队列 + connectionTimeout|
|连接失效|keepalive / validation|
|连接泄漏|leakDetectionThreshold|
|空闲连接浪费|idleTimeout|
|连接生命周期不可控|maxLifetime|
|生产观测困难|暴露 active、idle、pending、timeout 等指标|

---

## 5. 连接池不解决哪些问题？

这是工程上最容易误判的地方。

连接池**不能解决**：

|问题|原因|
|---|---|
|慢 SQL|SQL 执行慢会长期占用连接|
|缺索引|连接池不参与 SQL 优化|
|锁等待|事务阻塞仍然会占用连接|
|数据库 CPU 高|连接池只能限制并发，不能降低单 SQL 成本|
|事务设计不合理|长事务会占用连接|
|连接数规划错误|多实例场景仍可能把 MySQL 连接打满|
|MySQL 容量不足|连接池不是扩容数据库的替代品|

一句话：

> **连接池解决“连接管理问题”，不解决“数据库执行能力问题”。**

---

# 二、从底层原理讲解

## 1. JDBC 连接的本质是什么？

Java 里的 `java.sql.Connection` 是一个接口。

它背后通常对应：

```text
Connection 代理对象 / 驱动实现对象
  ↓
JDBC Driver
  ↓
Socket / TCP 连接
  ↓
MySQL Server Session
```

所以，一个 JDBC Connection 本质上是：

> **应用进程内的一个数据库会话句柄，背后绑定着 MySQL 服务端的一个会话资源。**

它不是普通对象，也不是无状态对象。

它有状态，例如：

|状态|示例|
|---|---|
|autoCommit|true / false|
|transaction isolation|READ_COMMITTED / REPEATABLE_READ|
|schema|当前数据库|
|session variables|time_zone、sql_mode|
|transaction state|当前事务是否开启|
|prepared statement cache|驱动层可能缓存|

这就是为什么连接归还连接池前，需要被“重置”或确保状态干净。

---

## 2. 一个数据库连接从创建到关闭经历哪些步骤？

不使用连接池时：

```text
DriverManager.getConnection()
  ↓
加载 MySQL Driver
  ↓
创建 TCP 连接
  ↓
MySQL 握手
  ↓
认证
  ↓
创建 Connection 对象
  ↓
执行业务 SQL
  ↓
connection.close()
  ↓
关闭 socket
  ↓
MySQL 释放服务端会话
```

使用连接池时：

```text
应用启动
  ↓
HikariDataSource 初始化
  ↓
按需或预热创建连接
  ↓
业务线程 getConnection()
  ↓
从池中借出连接
  ↓
执行业务 SQL
  ↓
connection.close()
  ↓
归还连接池
  ↓
连接继续保留
  ↓
超过生命周期或失效后再真正关闭
```

区别在于：

|场景|close 行为|
|---|---|
|原生 JDBC|关闭物理连接|
|连接池|归还连接池|
|连接超寿命 / 失效|真正关闭并重建|

---

## 3. 连接池如何复用连接？

连接池内部一般维护两类连接：

```text
Pool
├── idle connections      空闲连接
├── active connections    正在使用的连接
├── waiting threads       等待获取连接的线程
└── housekeeping thread   检测、保活、淘汰连接
```

业务线程获取连接：

```text
有 idle 连接？
  ├── 有：取出，标记 active，返回给业务
  └── 无：
       ├── 当前连接数 < maximumPoolSize：创建新连接
       └── 当前连接数 >= maximumPoolSize：进入等待
```

业务线程归还连接：

```text
connection.close()
  ↓
连接池拦截 close()
  ↓
检查连接状态
  ↓
重置必要状态
  ↓
标记 idle
  ↓
唤醒等待线程
```

---

## 4. 调用 connection.close() 时，连接真的关闭了吗？

**不一定。**

如果你拿到的是连接池返回的连接，`close()` 通常不是物理关闭，而是归还连接。

典型伪代码：

```java
class ProxyConnection implements Connection {

    private final RealConnection realConnection;
    private final Pool pool;

    @Override
    public void close() {
        // 不是 realConnection.close()
        // 而是归还给连接池
        pool.recycle(realConnection);
    }
}
```

所以在连接池场景中：

```java
try (Connection conn = dataSource.getConnection()) {
    // use connection
}
```

这里的 `try-with-resources` 仍然是正确写法。

因为它触发 `close()`，而连接池会把连接回收。

---

## 5. 连接池内部核心组件

### 5.1 空闲连接队列

存放可立即复用的连接。

```text
idle queue:
[conn-1] [conn-2] [conn-3]
```

当业务调用 `getConnection()` 时，优先从这里取。

---

### 5.2 活跃连接集合

记录已经借出去、还没归还的连接。

```text
active set:
[conn-4 used by thread-A]
[conn-5 used by thread-B]
```

如果活跃连接长期不归还，就可能是：

1. 慢 SQL
    
2. 长事务
    
3. 连接泄漏
    
4. 线程阻塞
    
5. 外部接口调用夹在事务中
    

---

### 5.3 最大连接数限制

`maximumPoolSize` 决定连接池最多持有多少连接。

HikariCP 官方说明，`maximumPoolSize` 控制池内最大连接数，包含空闲和正在使用的连接；达到该值且没有空闲连接时，`getConnection()` 会等待，直到超过 `connectionTimeout` 抛出异常。其默认值是 10。([GitHub](https://github.com/brettwooldridge/hikaricp "GitHub - brettwooldridge/HikariCP: 光 HikariCP・A solid, high-performance, JDBC connection pool at last. · GitHub"))

---

### 5.4 连接获取等待队列

当连接池满了：

```text
maximumPoolSize = 10
active = 10
idle = 0
```

第 11 个线程调用 `getConnection()`：

```text
等待其他线程归还连接
  ↓
如果 connectionTimeout 内拿到连接：继续执行
  ↓
否则抛出 SQLTransientConnectionException
```

常见异常：

```text
java.sql.SQLTransientConnectionException:
HikariPool-1 - Connection is not available, request timed out after 30000ms.
```

---

### 5.5 连接检测机制

连接可能失效：

1. MySQL 重启
    
2. 网络断开
    
3. 防火墙清理 idle TCP
    
4. MySQL wait_timeout 到期
    
5. 代理层关闭连接
    

HikariCP 通常使用 JDBC4 `Connection.isValid()` 检测连接。官方建议，如果驱动支持 JDBC4，不要配置 `connectionTestQuery`。([GitHub](https://github.com/brettwooldridge/hikaricp "GitHub - brettwooldridge/HikariCP: 光 HikariCP・A solid, high-performance, JDBC connection pool at last. · GitHub"))

---

### 5.6 连接淘汰机制

连接不能无限存活。

需要被淘汰的情况：

|情况|对应参数|
|---|---|
|空闲太久|idleTimeout|
|生命周期太长|maxLifetime|
|连接不可用|validation 失败|
|连接异常|SQLException 判断|
|池关闭|shutdown|

HikariCP 官方建议 `maxLifetime` 应该比数据库或基础设施施加的连接时间限制短几秒，默认是 30 分钟。([GitHub](https://github.com/brettwooldridge/hikaricp "GitHub - brettwooldridge/HikariCP: 光 HikariCP・A solid, high-performance, JDBC connection pool at last. · GitHub"))

---

# 三、Java 主流连接池对比：HikariCP、Druid、DBCP

## 1. 总体对比

|连接池|设计目标|性能特点|监控能力|Spring Boot 默认|适合场景|
|---|---|---|---|---|---|
|HikariCP|极简、高性能、低开销|非常强|基础指标强，依赖 Micrometer/JMX|是|大多数 Spring Boot 生产项目|
|Druid|连接池 + SQL 监控 + 防火墙|性能可用，但偏重功能|很强，自带监控页、SQL 统计|否|国内项目、需要 SQL 监控、防 SQL 注入墙|
|Apache DBCP|老牌通用连接池|相对传统|一般|否|老项目、兼容性场景|

---

## 2. HikariCP

核心特点：

1. 轻量
    
2. 性能优秀
    
3. 参数克制
    
4. 适合云原生和 Spring Boot
    
5. 不试图做 SQL 监控平台
    

HikariCP 的哲学是：

> 连接池就做好连接池，不把自己变成数据库治理平台。

适合：

1. Spring Boot 微服务
    
2. 高并发接口
    
3. 云原生部署
    
4. 希望用 Micrometer + Prometheus + Grafana 做统一监控的项目
    

---

## 3. Druid

Druid 在国内 Java 项目中很常见。

它的优势是：

1. SQL 监控方便
    
2. 慢 SQL 统计直观
    
3. Web 控制台友好
    
4. 支持 WallFilter
    
5. 适合传统后台管理系统
    

但也要注意：

1. 功能多，配置复杂
    
2. 部分团队会过度依赖 Druid 监控页，而忽略统一可观测体系
    
3. 在 Spring Boot 新项目中，默认优先级通常不如 HikariCP
    

适合：

1. 需要内置 SQL 监控页
    
2. 老系统迁移
    
3. 国内企业内部后台
    
4. DBA / 开发习惯使用 Druid SQL 统计
    

---

## 4. Apache DBCP

DBCP 是老牌连接池。

特点：

1. 稳定
    
2. 历史悠久
    
3. 性能和易用性不如 HikariCP 突出
    
4. 常见于老项目或框架默认遗留场景
    

新项目一般不优先选它。

---

## 5. 选择建议

|场景|建议|
|---|---|
|Spring Boot 新项目|HikariCP|
|微服务 / K8s / 云原生|HikariCP|
|需要统一 Prometheus 监控|HikariCP|
|国内传统管理系统，希望快速看 SQL 统计|Druid|
|老系统已经用了 DBCP|不急着换，除非有明确问题|
|高并发核心链路|HikariCP + SQL 监控平台 + 慢 SQL 治理|

# 四、重点讲解 HikariCP 参数

下面这些默认值以 HikariCP 官方 README 为准。不同 Spring Boot 版本可能有封装差异，但底层 Hikari 默认值可按官方文档理解。

## 1. maximumPoolSize

### 控制什么？

连接池最大连接数。

包括：

```text
active connections + idle connections <= maximumPoolSize
```

官方默认值：`10`。([GitHub](https://github.com/brettwooldridge/hikaricp "GitHub - brettwooldridge/HikariCP: 光 HikariCP・A solid, high-performance, JDBC connection pool at last. · GitHub"))

### 设置过大有什么问题？

1. MySQL 连接数被打满
    
2. 数据库上下文切换增加
    
3. 更多并发 SQL 争抢 CPU / IO / 锁
    
4. 慢 SQL 时放大数据库压力
    
5. 多实例部署时连接数爆炸
    

例如：

```text
单实例 maximumPoolSize = 50
K8s 副本数 = 20
总潜在连接数 = 50 * 20 = 1000
```

如果 MySQL `max_connections = 800`，这就是事故隐患。

MySQL 官方说明，`Too many connections` 表示所有可用连接都被其他客户端占用，而允许连接数由 `max_connections` 控制。([MySQL开发者专区](https://dev.mysql.com/doc/refman/8.2/en/too-many-connections.html "MySQL :: MySQL 8.4 Reference Manual :: B.3.2.5 Too many connections"))

### 设置过小有什么问题？

1. 高峰期业务线程拿不到连接
    
2. 出现 `Connection is not available`
    
3. pending threads 上升
    
4. 接口 RT 增加
    
5. 连接池成为瓶颈
    

### 生产建议值

不要一上来设 100。

常见起点：

|应用类型|建议起点|
|---|---|
|普通后台服务|5～10|
|核心交易服务|10～30|
|高 QPS 读服务|20～50，但必须压测|
|定时任务服务|2～5|
|管理后台|5～10|

更准确的方式是根据压测和数据库指标调整。

### 和其他参数关系

|参数|关系|
|---|---|
|minimumIdle|通常不单独设置，等于 maximumPoolSize，形成固定池|
|connectionTimeout|池满时等待多久|
|MySQL max_connections|所有实例连接总和不能超过它|
|慢 SQL|SQL 越慢，需要连接时间越长，池越容易耗尽|

---

## 2. minimumIdle

### 控制什么？

连接池尽量维持的最小空闲连接数。

官方默认值：与 `maximumPoolSize` 相同。HikariCP 官方建议，为了应对突发流量和获得较好性能，通常不设置该值，让 HikariCP 作为固定大小连接池运行。([GitHub](https://github.com/brettwooldridge/hikaricp "GitHub - brettwooldridge/HikariCP: 光 HikariCP・A solid, high-performance, JDBC connection pool at last. · GitHub"))

### 设置过大有什么问题？

1. 应用启动时创建太多连接
    
2. 空闲连接长期占用 MySQL 资源
    
3. 多实例场景连接浪费严重
    

### 设置过小有什么问题？

1. 突发流量时临时创建连接
    
2. 首批请求 RT 抖动
    
3. 数据库连接创建压力突然上升
    

### 生产建议

两种策略：

|策略|配置|
|---|---|
|固定池|不设置 minimumIdle，让它等于 maximumPoolSize|
|弹性池|minimumIdle < maximumPoolSize|

微服务核心链路更常用固定池。

例如：

```yaml
maximum-pool-size: 20
# 不配置 minimum-idle
```

或者：

```yaml
maximum-pool-size: 20
minimum-idle: 5
```

弹性池适合流量波动大、但不希望长期占用太多连接的业务。

---

## 3. connectionTimeout

### 控制什么？

业务线程从连接池获取连接的最大等待时间。

官方默认值：`30000ms`，最低可接受值是 `250ms`。([GitHub](https://github.com/brettwooldridge/hikaricp "GitHub - brettwooldridge/HikariCP: 光 HikariCP・A solid, high-performance, JDBC connection pool at last. · GitHub"))

### 设置过大有什么问题？

1. 线程长时间阻塞
    
2. 上游接口一直等待
    
3. Tomcat 线程也被拖住
    
4. 故障发现变慢
    
5. 雪崩时堆积更严重
    

### 设置过小有什么问题？

1. 短暂抖动也会报错
    
2. 高峰期误伤正常请求
    
3. 对偶发 GC、网络抖动容忍度低
    

### 生产建议

|场景|建议|
|---|---|
|在线接口|1s～3s|
|普通后台|3s～5s|
|批处理|5s～30s|
|默认值|30s 偏保守，故障暴露慢|

推荐在线接口不要用 30s 默认值。

示例：

```yaml
connection-timeout: 3000
```

### 和其他参数关系

|参数|关系|
|---|---|
|maximumPoolSize|池满后才会等待|
|接口超时时间|connectionTimeout 应小于接口整体超时|
|validationTimeout|必须小于 connectionTimeout|

---

## 4. idleTimeout

### 控制什么？

空闲连接最多保留多久。

官方默认值：`600000ms`，即 10 分钟；只有当 `minimumIdle < maximumPoolSize` 时才主要生效。([GitHub](https://github.com/brettwooldridge/hikaricp "GitHub - brettwooldridge/HikariCP: 光 HikariCP・A solid, high-performance, JDBC connection pool at last. · GitHub"))

### 设置过大有什么问题？

1. 空闲连接占用数据库资源
    
2. 低峰期连接不释放
    
3. 多实例场景浪费明显
    

### 设置过小有什么问题？

1. 连接频繁创建和销毁
    
2. 请求波动时 RT 抖动
    
3. MySQL 连接创建压力增加
    

### 生产建议

如果是固定池：

```yaml
# minimumIdle 默认等于 maximumPoolSize
# idleTimeout 实际意义不大
```

如果是弹性池：

```yaml
idle-timeout: 300000 # 5 分钟
```

常见范围：`5～10 分钟`。

---

## 5. maxLifetime

### 控制什么？

连接在池中的最大生命周期。

官方默认值：`1800000ms`，即 30 分钟。HikariCP 官方建议，该值应该比数据库或基础设施强制连接时间限制短几秒。([GitHub](https://github.com/brettwooldridge/hikaricp "GitHub - brettwooldridge/HikariCP: 光 HikariCP・A solid, high-performance, JDBC connection pool at last. · GitHub"))

### 设置过大有什么问题？

1. MySQL / 代理 / 防火墙先关闭连接
    
2. 应用拿到已经失效的连接
    
3. 出现通信异常
    

常见异常：

```text
Communications link failure
The last packet successfully received from the server was ...
```

### 设置过小有什么问题？

1. 连接频繁重建
    
2. 增加数据库建连压力
    
3. 高峰期连接抖动
    

### 生产建议

核心原则：

```text
maxLifetime < MySQL wait_timeout
maxLifetime < SLB / NAT / Proxy idle timeout
```

例如：

```text
MySQL wait_timeout = 1800s
Hikari maxLifetime = 1700s
```

如果基础设施空闲连接 15 分钟会被清理：

```text
maxLifetime = 14 分钟左右
```

常见值：

```yaml
max-lifetime: 1800000 # 30 分钟，默认
```

或：

```yaml
max-lifetime: 1700000
```

---

## 6. keepaliveTime

### 控制什么？

HikariCP 对空闲连接做保活检测的频率。

官方默认值：`120000ms`，即 2 分钟；必须小于 `maxLifetime`；只会作用于空闲连接。([GitHub](https://github.com/brettwooldridge/hikaricp "GitHub - brettwooldridge/HikariCP: 光 HikariCP・A solid, high-performance, JDBC connection pool at last. · GitHub"))

### 设置过大有什么问题？

1. 连接可能先被 MySQL / 网络设备断开
    
2. 保活来不及执行
    

### 设置过小有什么问题？

1. 无意义 ping 增多
    
2. 对数据库产生额外轻微压力
    
3. 多实例下保活流量叠加
    

### 生产建议

一般不需要频繁改。

适合配置的场景：

1. 云数据库
    
2. NAT 网关
    
3. 中间代理
    
4. 防火墙会清理 idle TCP
    

示例：

```yaml
keepalive-time: 300000 # 5 分钟
```

关系：

```text
keepaliveTime < maxLifetime < wait_timeout
```

---

## 7. validationTimeout

### 控制什么？

检测连接是否存活的最大耗时。

官方默认值：`5000ms`，最低 `250ms`，并且必须小于 `connectionTimeout`。([GitHub](https://github.com/brettwooldridge/hikaricp "GitHub - brettwooldridge/HikariCP: 光 HikariCP・A solid, high-performance, JDBC connection pool at last. · GitHub"))

### 设置过大有什么问题？

1. 连接检测卡住
    
2. 获取连接耗时增加
    
3. 故障恢复慢
    

### 设置过小有什么问题？

1. 网络轻微抖动就误判连接不可用
    
2. 连接被频繁丢弃
    

### 生产建议

```yaml
validation-timeout: 1000
```

或：

```yaml
validation-timeout: 2000
```

如果 `connectionTimeout = 3000ms`，那么 `validationTimeout` 不应该设到 5000ms。

---

## 8. leakDetectionThreshold

### 控制什么？

连接借出去超过多久还没归还，就打印疑似泄漏日志。

官方默认值：`0`，表示关闭；启用时最低值是 `2000ms`。([GitHub](https://github.com/brettwooldridge/hikaricp "GitHub - brettwooldridge/HikariCP: 光 HikariCP・A solid, high-performance, JDBC connection pool at last. · GitHub"))

### 设置过大有什么问题？

1. 泄漏发现太晚
    
2. 定位困难
    

### 设置过小有什么问题？

1. 正常慢 SQL 被误报
    
2. 日志噪音大
    
3. 影响排查判断
    

### 生产建议

平时可以关闭，排查问题时打开。

例如：

```yaml
leak-detection-threshold: 10000
```

含义：

```text
连接被借出超过 10 秒未归还，打印日志
```

注意：

> 它只能发现“连接长时间未归还”，不能百分百证明是代码泄漏。慢 SQL、长事务、线程阻塞也会触发。

---

## 9. poolName

### 控制什么？

连接池名称，用于日志、JMX、监控指标区分。

官方默认值：自动生成。([GitHub](https://github.com/brettwooldridge/hikaricp "GitHub - brettwooldridge/HikariCP: 光 HikariCP・A solid, high-performance, JDBC connection pool at last. · GitHub"))

### 生产建议

强烈建议显式配置。

```yaml
pool-name: order-service-hikari
```

多数据源尤其重要：

```yaml
pool-name: order-master-pool
pool-name: order-slave-pool
```

否则排查日志时很痛苦。

---

## 10. autoCommit

### 控制什么？

连接返回给业务时默认是否自动提交事务。

官方默认值：`true`。([GitHub](https://github.com/brettwooldridge/hikaricp "GitHub - brettwooldridge/HikariCP: 光 HikariCP・A solid, high-performance, JDBC connection pool at last. · GitHub"))

### 设置过大 / 过小的问题？

它不是大小参数，而是行为参数。

`autoCommit=true`：

适合：

1. 单条 SQL
    
2. 普通查询
    
3. 非事务性操作
    

`autoCommit=false`：

风险：

1. 忘记 commit / rollback
    
2. 长事务
    
3. 锁不释放
    
4. 连接归还前状态复杂
    

### 生产建议

Spring Boot + Spring 事务管理下，通常保持：

```yaml
auto-commit: true
```

事务由：

```java
@Transactional
```

控制。

不要为了“看起来专业”全局关掉 autoCommit。

# 五、连接池与 MySQL 的关系

## 1. 应用连接池最大连接数和 MySQL max_connections 的关系

MySQL 的 `max_connections` 是数据库服务端允许的最大客户端连接数。MySQL 官方说明，`Too many connections` 表示所有可用连接都已被占用，允许数量由 `max_connections` 控制。([MySQL开发者专区](https://dev.mysql.com/doc/refman/8.2/en/too-many-connections.html "MySQL :: MySQL 8.4 Reference Manual :: B.3.2.5 Too many connections"))

关系公式：

```text
所有应用实例连接池上限总和 + 运维连接 + 其他任务连接 < MySQL max_connections
```

例如：

```text
MySQL max_connections = 800

order-service: 10 个实例，每个 maximumPoolSize = 30 => 300
user-service:  8 个实例，每个 maximumPoolSize = 20 => 160
admin-service: 2 个实例，每个 maximumPoolSize = 10 => 20
定时任务:      3 个实例，每个 maximumPoolSize = 10 => 30
其他工具/BI/运维连接预留 => 100

总计 = 610
```

这个还算安全。

如果每个服务都拍脑袋设 100：

```text
10 * 100 + 8 * 100 + 2 * 100 + 3 * 100 = 2300
```

MySQL 直接被连接数上限打爆。

---

## 2. 多个应用实例如何共同消耗数据库连接？

在 K8s / 微服务中，真实连接数不是看一个应用。

而是：

```text
总连接上限 =
  Σ（服务副本数 × 每个副本 maximumPoolSize × 数据源数量）
```

如果一个服务有主库、从库两个数据源：

```text
order-service 副本数 = 20
master pool = 20
slave pool = 30

总连接上限 = 20 * (20 + 30) = 1000
```

多数据源会放大连接数。

---

## 3. 为什么连接池不是越大越好？

因为数据库并发能力是有限的。

连接数增加不等于吞吐增加。

当连接数过大时：

```text
更多连接
  ↓
更多并发 SQL
  ↓
更多 CPU 竞争 / IO 竞争 / 锁竞争
  ↓
单 SQL 变慢
  ↓
连接占用时间变长
  ↓
连接池更容易满
  ↓
系统恶化
```

这就是典型的正反馈故障。

---

## 4. 连接数过大会带来哪些问题？

|问题|表现|
|---|---|
|MySQL 线程数增加|Threads_connected 高|
|活跃执行线程增加|Threads_running 高|
|CPU 上升|上下文切换、SQL 执行并发增加|
|IO 抖动|随机读写压力增加|
|锁等待增加|并发事务争抢资源|
|Buffer Pool 命中率下降|工作集变大|
|慢 SQL 增多|排队和竞争变严重|
|故障扩散|一个服务拖垮数据库，影响其他服务|

MySQL 指标中，`Threads_connected` 表示当前打开连接数，`Threads_running` 表示非 sleep 状态线程数；`Slow_queries` 表示超过 `long_query_time` 的查询数量。([MySQL开发者专区](https://dev.mysql.com/doc/refman/8.2/en/telemetry-metrics-meter-metrics.html "MySQL :: MySQL 8.4 Reference Manual :: 35.4.3 Server Metrics"))

---

## 5. 如何估算合理连接池大小？

先用 Little's Law 做初步估算：

```text
需要连接数 ≈ 峰值 QPS × 平均数据库连接占用时间（秒）
```

注意是“连接占用时间”，不是单纯 SQL 执行时间。

连接占用时间包括：

```text
获取连接后
  ↓
执行 SQL
  ↓
业务处理
  ↓
事务提交/回滚
  ↓
归还连接前
```

如果事务中夹了远程调用，连接占用时间会被严重放大。

---

## 6. 具体例子

假设订单服务：

```text
峰值 QPS = 500
每个请求平均访问数据库 2 次
每次数据库操作平均占用连接 20ms
```

估算：

```text
数据库操作 QPS = 500 * 2 = 1000
平均连接占用时间 = 20ms = 0.02s

需要连接数 ≈ 1000 * 0.02 = 20
```

考虑波动，设置：

```text
maximumPoolSize = 25～30
```

如果慢 SQL 导致平均连接占用时间变成 200ms：

```text
需要连接数 ≈ 1000 * 0.2 = 200
```

这时不应该简单把连接池调到 200，而应该优化 SQL。

因为数据库可能根本扛不住 200 个并发 SQL。

---

## 7. CPU、IO、事务耗时、慢 SQL 对连接池大小的影响

|因素|对连接池的影响|
|---|---|
|CPU 强|可以承载更多并发 SQL|
|IO 慢|连接占用时间变长，需要更多连接，但更应该优化 IO|
|慢 SQL 多|连接池容易耗尽|
|事务长|连接长期被占用|
|锁等待多|active connections 高，但数据库 CPU 可能不高|
|外部接口放在事务内|连接池被业务线程拖死|
|单实例多副本|总连接数放大|

# 六、常见生产问题

## 1. 获取连接超时

常见异常：

```text
SQLTransientConnectionException
Connection is not available, request timed out
```

### 本质

线程调用：

```java
dataSource.getConnection()
```

但在 `connectionTimeout` 时间内没有拿到连接。

### 常见原因

|原因|现象|
|---|---|
|慢 SQL|active 高，pending 高，MySQL 慢日志增加|
|连接泄漏|active 持续不下降|
|长事务|active 高，锁等待多|
|连接池太小|pending 高，但 MySQL 负载不高|
|MySQL 卡顿|acquire time 上升|
|网络问题|validation / acquire 异常|
|线程阻塞|连接借出后迟迟不归还|

### 排查步骤

1. 看 Hikari 指标：
    
    - active
        
    - idle
        
    - pending
        
    - timeout
        
2. 看 MySQL：
    
    - `SHOW PROCESSLIST`
        
    - `Threads_connected`
        
    - `Threads_running`
        
    - 慢 SQL 日志
        
3. 看应用线程栈：
    
    - 是否卡在 SQL？
        
    - 是否卡在 Redis / HTTP / MQ？
        
    - 是否事务中做远程调用？
        
4. 看连接泄漏日志：
    
    - `leakDetectionThreshold`
        

### 解决方案

|原因|解决|
|---|---|
|池太小|基于压测适当增加|
|慢 SQL|加索引、改 SQL、分页、拆查询|
|长事务|缩小事务边界|
|泄漏|try-with-resources / Spring 事务规范|
|远程调用占连接|远程调用移出事务|
|突发流量|限流、熔断、队列削峰|

---

## 2. 连接泄漏

### 错误示例

```java
Connection conn = dataSource.getConnection();
PreparedStatement ps = conn.prepareStatement("select * from user");
// 异常发生后没有 close
```

如果连接没有归还：

```text
active connections 持续上升
idle connections 下降到 0
pending threads 上升
最终 connection timeout
```

### 常见泄漏来源

1. 手写 JDBC 忘记 close
    
2. 异常路径没有释放连接
    
3. `ResultSet` / `Statement` / `Connection` 关闭顺序错误
    
4. 手动事务忘记 commit / rollback
    
5. 线程阻塞导致连接长期占用
    
6. `@Transactional` 方法中执行远程调用
    

### 正确姿势

```java
try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql);
     ResultSet rs = ps.executeQuery()) {

    while (rs.next()) {
        // handle row
    }
}
```

在 Spring 中，优先使用：

```java
@Transactional
public void doBusiness() {
    // mapper / repository 操作
}
```

不要自己到处手动管理连接。

---

## 3. 数据库连接被服务端断开

### 典型原因

MySQL 的 `wait_timeout` 会关闭长时间空闲连接。

如果：

```text
MySQL wait_timeout = 600s
Hikari maxLifetime = 1800s
```

那么 MySQL 可能先把连接关掉。

连接池还以为连接可用，就可能出现：

```text
Communications link failure
No operations allowed after connection closed
```

### 解决原则

```text
maxLifetime < wait_timeout
keepaliveTime < maxLifetime
```

示例：

```text
wait_timeout = 1800s
maxLifetime = 1700s
keepaliveTime = 300s
```

---

## 4. 慢 SQL 导致连接池耗尽

假设：

```text
maximumPoolSize = 20
接口 QPS = 100
每个 SQL 执行 2 秒
```

一个连接 2 秒才能释放。

理论并发需求：

```text
100 * 2 = 200 个连接
```

但池只有 20 个。

结果：

```text
20 个请求占满连接
80 个请求等待
等待超过 connectionTimeout 后报错
```

解决方向不是把池调到 200，而是：

1. 优化 SQL
    
2. 增加索引
    
3. 降低扫描行数
    
4. 分页限制
    
5. 拆分大查询
    
6. 加缓存
    
7. 异步化非核心路径
    
8. 限流保护数据库
    

---

## 5. 连接池满了，但数据库 CPU 并不高

这很常见。

原因可能是：

|原因|为什么 CPU 不高|
|---|---|
|锁等待|线程在等锁，不消耗 CPU|
|IO 等待|等磁盘，不消耗 CPU|
|网络等待|等返回，不消耗 CPU|
|事务中远程调用|连接被占着，但数据库没执行|
|应用线程阻塞|连接未归还|
|慢查询在等待行锁|MySQL 活跃但 CPU 低|

排查：

```sql
SHOW FULL PROCESSLIST;
```

重点看：

1. State
    
2. Time
    
3. Info
    
4. 是否大量 `Waiting for ... lock`
    
5. 是否大量 Sleep 连接
    
6. 是否存在长事务
    

---

## 6. 应用启动时大量创建连接导致数据库压力升高

如果：

```text
minimumIdle = maximumPoolSize = 50
副本数 = 30
```

滚动发布时可能瞬间创建：

```text
50 * 30 = 1500 个连接
```

问题：

1. MySQL 突然建连压力大
    
2. max_connections 被打满
    
3. 新旧实例同时存在，连接数翻倍
    
4. 发布期间其他服务受影响
    

解决：

1. 降低单实例 maximumPoolSize
    
2. 控制滚动发布并发
    
3. 配置合理 readinessProbe
    
4. 分批发布
    
5. 给数据库连接预留容量
    
6. 避免所有服务同时重启
    

---

## 7. Kubernetes / 微服务多实例连接数爆炸

这是生产中最典型的问题之一。

公式：

```text
总连接上限 =
服务数 × 副本数 × 每实例连接池大小 × 数据源数量
```

例如：

```text
20 个服务
每个服务 10 个副本
每个副本 maximumPoolSize = 30
每个服务 2 个数据源

总连接上限 = 20 * 10 * 30 * 2 = 12000
```

这对单个 MySQL 集群通常不可接受。

解决：

1. 服务分级配置连接池
    
2. 核心服务和边缘服务区别配置
    
3. 读写分离时分别规划连接数
    
4. K8s HPA 扩容要联动数据库容量
    
5. 使用限流保护数据库
    
6. 必要时引入数据库代理或中间层，但不能把它当万能药

# 七、Spring Boot 使用 HikariCP 实战

## 1. application.yml 示例

```yaml
spring:
  datasource:
    url: jdbc:mysql://127.0.0.1:3306/order_db?useUnicode=true&characterEncoding=utf8&serverTimezone=Asia/Shanghai&useSSL=false
    username: order_user
    password: ${DB_PASSWORD}
    driver-class-name: com.mysql.cj.jdbc.Driver

    hikari:
      pool-name: order-service-hikari

      # 最大连接数：不要拍脑袋设置，结合实例数、MySQL max_connections、压测结果决定
      maximum-pool-size: 20

      # 可不设置，让 Hikari 默认等于 maximumPoolSize，形成固定池
      # minimum-idle: 20

      # 获取连接最大等待时间。在线接口建议不要太长
      connection-timeout: 3000

      # 空闲连接保留时间。minimumIdle < maximumPoolSize 时更有意义
      idle-timeout: 600000

      # 连接最大生命周期，应小于 MySQL wait_timeout / 网络设备连接限制
      max-lifetime: 1800000

      # 空闲连接保活，必须小于 maxLifetime
      keepalive-time: 300000

      # 连接有效性检测超时，必须小于 connectionTimeout
      validation-timeout: 1000

      # 排查连接泄漏时开启，平时可关闭
      leak-detection-threshold: 10000

      # 默认自动提交。Spring 事务会按需接管
      auto-commit: true

management:
  endpoints:
    web:
      exposure:
        include: health,info,metrics,prometheus

  metrics:
    tags:
      application: order-service
```

---

## 2. 推荐生产参数解释

|参数|示例值|说明|
|---|--:|---|
|maximum-pool-size|20|根据 QPS、SQL 耗时、实例数、MySQL 容量决定|
|minimum-idle|不配置或 5～20|核心服务可固定池；低频服务可弹性|
|connection-timeout|3000ms|在线接口快速失败|
|idle-timeout|600000ms|弹性池释放空闲连接|
|max-lifetime|1800000ms|小于数据库 / 网络连接限制|
|keepalive-time|300000ms|防止空闲连接被中间设备清理|
|validation-timeout|1000ms|快速判断连接是否可用|
|leak-detection-threshold|10000ms|排查问题时开启|
|pool-name|明确命名|方便日志和监控定位|

## 3. 多数据源注意事项

多数据源不是配置多个 URL 这么简单。

需要关注：

```text
总连接数 = master pool + slave pool + 其他数据源 pool
```

示例：

```yaml
app:
  datasource:
    master:
      hikari:
        pool-name: order-master-pool
        maximum-pool-size: 20
    slave:
      hikari:
        pool-name: order-slave-pool
        maximum-pool-size: 30
```

如果一个应用有 10 个副本：

```text
master 连接 = 10 * 20 = 200
slave 连接 = 10 * 30 = 300
总连接 = 500
```

多数据源必须单独做容量规划。

## 4. 如何开启连接池指标监控

Spring Boot Actuator + Micrometer 会暴露 Hikari 指标。

常见指标名称：

# 八、监控与调优

## 1. active connections

### 含义

正在被业务线程使用的连接数。

### 异常表现

|表现|可能原因|
|---|---|
|长期接近 maximumPoolSize|池快耗尽|
|突然升高|流量突增 / SQL 变慢|
|高但 MySQL CPU 低|锁等待 / 长事务 / 应用阻塞|

### 调优方向

1. 看 SQL 耗时
    
2. 看事务耗时
    
3. 看连接使用时间
    
4. 必要时增加池大小，但先确认数据库能承受
    

---

## 2. idle connections

### 含义

空闲可用连接数。

### 异常表现

|表现|可能原因|
|---|---|
|长期为 0|池持续紧张|
|很高但 active 很低|池配置过大|
|波动剧烈|流量尖峰 / 连接频繁创建销毁|

### 调优方向

1. idle 长期很高：降低 maximumPoolSize 或 minimumIdle
    
2. idle 长期为 0：分析 active 和 pending
    

---

## 3. pending threads

### 含义

正在等待获取连接的线程数。

### 异常表现

|表现|说明|
|---|---|
|偶发 > 0|可接受，需要观察|
|持续 > 0|连接池瓶颈|
|pending 高 + active 满|连接耗尽|
|pending 高 + MySQL CPU 低|可能锁等待、泄漏、长事务|

### 调优方向

优先查：

1. 慢 SQL
    
2. 长事务
    
3. 连接泄漏
    
4. 线程阻塞
    

再考虑加连接。

---

## 4. connection acquire time

### 含义

获取连接耗时。

### 异常表现

|表现|说明|
|---|---|
|平时几毫秒以内|正常|
|P95/P99 升高|连接池紧张|
|接近 connectionTimeout|即将大量超时|

### 调优方向

1. 降低连接占用时间
    
2. 优化 SQL
    
3. 缩短事务
    
4. 增加池大小，但要压测
    

---

## 5. connection usage time

### 含义

连接从借出到归还的时间。

这是非常关键的指标。

### 异常表现

|表现|可能原因|
|---|---|
|usage time 高|SQL 慢 / 事务长|
|usage time 远大于 SQL 时间|事务中夹杂业务逻辑或远程调用|
|P99 很高|少量请求拖住连接|

### 调优方向

1. 缩小 `@Transactional` 范围
    
2. 禁止事务内远程调用
    
3. 拆分大查询
    
4. 优化慢 SQL
    

---

## 6. connection timeout count

### 含义

获取连接超时次数。

### 异常表现

任何持续增长都要关注。

说明用户请求已经失败，或者后台任务已经失败。

### 调优方向

1. 查 active 是否打满
    
2. 查 pending 是否持续增加
    
3. 查慢 SQL
    
4. 查泄漏日志
    
5. 查 MySQL 是否有锁等待
    

---

## 7. leak detection log

### 含义

连接借出超过阈值未归还。

### 注意

它不是最终判决，只是“疑似”。

可能原因：

1. 真泄漏
    
2. 慢 SQL
    
3. 长事务
    
4. 线程阻塞
    
5. 大批量处理
    

### 调优方向

结合线程栈、SQL 日志、事务边界判断。

---

## 8. MySQL Threads_connected

### 含义

MySQL 当前打开连接数。MySQL 官方指标说明中，`Threads_connected` 表示当前打开的连接数。([MySQL开发者专区](https://dev.mysql.com/doc/refman/8.2/en/telemetry-metrics-meter-metrics.html "MySQL :: MySQL 8.4 Reference Manual :: 35.4.3 Server Metrics"))

### 异常表现

|表现|说明|
|---|---|
|接近 max_connections|连接数风险|
|发布时飙升|实例启动建连太猛|
|长期高但业务低|连接池过大或连接未释放|

---

## 9. MySQL Threads_running

### 含义

MySQL 当前非 sleep 状态线程数。MySQL 官方指标说明中，`Threads_running` 表示非 sleeping 状态的线程数。([MySQL开发者专区](https://dev.mysql.com/doc/refman/8.2/en/telemetry-metrics-meter-metrics.html "MySQL :: MySQL 8.4 Reference Manual :: 35.4.3 Server Metrics"))

### 异常表现

|表现|说明|
|---|---|
|Threads_connected 高，Threads_running 低|很多连接空闲|
|Threads_running 高|数据库正在高并发执行|
|Threads_running 高 + CPU 高|数据库 CPU 压力大|
|Threads_running 高 + CPU 不高|可能 IO / 锁等待|

---

## 10. 慢 SQL 数量和耗时

MySQL 官方指标中，`Slow_queries` 表示超过 `long_query_time` 的查询数量。([MySQL开发者专区](https://dev.mysql.com/doc/refman/8.2/en/telemetry-metrics-meter-metrics.html "MySQL :: MySQL 8.4 Reference Manual :: 35.4.3 Server Metrics"))

### 重点看

1. 慢 SQL 数量
    
2. P95 / P99 耗时
    
3. 扫描行数
    
4. 返回行数
    
5. 是否 filesort
    
6. 是否临时表
    
7. 是否锁等待
    

### 调优顺序

```text
慢 SQL 治理
  ↓
事务边界治理
  ↓
连接池参数调整
  ↓
数据库容量规划
```

不要反过来。

# 九、代码示例

## 1. 原生 JDBC 不使用连接池

```java
import java.sql.Connection;
import java.sql.DriverManager;
import java.sql.PreparedStatement;
import java.sql.ResultSet;

public class RawJdbcWithoutPoolDemo {

    public static void main(String[] args) throws Exception {
        String url = "jdbc:mysql://127.0.0.1:3306/order_db?serverTimezone=Asia/Shanghai";
        String username = "root";
        String password = "123456";

        // 每次调用都会创建一个新的物理数据库连接
        try (Connection connection = DriverManager.getConnection(url, username, password);
             PreparedStatement ps = connection.prepareStatement(
                     "select id, order_no, user_id, status from user_order where id = ?")) {

            ps.setLong(1, 1L);

            try (ResultSet rs = ps.executeQuery()) {
                while (rs.next()) {
                    System.out.println("orderNo = " + rs.getString("order_no"));
                }
            }
        }

        // try-with-resources 结束后，物理连接关闭
    }
}
```

关键点：

```text
DriverManager.getConnection()
```

每次都会走建连流程，高并发下成本很高。

## 2. 使用 HikariCP 的原生 Java 示例

Maven 依赖：

```xml
<dependency>
    <groupId>com.zaxxer</groupId>
    <artifactId>HikariCP</artifactId>
</dependency>

<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
</dependency>
```

代码：

```java
import com.zaxxer.hikari.HikariConfig;
import com.zaxxer.hikari.HikariDataSource;

import java.sql.Connection;
import java.sql.PreparedStatement;
import java.sql.ResultSet;

public class HikariRawJavaDemo {

    public static void main(String[] args) throws Exception {
        HikariConfig config = new HikariConfig();

        config.setJdbcUrl("jdbc:mysql://127.0.0.1:3306/order_db?serverTimezone=Asia/Shanghai");
        config.setUsername("root");
        config.setPassword("123456");

        // 连接池名称，方便日志和监控定位
        config.setPoolName("order-demo-pool");

        // 最大连接数
        config.setMaximumPoolSize(10);

        // 获取连接最大等待时间
        config.setConnectionTimeout(3000);

        // 连接最大生命周期
        config.setMaxLifetime(1800000);

        // 疑似泄漏检测，超过 10 秒未归还连接会打印日志
        config.setLeakDetectionThreshold(10000);

        try (HikariDataSource dataSource = new HikariDataSource(config)) {

            // 从连接池借连接
            try (Connection connection = dataSource.getConnection();
                 PreparedStatement ps = connection.prepareStatement(
                         "select id, order_no, user_id, status from user_order where id = ?")) {

                ps.setLong(1, 1L);

                try (ResultSet rs = ps.executeQuery()) {
                    while (rs.next()) {
                        System.out.println("orderNo = " + rs.getString("order_no"));
                    }
                }
            }

            // connection.close() 后，连接归还连接池，不是物理关闭
        }
    }
}
```

## 4. 连接泄漏的错误示例

```java
import javax.sql.DataSource;
import java.sql.Connection;
import java.sql.PreparedStatement;
import java.sql.ResultSet;

public class ConnectionLeakBadExample {

    private final DataSource dataSource;

    public ConnectionLeakBadExample(DataSource dataSource) {
        this.dataSource = dataSource;
    }

    public void queryOrder(Long id) throws Exception {
        Connection connection = dataSource.getConnection();

        // 如果下面任意一行抛异常，connection 就不会被关闭
        PreparedStatement ps = connection.prepareStatement(
                "select id, order_no from user_order where id = ?"
        );
        ps.setLong(1, id);

        ResultSet rs = ps.executeQuery();

        while (rs.next()) {
            System.out.println(rs.getString("order_no"));
        }

        // 错误：只关闭 ResultSet / Statement 也不够
        // 更严重的是异常路径下 connection 根本不会 close
        rs.close();
        ps.close();

        // 如果忘了这一行，连接永远不归还连接池
        // connection.close();
    }
}
```

问题：

```text
异常路径没有释放连接
  ↓
active connections 持续增加
  ↓
idle connections 变成 0
  ↓
pending threads 增加
  ↓
connection timeout
```

## 6. 慢 SQL 导致连接池耗尽的模拟示例

下面是一个教学模拟，不建议在生产执行。

```java
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

import javax.sql.DataSource;
import java.sql.Connection;
import java.sql.PreparedStatement;
import java.sql.ResultSet;

@RestController
public class SlowSqlPoolExhaustDemoController {

    private final DataSource dataSource;

    public SlowSqlPoolExhaustDemoController(DataSource dataSource) {
        this.dataSource = dataSource;
    }

    @GetMapping("/demo/slow-query")
    public String slowQuery() throws Exception {
        try (Connection connection = dataSource.getConnection();
             PreparedStatement ps = connection.prepareStatement("select sleep(5)")) {

            // 这个 SQL 会让连接被占用 5 秒
            try (ResultSet rs = ps.executeQuery()) {
                if (rs.next()) {
                    return "done";
                }
            }
        }

        return "done";
    }
}
```

如果配置：

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 10
      connection-timeout: 3000
```

然后并发 50 个请求：

```text
前 10 个请求拿到连接，执行 sleep(5)
后 40 个请求等待连接
等待超过 3 秒后，大量请求报 connection timeout
```

这说明：

> 慢 SQL 会直接放大连接占用时间，最终耗尽连接池。

# 十、深入理解

## 1. 为什么连接池大小不是线程池大小？

线程池控制的是：

```text
应用侧能同时执行多少任务
```

连接池控制的是：

```text
应用侧能同时占用多少数据库连接
```

一个请求线程不一定一直占用数据库连接。

例如：

```text
HTTP 请求进入
  ↓
参数校验
  ↓
查 Redis
  ↓
调用数据库，此时才占用连接
  ↓
业务计算
  ↓
返回
```

所以：

```text
Tomcat 线程数 200
不代表 Hikari maximumPoolSize 也要 200
```

否则数据库会被过度并发打爆。

---

## 2. 为什么连接池过大会降低性能？

因为数据库不是无限并行系统。

连接池过大后：

```text
并发 SQL 增加
  ↓
CPU / IO / 锁竞争增加
  ↓
单 SQL 耗时变长
  ↓
连接占用时间变长
  ↓
更多请求等待连接
  ↓
系统吞吐下降
```

连接池调优的目标不是“让等待线程为 0”，而是：

```text
在数据库可承受范围内，获得稳定吞吐和可接受延迟
```

---

## 3. 为什么 maxLifetime 通常要小于数据库 wait_timeout？

因为如果数据库先关连接，连接池可能持有一个已经失效的连接。

正确做法：

```text
连接池主动让连接退休
  优于
数据库/网络设备被动断开连接
```

所以：

```text
maxLifetime < wait_timeout
```

HikariCP 官方也建议 `maxLifetime` 应比数据库或基础设施的连接时间限制短几秒。([GitHub](https://github.com/brettwooldridge/hikaricp "GitHub - brettwooldridge/HikariCP: 光 HikariCP・A solid, high-performance, JDBC connection pool at last. · GitHub"))

---

## 4. 为什么连接池可以缓解连接创建成本，但不能解决慢 SQL？

连接池优化的是：

```text
建连成本
```

慢 SQL 的问题在于：

```text
执行成本
```

例如：

```sql
select *
from order_log
where user_id = 10001
order by create_time desc;
```

如果缺索引，扫描 1000 万行。

连接池只能帮你快速拿到连接，但拿到连接后，这条 SQL 仍然慢。

甚至更糟：

```text
慢 SQL 占用连接更久
  ↓
连接池更容易耗尽
```

---

## 5. 为什么事务时间越长，连接池压力越大？

因为 Spring 事务通常会在事务开始时绑定数据库连接到当前线程。

例如：

```java
@Transactional
public void createOrder() {
    orderMapper.insert(order);

    // 错误示例：事务内调用外部接口
    paymentClient.prepay(order);

    orderMapper.updateStatus(order.getId());
}
```

如果 `paymentClient.prepay()` 耗时 2 秒，这 2 秒内连接可能一直被占用。

正确思路：

```text
事务内只做数据库一致性操作
远程调用、MQ、文件上传、大计算尽量移出事务
```

---

## 6. 为什么微服务架构下需要特别关注数据库连接数？

因为微服务会放大连接池数量。

单体应用：

```text
1 个应用 * 1 个连接池 * 30 = 30
```

微服务：

```text
20 个服务 * 每个 10 副本 * 每个 30 连接 = 6000
```

数据库看到的不是“服务”，而是连接。

K8s 自动扩容时，如果只看 CPU 扩容应用，不看数据库连接容量，很容易把数据库打爆。

---

## 7. 连接池和线程池、HTTP 连接池有什么相似和不同？

### 相似点

|池化对象|共同目标|
|---|---|
|数据库连接池|复用昂贵资源|
|线程池|复用线程，限制并发|
|HTTP 连接池|复用 TCP 连接|
|Redis 连接池|复用连接，控制并发|

共同思想：

```text
资源复用 + 上限控制 + 等待队列 + 超时失败 + 监控指标
```

### 不同点

|类型|管理对象|主要瓶颈|
|---|---|---|
|线程池|CPU 执行线程|CPU / 队列|
|DB 连接池|数据库会话|DB CPU / IO / 锁|
|HTTP 连接池|远程服务连接|网络 / 对端服务|
|Redis 连接池|Redis 连接|Redis 单线程执行 / 网络|

数据库连接池尤其敏感，因为它直接消耗数据库服务端资源。

|connectionTimeout 越长越稳|太长会拖垮线程|
|leak 日志一定是代码泄漏|也可能是慢 SQL 或长事务|

### 自测问题

# 最后给你一个生产判断框架

排查连接池问题时，不要先改参数。

按这个顺序：

```text
1. active 是否接近 maximumPoolSize？
2. idle 是否长期为 0？
3. pending 是否持续大于 0？
4. acquire time 是否升高？
5. usage time 是否升高？
6. timeout count 是否增长？
7. MySQL Threads_connected 是否接近 max_connections？
8. MySQL Threads_running 是否异常？
9. 慢 SQL 是否增多？
10. 是否存在长事务、锁等待、连接泄漏？
```

调优优先级：

```text
先治理 SQL 和事务
再调整连接池参数
最后做数据库容量扩展
```

一句话总结：

> **连接池是数据库访问的“并发闸门”。它既要保护应用，也要保护数据库。真正的能力不是把参数调大，而是知道什么时候该限、什么时候该扩、什么时候该优化 SQL。**
