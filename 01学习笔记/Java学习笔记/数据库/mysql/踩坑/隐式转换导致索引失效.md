## 结论先行

**MySQL 隐式转换导致索引失效**，本质是：

> 查询条件两边的数据类型不一致，MySQL 为了比较它们，会自动做类型转换；如果转换发生在“索引列”这一侧，索引中的有序结构就无法直接使用，可能退化成全表扫描。

典型事故：

```sql
-- user_phone 是 varchar 类型，但查询时用了数字
SELECT * FROM user WHERE user_phone = 13800138000;
```

看起来没问题，实际可能导致索引失效。

---

# 1. 最经典例子：varchar 字段用数字查询

## 表结构

```sql
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    user_phone VARCHAR(20) NOT NULL,
    name VARCHAR(50),
    KEY idx_user_phone (user_phone)
);
```

`user_phone` 是字符串类型，并且有索引。

---

## 错误写法

```sql
SELECT * FROM user WHERE user_phone = 13800138000;
```

这里右边是数字，左边是 `VARCHAR`。

MySQL 会尝试做类型转换，大致等价于：

```sql
SELECT * FROM user WHERE CAST(user_phone AS SIGNED) = 13800138000;
```

问题在于：

```sql
CAST(user_phone AS SIGNED)
```

作用在了索引列 `user_phone` 上。

索引中存的是字符串顺序，例如：

```text
'10001'
'13800138000'
'20001'
```

但现在 MySQL 要把每一行的 `user_phone` 转成数字再比较，原来的字符串索引顺序没法直接用。

所以可能变成：

```text
逐行扫描 -> 转数字 -> 比较
```

索引失效。

---

## 正确写法

```sql
SELECT * FROM user WHERE user_phone = '13800138000';
```

让查询值和字段类型保持一致。

---

# 2. 为什么字符串转数字特别危险？

假设表中有这些数据：

```text
user_phone
--------------
'123'
'123abc'
'00123'
'abc123'
```

执行：

```sql
SELECT * FROM user WHERE user_phone = 123;
```

MySQL 的数字转换规则可能会出现很诡异的结果：

```text
'123'    -> 123
'123abc' -> 123
'00123'  -> 123
'abc123' -> 0
```

所以这条 SQL 可能查出：

```text
'123'
'123abc'
'00123'
```

这不仅是**索引问题**，还是**数据正确性问题**。

# 3. 另一个常见例子：字符串主键被数字查询

## 表结构

```sql
CREATE TABLE order_info (
    order_no VARCHAR(64) PRIMARY KEY,
    user_id BIGINT,
    amount DECIMAL(10, 2)
);
```

---

## 错误写法

```sql
SELECT * FROM order_info WHERE order_no = 202405130001;
```

`order_no` 是 `VARCHAR`，但条件值是数字。

可能导致：

```sql
CAST(order_no AS SIGNED) = 202405130001
```

这样主键索引也可能无法正常使用。

---

## 正确写法

```sql
SELECT * FROM order_info WHERE order_no = '202405130001';
```

订单号、手机号、身份证号、编码类字段，只要是 `VARCHAR`，查询时都必须加引号。

# 4. 数字字段传字符串，一定会索引失效吗？

不一定。

例如：

```sql
CREATE TABLE user (
    id BIGINT PRIMARY KEY,
    age INT,
    KEY idx_age (age)
);
```

查询：

```sql
SELECT * FROM user WHERE age = '18';
```

`age` 是数字，右边是字符串 `'18'`。

MySQL 通常会把 `'18'` 转成数字：

```sql
age = 18
```

转换发生在常量侧，不是索引列侧，所以索引一般还能用。

---

## 对比一下

|字段类型|查询条件|转换方向|索引风险|
|---|---|---|---|
|`VARCHAR`|`phone = 13800138000`|字段转数字|高风险|
|`VARCHAR`|`phone = '13800138000'`|无需转换|正常|
|`INT`|`age = '18'`|常量转数字|通常正常|
|`INT`|`age = 18`|无需转换|正常|

# 5. 怎么确认是否索引失效？

用 `EXPLAIN`。

```sql
EXPLAIN SELECT * FROM user WHERE user_phone = 13800138000;
```

重点看：

```text
type: ALL
key: NULL
rows: 很大
```

说明可能走了全表扫描。

正确写法：

```sql
EXPLAIN SELECT * FROM user WHERE user_phone = '13800138000';
```

理想结果：

```text
type: ref / const
key: idx_user_phone
rows: 1
```

# 6. 后端开发中最容易踩坑的地方

Java 代码里经常这样写：

```java
String phone = "13800138000";

LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(User::getUserPhone, Long.valueOf(phone));
```

生成 SQL 类似：

```sql
WHERE user_phone = 13800138000
```

这就有问题。

正确写法：

```java
wrapper.eq(User::getUserPhone, phone);
```

字段是 `VARCHAR`，参数就应该传 `String`。

# 7. 生产规范

## 核心原则

> 字段是什么类型，查询参数就用什么类型。

具体规范：

|字段含义|推荐类型|查询参数|
|---|---|---|
|手机号|`VARCHAR`|字符串|
|身份证号|`VARCHAR`|字符串|
|订单号|`VARCHAR`|字符串|
|用户 ID|`BIGINT`|数字|
|状态码|`TINYINT` / `INT`|数字|
|金额|`DECIMAL`|Decimal|
|时间|`DATETIME` / `TIMESTAMP`|时间类型|

# 面试口述版

MySQL 隐式转换导致索引失效，主要是因为 SQL 条件两边类型不一致，MySQL 会自动做类型转换。如果转换发生在索引字段上，比如 `VARCHAR` 类型的手机号字段写成 `phone = 13800138000`，MySQL 可能会把 `phone` 字段逐行转成数字再比较，相当于对索引列做了函数计算，B+Tree 索引原有的有序性无法直接利用，最终可能退化成全表扫描。

实际开发中，手机号、订单号、身份证号这类 `VARCHAR` 字段，查询条件一定要加引号，Java 参数也要用 `String` 传入。排查时可以用 `EXPLAIN` 看 `key`、`type`、`rows`，如果 `key` 是 `NULL`，`type` 是 `ALL`，基本就说明索引没有正常使用。
