水平分表用取模，日志、报表类，可以用日期

不同业务的数据分布特点不同，所以：

- **高并发业务表** → 常用“取模分表”
    
- **时间序列业务表** → 常用“按日期分表”
    

这是企业里最经典的两种水平分表策略。

---

# 1. 什么是“水平分表”

比如订单表：

```sql
t_order
```

数据越来越大：

```text
5000万
1亿
5亿
10亿
```

会出现：

- 查询慢
    
- 索引巨大
    
- Buffer Pool 命中率下降
    
- DDL 风险高
    
- 主从同步压力大
    

于是拆表：

```text
t_order_0
t_order_1
t_order_2
...
```

这就是：

```text
水平分表
```

本质：

```text
表结构一样
数据拆到不同表
```

---

# 2. 什么叫“用取模分表”

## 核心思想

根据某个字段：

```text
id % N
```

决定数据进入哪张表。

---

## 例子

分成 4 张表：

```text
t_order_0
t_order_1
t_order_2
t_order_3
```

路由规则：

```java
tableIndex = userId % 4;
```

例如：

|userId|userId % 4|表|
|---|---|---|
|1001|1|t_order_1|
|1002|2|t_order_2|
|1003|3|t_order_3|
|1004|0|t_order_0|

---

# 3. 为什么高并发业务喜欢“取模分表”

因为：

```text
数据分布均匀
```

例如订单系统：

```text
用户1下单
用户2下单
用户3下单
```

通过取模：

```text
数据平均分散到多个表
```

这样：

- 热点降低
    
- 并发分散
    
- 单表压力下降
    
- 索引更小
    

---

# 4. 为什么订单、用户、支付喜欢取模

因为它们的查询特点是：

## 按用户查询

例如：

```sql
SELECT *
FROM t_order
WHERE user_id = 1001;
```

如果：

```text
user_id % 4
```

作为分表键。

那么：

```text
user_id = 1001
永远在同一张表
```

查询只查：

```text
t_order_1
```

不需要扫全库。

这叫：

```text
路由可定位
```

这是分表最重要的设计原则。

---

# 5. 为什么“日志、报表类”适合按日期分表

因为这类数据：

## 最大特点：

```text
天然带时间属性
```

例如：

- 操作日志
    
- 登录日志
    
- 审计日志
    
- 埋点日志
    
- 监控数据
    
- BI报表
    
- 消息流水
    

它们的查询通常是：

```sql
WHERE create_time BETWEEN ...
```

例如：

```sql
SELECT *
FROM t_log
WHERE create_time >= '2026-05-01'
AND create_time < '2026-06-01';
```

---

# 6. 日期分表怎么做

例如按月：

```text
t_log_202601
t_log_202602
t_log_202603
```

或者按天：

```text
t_log_20260501
t_log_20260502
```

插入：

```text
2026-05-13 的数据
→ t_log_202605
```

查询：

```text
查 2026-05
→ 只查 t_log_202605
```

---

# 7. 为什么日志不用取模

因为日志查询：

很少：

```sql
WHERE user_id = ?
```

更多是：

```sql
WHERE create_time BETWEEN ...
```

所以：

## 如果用取模：

```text
5月日志
可能散落在 64 张表
```

查询：

```text
需要扫 64 张表
```

非常痛苦。

---

## 但按日期：

```text
5月数据就在 5月表
```

天然适合范围查询。

---

# 8. 两种分表策略的核心区别

|维度|取模分表|日期分表|
|---|---|---|
|核心目标|均匀分散压力|控制时间范围|
|典型业务|订单、支付、用户|日志、埋点、报表|
|常见查询|按 user_id|按时间|
|路由方式|id % N|yyyyMM|
|优势|并发均衡|时间查询极快|
|缺点|时间范围查询麻烦|热点月份可能过大|

---

# 9. 企业里的真实情况：经常组合使用

比如：

```text
按月 + 取模
```

例如：

```text
t_order_202605_0
t_order_202605_1
t_order_202605_2
t_order_202605_3
```

规则：

```text
先按月份
再按 user_id % 4
```

这样：

- 控制单月数据量
    
- 分散热点
    
- 查询效率高
    

这是大型系统常见方案。

---

# 10. 真正难的地方：分表键设计

很多人以为：

```text
分表 = ShardingSphere
```

其实：

```text
最难的是：
分表键怎么选
```

因为分表键决定：

- 查询是否能路由
    
- 是否会跨表
    
- 是否会全库扫描
    
- 是否会热点
    
- 是否方便扩容
    

这是架构设计问题。

---

# 11. 面试口述版

水平分表一般有两种经典策略：取模分表和日期分表。  
取模分表通常用于订单、支付、用户等高并发业务，通过 `user_id % N` 或 `order_id % N` 将数据均匀分散到多个表中，降低单表压力，并且按用户查询时可以精准路由到单表。  
日志、埋点、报表等时间序列数据，更适合按日期分表，例如按月或按天拆分，因为这类查询通常带时间范围条件，按时间分表能减少扫描范围。  
企业里也经常组合使用，比如“按月 + 取模”，既控制单表数据量，又分散并发热点。  
真正困难的地方不是分表工具，而是分表键设计，因为它直接决定系统后续查询、扩容和性能表现。
