## 结论

**N+1 查询**是 ORM / 数据访问层里很常见的性能问题：

> 先查出 1 次主数据列表，再对列表里的每一条数据分别追加查询一次关联数据，最终变成 `1 + N` 次 SQL。

这里的：

- `1`：查询主表列表的一次 SQL
    
- `N`：对每条主记录分别查询关联数据的 N 次 SQL
    

当 N 很大时，数据库压力、网络往返、接口响应时间都会明显变差。

---

## 1. 一个典型例子：博客文章 + 作者信息

假设有两张表：

```sql
article
-------
id
title
author_id

author
-------
id
name
```

你要查询文章列表，并展示每篇文章的作者名称。

### 错误写法：N+1 查询

第一步，先查文章列表：

```sql
SELECT id, title, author_id
FROM article
LIMIT 10;
```

这是一条 SQL。

假设查出了 10 篇文章，然后代码里循环查作者：

```java
List<Article> articles = articleMapper.selectList();

for (Article article : articles) {
    Author author = authorMapper.selectById(article.getAuthorId());
    article.setAuthor(author);
}
```

实际执行的 SQL 类似：

```sql
SELECT id, title, author_id FROM article LIMIT 10;

SELECT id, name FROM author WHERE id = 1;
SELECT id, name FROM author WHERE id = 2;
SELECT id, name FROM author WHERE id = 3;
...
SELECT id, name FROM author WHERE id = 10;
```

最终一共执行：

```text
1 次文章查询 + 10 次作者查询 = 11 次 SQL
```

如果文章是 100 条，就是：

```text
1 + 100 = 101 次 SQL
```

这就是 **N+1 查询问题**。

---

## 2. 为什么它危险？

N+1 查询的问题不是单条 SQL 慢，而是**SQL 次数失控**。

### 小数据时看不出来

开发环境可能只有 10 条数据：

```text
1 + 10 = 11 次查询
```

感觉没问题。

### 线上数据一多就炸

线上分页查 100 条、500 条，甚至批处理查几千条：

```text
1 + 500 = 501 次查询
```

每次 SQL 都涉及：

- 数据库连接占用
    
- 网络往返
    
- SQL 解析与执行
    
- 结果集返回
    
- ORM 对象映射
    

所以它会导致：

- 接口响应慢
    
- 数据库 QPS 飙升
    
- 连接池被打满
    
- 慢查询变多
    
- 链路追踪里出现大量重复 SQL
    

---

## 3. 正确做法一：使用 JOIN 一次查出来

```sql
SELECT 
    a.id,
    a.title,
    au.id AS author_id,
    au.name AS author_name
FROM article a
LEFT JOIN author au ON a.author_id = au.id
LIMIT 10;
```

这样只需要一次 SQL：

```text
1 次 SQL
```

适合：

- 一对一
    
- 多对一
    
- 数据量不大的一对多
    
- 页面展示型查询
    

比如：

- 文章列表 + 作者
    
- 订单列表 + 用户
    
- 评论列表 + 用户头像
    

---

## 4. 正确做法二：批量查询，再内存组装

有些时候不想 JOIN，比如为了避免结果集膨胀，可以分两步批量查。

### 第一步：查文章

```sql
SELECT id, title, author_id
FROM article
LIMIT 10;
```

### 第二步：收集 author_id，批量查询作者

```sql
SELECT id, name
FROM author
WHERE id IN (1, 2, 3, 4, 5);
```

Java 代码大概是：

```java
List<Article> articles = articleMapper.selectList();

List<Long> authorIds = articles.stream()
        .map(Article::getAuthorId)
        .distinct()
        .toList();

List<Author> authors = authorMapper.selectByIds(authorIds);

Map<Long, Author> authorMap = authors.stream()
        .collect(Collectors.toMap(Author::getId, Function.identity()));

for (Article article : articles) {
    article.setAuthor(authorMap.get(article.getAuthorId()));
}
```

最终 SQL 次数：

```text
1 次文章查询 + 1 次作者批量查询 = 2 次 SQL
```

这比 `1 + N` 稳定很多。

---

## 5. 正确做法三：MyBatis 的 nested select 要慎用

MyBatis 里常见的 N+1 来源是 `association` 或 `collection` 使用嵌套查询。

例如：

```xml
<resultMap id="ArticleResultMap" type="Article">
    <id property="id" column="id"/>
    <result property="title" column="title"/>
    <association 
        property="author"
        column="author_id"
        select="selectAuthorById"/>
</resultMap>
```

这类配置的意思是：

> 查出 Article 后，再根据每条记录的 author_id 去调用 `selectAuthorById`。

如果查出 100 篇文章，就可能触发 100 次作者查询。

这就是 MyBatis 里的典型 N+1。

更推荐：

### 方案一：JOIN + resultMap

```xml
<select id="selectArticleListWithAuthor" resultMap="ArticleAuthorResultMap">
    SELECT
        a.id AS article_id,
        a.title AS article_title,
        au.id AS author_id,
        au.name AS author_name
    FROM article a
    LEFT JOIN author au ON a.author_id = au.id
</select>
```

### 方案二：业务层批量查询组装

也就是前面说的 `IN (...)` 查询。

## 6. 和懒加载的关系

N+1 经常和 **懒加载 Lazy Loading** 一起出现。

例如在 ORM 中：

```java
List<Article> articles = articleRepository.findAll();

for (Article article : articles) {
    System.out.println(article.getAuthor().getName());
}
```

表面上看只查了一次文章。

但当调用：

```java
article.getAuthor()
```

时，ORM 才去数据库查作者。

于是循环 100 次，就触发 100 次额外 SQL。

所以懒加载本身不是错，但在列表场景里很容易制造 N+1。

---

## 7. 怎么判断有没有 N+1？

看日志最直接。

如果你发现接口里出现大量类似 SQL：

```sql
SELECT * FROM author WHERE id = ?
SELECT * FROM author WHERE id = ?
SELECT * FROM author WHERE id = ?
SELECT * FROM author WHERE id = ?
```

而且这些 SQL 只是在参数不同，那么大概率就是 N+1。

常见排查方式：

- 开启 SQL 日志
    
- 看接口执行了多少条 SQL
    
- 用链路追踪观察 DB 调用次数
    
- 用 APM 工具看重复 SQL
    
- 在测试环境压测分页接口
    

---

## 8. 工程经验：什么时候 JOIN，什么时候批量查？

|场景|推荐方式|
|---|---|
|多对一 / 一对一|JOIN|
|文章列表查作者|JOIN 或批量查|
|订单列表查用户|JOIN 或批量查|
|一对多集合，例如文章查评论列表|谨慎 JOIN，容易数据膨胀|
|多层嵌套对象|批量查 + 内存组装|
|数据量小、后台管理页|JOIN 简单直接|
|高并发核心接口|优先控制 SQL 次数和结果集大小|

---

## 9. 一句话总结

**N+1 查询的本质是：把本该批量完成的数据访问，错误地放进了循环里。**

对于 Java 后端来说，重点记住：

```text
不要在 for 循环里查数据库。
不要让 ORM 的懒加载在列表场景里偷偷查数据库。
列表页关联数据优先考虑 JOIN 或批量查询。
```

关键词：

```text
N+1 查询、ORM、懒加载、JOIN、批量查询、IN 查询、MyBatis association、数据库性能优化
```
