## 结论

**覆盖索引 = 查询需要的字段，全部都在同一个二级索引里。**

这样 MySQL 只查二级索引就能返回结果，不需要再根据主键 ID 回到聚簇索引里查整行数据，也就是**避免回表**。

---

# 1. 先看普通二级索引为什么会回表

假设有一张订单表：

```sql
CREATE TABLE t_order (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    order_no VARCHAR(64) NOT NULL,
    status TINYINT NOT NULL,
    amount DECIMAL(10, 2) NOT NULL,
    create_time DATETIME NOT NULL,
    remark VARCHAR(255),
    KEY idx_user_id (user_id)
);
```

执行查询：

```sql
SELECT order_no, amount
FROM t_order
WHERE user_id = 1001;
```

虽然有索引：

```sql
KEY idx_user_id (user_id)
```

但这个索引里只有：

```text
user_id + 主键 id
```

查询要返回：

```text
order_no, amount
```

这两个字段不在 `idx_user_id` 里。

所以流程是：

```text
idx_user_id 找到 user_id = 1001 的记录
        ↓
拿到主键 id
        ↓
回到聚簇索引查整行数据
        ↓
取出 order_no, amount
```

这就是**回表**。

---

# 2. 怎么用覆盖索引避免回表？

把查询需要的字段放进联合索引里：

```sql
CREATE INDEX idx_user_order_amount
ON t_order(user_id, order_no, amount);
```

然后查询：

```sql
SELECT order_no, amount
FROM t_order
WHERE user_id = 1001;
```

这时索引 `idx_user_order_amount` 里已经包含：

```text
user_id, order_no, amount
```

查询条件需要的 `user_id` 在索引里。

返回字段需要的 `order_no, amount` 也在索引里。

所以 MySQL 直接查二级索引即可：

```text
idx_user_order_amount
        ↓
找到 user_id = 1001
        ↓
直接返回 order_no, amount
```

不需要回表。

---

# 3. 如何判断是否用了覆盖索引？

用 `EXPLAIN`：

```sql
EXPLAIN
SELECT order_no, amount
FROM t_order
WHERE user_id = 1001;
```

重点看 `Extra` 字段。

如果出现：

```text
Using index
```

说明使用了覆盖索引。

示例：

```text
Extra: Using where; Using index
```

含义：

```text
Using index = 查询字段都从索引里拿到了，没有回表
```

---

# 4. 覆盖索引的核心写法

## 写法一：WHERE 字段 + SELECT 字段

查询：

```sql
SELECT order_no, amount
FROM t_order
WHERE user_id = 1001;
```

索引：

```sql
CREATE INDEX idx_user_order_amount
ON t_order(user_id, order_no, amount);
```

原则：

```text
WHERE 条件字段放前面
SELECT 返回字段放后面
```

---

## 写法二：WHERE + ORDER BY + SELECT

查询：

```sql
SELECT order_no, amount
FROM t_order
WHERE user_id = 1001
ORDER BY create_time DESC;
```

索引：

```sql
CREATE INDEX idx_user_time_order_amount
ON t_order(user_id, create_time, order_no, amount);
```

这样索引同时覆盖：

```text
WHERE: user_id
ORDER BY: create_time
SELECT: order_no, amount
```

---

## 写法三：分页查询减少回表

普通分页：

```sql
SELECT *
FROM t_order
WHERE user_id = 1001
ORDER BY create_time DESC
LIMIT 10000, 20;
```

这个很容易大量回表。

优化成两步：

```sql
SELECT id
FROM t_order
WHERE user_id = 1001
ORDER BY create_time DESC
LIMIT 10000, 20;
```

配合索引：

```sql
CREATE INDEX idx_user_time
ON t_order(user_id, create_time);
```

第一步只查 `id`，可以走覆盖索引。

然后再查详情：

```sql
SELECT *
FROM t_order
WHERE id IN (...);
```

这叫：

```text
先用覆盖索引定位主键，再回表少量数据
```

# 5. 一个典型业务案例

## 用户订单列表页

页面只展示：

```text
订单号、状态、金额、创建时间
```

SQL：

```sql
SELECT order_no, status, amount, create_time
FROM t_order
WHERE user_id = 1001
ORDER BY create_time DESC
LIMIT 20;
```

索引应该这样建：

```sql
CREATE INDEX idx_user_time_cover
ON t_order(user_id, create_time, order_no, status, amount);
```

这个索引覆盖了：

```text
user_id       用于 WHERE
create_time   用于 ORDER BY
order_no      用于 SELECT
status        用于 SELECT
amount        用于 SELECT
```

查询可以尽量避免回表。

---

# 6. 不要这样写

## 错误 1：SELECT *

```sql
SELECT *
FROM t_order
WHERE user_id = 1001;
```

`SELECT *` 基本很难使用覆盖索引，因为它要求返回整行数据。

正确做法：

```sql
SELECT order_no, status, amount
FROM t_order
WHERE user_id = 1001;
```

---

## 错误 2：索引字段太少

```sql
CREATE INDEX idx_user_id
ON t_order(user_id);
```

如果查询：

```sql
SELECT order_no, amount
FROM t_order
WHERE user_id = 1001;
```

这个索引只能用于过滤，不能覆盖返回字段。

---

## 错误 3：为了覆盖索引塞太多字段

```sql
CREATE INDEX idx_big_cover
ON t_order(user_id, create_time, order_no, status, amount, remark, address, phone, ext_json);
```

这样会导致：

```text
索引变大
写入变慢
更新变慢
占用更多磁盘
Buffer Pool 命中率下降
```

覆盖索引不是字段越多越好。

---

# 7. 面试口述版

覆盖索引就是查询需要的字段全部包含在某个索引中。  
在 InnoDB 中，二级索引叶子节点保存的是索引列和主键值，如果查询字段不在二级索引中，就需要通过主键回到聚簇索引查询整行数据，这个过程叫回表。  
为了避免回表，可以把 `WHERE` 条件字段、`ORDER BY` 字段和 `SELECT` 返回字段设计到同一个联合索引中。  
比如查询 `user_id` 下的订单号和金额，可以建立 `(user_id, order_no, amount)` 联合索引。  
执行计划中如果 `Extra` 出现 `Using index`，通常说明使用了覆盖索引。  
但覆盖索引不能滥用，字段太多会增加索引体积，影响写入性能，所以一般只用于高频查询、列表页、分页查询和性能敏感接口。
