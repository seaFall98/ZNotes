# 1. 通用文本 / 对话大模型

这就是最常见的 LLM / Chat Model。

|类型|真实例子|主要用途|
|---|---|---|
|通用对话模型|GPT 系列、Claude、Gemini、Qwen、DeepSeek、Kimi、智谱 GLM|问答、总结、翻译、写作、代码、推理|
|指令模型|Qwen-Instruct、Llama-Instruct、DeepSeek-Chat|按自然语言指令完成任务|
|长上下文模型|Gemini、Claude、Qwen 长上下文版本|读长文档、长代码仓库、长对话|
|推理模型|OpenAI o 系列、DeepSeek-R1、Qwen3 Thinking 类模型|数学、代码、复杂规划、Agent 决策|

OpenAI 官方模型页现在也明确说明其较新的模型普遍支持文本、图像输入以及文本输出，所以“通用模型”和“多模态模型”的边界正在变模糊。([OpenAI 开发者](https://developers.openai.com/api/docs/models?utm_source=chatgpt.com "Models | OpenAI API"))

**你可以理解为：**

```text
ChatGPT / Claude / Gemini / Qwen / DeepSeek
= 通用大脑
= 负责理解问题、生成答案、调用工具、做任务规划
```

# 2. 推理大模型

推理模型不是按模态分，而是按能力分。

|真实例子|特点|用途|
|---|---|---|
|DeepSeek-R1|强化推理、数学、代码|复杂问答、算法、代码分析|
|OpenAI o 系列|偏复杂推理|数学、规划、工程分析|
|Qwen3 Thinking|支持推理模式|复杂逻辑、代码、工具规划|

**典型场景：**

```text
普通模型：你问什么，它直接答。
推理模型：它更适合拆解复杂问题、规划步骤、检查约束。
```

在 Agent 里，推理模型常用于：

- 任务拆解
    
- 工具选择
    
- 多步执行规划
    
- 失败后的恢复策略
    
- 代码修复与架构分析

# 4. 文本嵌入模型：Text Embedding Model

这类模型不负责聊天，而是把文本变成向量。

|真实例子|用途|
|---|---|
|OpenAI text-embedding 系列|语义检索、RAG、聚类|
|Qwen3-Embedding-0.6B / 4B / 8B|中文、英文、多语言检索|
|BGE / bge-large-zh / bge-m3|RAG 常用开源 embedding|
|Jina Embeddings|多语言、长文本检索|
|GTE / GTE-Qwen|通义系 embedding|

OpenAI 官方 embedding 文档把 embedding 定义为把文本转换成数字向量，用于搜索、聚类等任务。([OpenAI 开发者](https://developers.openai.com/api/docs/guides/embeddings?utm_source=chatgpt.com "Vector embeddings | OpenAI API")) Qwen3 Embedding 系列则同时覆盖 embedding 和 reranking，提供 0.6B、4B、8B 不同规模，面向多语言和检索任务。([Hugging Face](https://huggingface.co/Qwen/Qwen3-Reranker-0.6B?utm_source=chatgpt.com "Qwen/Qwen3-Reranker-0.6B"))

**RAG 里它的位置：**

```text
用户问题
  ↓
Embedding 模型转向量
  ↓
Milvus / pgvector / Elasticsearch / FAISS 检索相似 chunk
  ↓
召回候选文档
```

**一句话：**

> Embedding 模型不负责生成答案，它负责“把文本变成可检索的语义坐标”。

# 5. Reranker 重排序模型

Reranker 也不是聊天模型，它负责判断“候选文档和问题到底有多相关”。

|真实例子|用途|
|---|---|
|Qwen3-Reranker-0.6B / 4B / 8B|文本检索精排|
|BGE-Reranker|RAG 常用精排模型|
|Cohere Rerank|商业 Rerank API|
|Jina Reranker|多语言检索精排|
|Qwen3-VL-Reranker|多模态检索精排|

Qwen3 Embedding 论文和模型说明都明确把 Qwen3 系列用于 embedding 与 reranking 任务，并提供多种尺寸以平衡效果和部署成本。([arXiv](https://arxiv.org/abs/2506.05176?utm_source=chatgpt.com "Qwen3 Embedding: Advancing Text Embedding and Reranking Through Foundation Models"))

**RAG 里的位置：**

```text
问题
  ↓
Embedding 粗召回 Top 50
  ↓
Reranker 精排 Top 5
  ↓
LLM 基于 Top 5 生成答案
```

**对比：**

|模型|作用|
|---|---|
|Embedding|快速粗搜|
|Reranker|慢一点，但判断更准|
|LLM|最终组织答案|

# 6. OCR / 多模态解析模型

这就是你提到的 **DeepSeek-OCR** 所在类别。

它不只是“看图聊天”，而是把图片、PDF、扫描件、复杂文档解析成文本或结构化内容。

|真实例子|类型|用途|
|---|---|---|
|DeepSeek-OCR|OCR / 文档解析 / 视觉文本压缩|图片转文字、文档解析、训练数据生成|
|DeepSeek-OCR2|DeepSeek-OCR 后续版本|更强视觉编码/解析|
|PaddleOCR-VL|多模态文档解析|OCR、表格、版面、文档理解|
|MinerU|PDF / 论文 / 文档解析|PDF 转 Markdown、表格/公式解析|
|GOT-OCR2.0|OCR 模型|文档图片文字识别|
|Nougat|学术论文 PDF 解析|论文转结构化文本|
|LayoutLM / LayoutLMv3|版面理解模型|表单、票据、合同抽取|

DeepSeek-OCR 官方 GitHub 在 2025 年 10 月发布 DeepSeek-OCR，并在 2026 年 1 月标注发布 DeepSeek-OCR2；仓库还说明 DeepSeek-OCR 已获得 vLLM 上游支持。([GitHub](https://github.com/deepseek-ai/DeepSeek-OCR?utm_source=chatgpt.com "DeepSeek-OCR: Contexts Optical Compression")) DeepSeek-OCR 论文中提到，它由 DeepEncoder 和 DeepSeek3B-MoE-A570M 解码器组成，目标是研究视觉文本压缩，并在低视觉 token 数下完成 OCR；论文还提到单张 A100-40G 在生产中可用于每天生成 20 万页以上训练数据。([arXiv](https://arxiv.org/abs/2510.18234?utm_source=chatgpt.com "DeepSeek-OCR: Contexts Optical Compression"))

**这类模型在 RAG 里的价值非常大：**

```text
PDF / 扫描件 / 图片 / 表格 / PPT
  ↓
OCR / Layout Parsing / Document Parsing
  ↓
Markdown / JSON / HTML / 结构化文本
  ↓
Chunk 切分
  ↓
Embedding
  ↓
RAG 检索问答
```

**举例：**

```text
你有一批合同 PDF。
不能直接丢给普通 LLM。
正确流程是：

合同 PDF
→ DeepSeek-OCR / MinerU / PaddleOCR-VL 解析
→ 提取标题、条款、表格、签署方、金额、日期
→ 存入知识库
→ Embedding + Rerank
→ LLM 回答合同问题
```

所以 DeepSeek-OCR 可以归类为：

> OCR 模型、文档解析模型、多模态解析模型，也可以说是 VLM 在文档场景下的专用化模型。

# 8. 多模态嵌入模型

这是你列的“多模态嵌入模型”。

它和普通文本 embedding 的区别是：  
它可以把 **文本、图片、音频、视频、文档** 放到同一个向量空间。

|真实例子|能力|
|---|---|
|Gemini Embedding 2|文本、图片、视频、音频、文档统一向量空间|
|Qwen3-VL-Embedding|文本、图片、文档图片、视频检索|
|CLIP|经典图文对齐模型|
|SigLIP|图文 embedding|
|ImageBind|多模态 embedding|
|Jina CLIP / jina-clip|图文检索|
|Nomic Embed Vision|图文检索|

Google 2026 年发布的 Gemini Embedding 2 是原生多模态 embedding 模型，可以把文本、图像、视频、音频和文档映射到统一 embedding 空间；Vertex AI 文档也说明它生成 3072 维向量，并支持图像、文本、文档、音频和视频输入。([blog.google](https://blog.google/innovation-and-ai/models-and-research/gemini-models/gemini-embedding-2/?utm_source=chatgpt.com "Gemini Embedding 2: Our first natively multimodal embedding model")) Qwen3-VL-Embedding / Reranker 报告也把它定义为面向文本、图像、文档图像和视频的统一多模态检索与排序框架。([arXiv](https://arxiv.org/abs/2601.04720?utm_source=chatgpt.com "Qwen3-VL-Embedding and Qwen3-VL-Reranker: A Unified Framework for State-of-the-Art Multimodal Retrieval and Ranking"))

**工程例子：**

```text
用户输入：“找一张蓝色跑车在海边的图片”
  ↓
文本 query 变向量
  ↓
图片库里的图片也变向量
  ↓
相似度检索
  ↓
返回匹配图片
```

这就是：

```text
以文搜图 / 以图搜图 / 以文搜视频 / 多模态 RAG
```

# 13. Agent / 工具调用模型

严格讲，Agent 不是一个模型类别，而是系统形态。  
但很多模型会专门强化工具调用能力。

|真实例子|相关能力|
|---|---|
|GPT 系列工具调用|Function Calling / Tool Calling|
|Claude 工具使用|Tool Use、Computer Use|
|Qwen-Agent / Qwen 工具调用能力|Agent 编排|
|DeepSeek / Qwen / Claude Code 场景|代码工具调用|
|LangChain Agent|工具编排框架|
|LangGraph|可控工作流 Agent|
|AutoGen / CrewAI|多 Agent 协作|

**核心是：**

```text
LLM 不只回答
而是能：
1. 判断要不要调用工具
2. 选择哪个工具
3. 构造参数
4. 读取工具结果
5. 再继续下一步
```

工程例子：

```text
用户：帮我查一下订单 123 是否能退款

Agent：
1. 调用订单查询 API
2. 判断订单状态
3. 调用退款规则 API
4. 如果满足条件，调用退款工具
5. 返回处理结果
```

|Preference Model|偏好学习|学用户偏好|

在企业项目里，它可以回答：

```text
这个答案是否引用了正确文档？
这个答案有没有幻觉？
这个工具调用是否成功？
这个 Agent 轨迹是否合理？
这个客服回答是否合规？
```

# 面试版回答

可以这样说：

> 大模型可以按模态、任务和场景分类。按模态有文本大模型、语音模型、图像/视频生成模型、多模态理解模型；按任务有通用对话模型、推理模型、代码模型、Embedding 模型、Reranker 模型、OCR/文档解析模型、安全审核模型、评估模型；按场景有通用大模型、垂直领域大模型、科学大模型、时间序列模型、推荐排序模型和端侧小模型。
> 
> 例如做企业 RAG，不是只用一个 Chat 模型，而是会组合 DeepSeek-OCR 或 MinerU 做文档解析，Qwen3-Embedding 或 BGE 做向量化，Qwen3-Reranker 或 BGE-Reranker 做精排，最后用 GPT、Qwen、DeepSeek、Claude 这类通用模型生成答案，再用 Guardrail 或 Evaluator 模型做安全和质量检查。

# 你当前最该重点掌握的模型类型

按 AI 应用工程优先级：

|优先级|类型|你要掌握到什么程度|
|---|---|---|
|P0|通用 LLM|会调用 API、设计 Prompt、处理上下文|
|P0|Embedding|明白向量化、相似度、chunk 检索|
|P0|Reranker|明白粗召回 + 精排|
|P0|OCR / 文档解析|知道 DeepSeek-OCR、MinerU、PaddleOCR-VL 在 RAG 前处理中的作用|
|P1|多模态理解|会处理图片、截图、PDF、图表|
|P1|Agent / Tool Calling|会接 API、MCP、工具、工作流|
|P1|Eval / Guardrail|会评估幻觉、忠实性、安全性|
|P2|代码模型|会用 AI Coding 改造工程效率|
|P2|语音模型|了解 ASR、TTS、实时语音助手|
|P3|图像/视频/科学/时间序列|看具体业务再深入|

一句话总结：

> 真实 AI 应用里的“大模型”不是一个模型，而是一组模型流水线：解析模型处理原始资料，Embedding 模型负责召回，Reranker 负责精排，LLM 负责生成，Guardrail 和 Evaluator 负责安全与质量，多模态模型负责处理图片、视频、音频等复杂输入。
