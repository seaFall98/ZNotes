```

这是langchain的官方文档,提到结构化输出可以用tool strategy,我使用DeepSeek的deepseek-v4-flash模型通过ToolStrategy结构输出时，报错Thinking mode does not support this tool choice 然后我禁用了思考模式才生效了,但是现在模型好像默认都是思考模式,思考模式它的输出质量也会更高吧,为什么会有这种官方文档和实际模型冲突呢？结构化输出现在还适用吗？请查阅最新的文档和工程实践看看到底是怎么回事
```

结论：**这不是你用错了，而是 LangChain 文档里的“ToolStrategy 适用于支持 tool calling 的模型”这句话，在 DeepSeek V4 thinking mode 上遇到了一个细分能力缺口：它支持工具调用，但不支持“强制工具调用”的 `tool_choice`。而 LangChain 的结构化输出恰恰常常依赖强制调用一个“伪工具”来拿到稳定 JSON。**

### 发生冲突的根因

LangChain 的 `ToolStrategy` 本质是：把你的 Pydantic / JSON Schema 包装成一个“人工工具”，让模型通过 tool call 返回结构化参数。官方文档也明确写了：`ToolStrategy` 使用 tool calling 做结构化输出；如果模型不支持 provider-native structured output，LangChain 会退到 `ToolStrategy`。同时文档也提醒：如果同时使用工具和结构化输出，模型必须支持二者同时使用。([LangChain 文档](https://docs.langchain.com/oss/python/langchain/structured-output "Structured output - Docs by LangChain"))

问题是 DeepSeek V4 的 thinking mode 现在是一个“半兼容”状态：

DeepSeek 官方文档说 thinking mode 支持 tool calls，并且从 DeepSeek-V3.2 起 thinking mode 支持工具使用。([DeepSeek API Docs](https://api-docs.deepseek.com/guides/thinking_mode "Thinking Mode | DeepSeek API Docs")) 但 DeepSeek 自己的 Oh My Pi 集成文档又明确要求设置 `supportsToolChoice: false`，原因是 **DeepSeek V4 thinking mode rejects the `tool_choice` parameter**；还要求在工具调用后保留并回传 `reasoning_content`，否则会 400。([DeepSeek API Docs](https://api-docs.deepseek.com/quick_start/agent_integrations/oh_my_pi "Using DeepSeek with Oh My Pi | DeepSeek API Docs"))

所以这里的关键差异是：

|能力|DeepSeek V4 thinking mode|
|---|---|
|普通 tool calling，模型自己决定是否调用工具|支持|
|`tool_choice="auto"`|通常支持|
|`tool_choice="required"`|不支持 / 400|
|`tool_choice={"type":"function", "function": {"name": ...}}` 强制指定某个工具|不支持 / 400|
|LangChain ToolStrategy 想强制模型调用结构化输出伪工具|因此容易失败|

社区最新 issue 也复现了你看到的错误：`deepseek-v4-pro` 和 `deepseek-v4-flash` 在 thinking mode 下会拒绝 `tool_choice="required"` 和指定函数形式的 `tool_choice`，报错就是 `"Thinking mode does not support this tool_choice"`；同一份测试显示 `deepseek-chat` / V3.2 不受这个限制。([GitHub](https://github.com/deepseek-ai/DeepSeek-V3/issues/1376 "[BUG] DeepSeek V4 rejects tool_choice=\"required\" and specific function tool_choice — breaks structured output in all agent frameworks · Issue #1376 · deepseek-ai/DeepSeek-V3 · GitHub"))

### 为什么“官方文档”和实际模型会冲突？

严格说，**LangChain 文档没有完全错，DeepSeek 文档也没有完全错，但两边抽象层级不一样**。

LangChain 说的是：“只要模型支持 tool calling，就可以用 ToolStrategy。”这在大多数模型上成立，因为很多模型不仅支持工具调用，也支持 `tool_choice` 强制调用某个工具。DeepSeek 说的是：“thinking mode 支持 tool calls。”这也成立，因为模型可以自己选择工具调用。

但结构化输出不是普通工具调用。结构化输出需要高可靠性，框架通常会强制模型调用那个 schema tool。否则模型可能直接输出自然语言，或者输出一段看似 JSON 的文本，导致 parser 拿不到 `tool_calls`。GitHub 上的最新 DeepSeek V4 复现报告也指出：LangChain `with_structured_output(method="function_calling")` 会发送指定函数形式的 `tool_choice` 来强制结构化输出，因此 V4 thinking mode 会失败；如果禁用 `tool_choice`，又会变成模型“可能不调用工具”，结构化输出可靠性下降。([GitHub](https://github.com/deepseek-ai/DeepSeek-V3/issues/1376 "[BUG] DeepSeek V4 rejects tool_choice=\"required\" and specific function tool_choice — breaks structured output in all agent frameworks · Issue #1376 · deepseek-ai/DeepSeek-V3 · GitHub"))

### 结构化输出现在还适用吗？

**适用，但要按模型能力选择策略。**

对 DeepSeek V4 Flash / Pro，我会这样分层：

#### 1. 需要强约束 JSON / Pydantic 校验：优先关闭 thinking mode

这是你现在验证成功的方案，也是目前最稳的工程方案。因为关闭 thinking 后，`tool_choice` 限制消失或至少更容易走传统 function calling 路径。

大致思路：

```python
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(
    model="deepseek-v4-flash",
    base_url="https://api.deepseek.com",
    api_key=DEEPSEEK_API_KEY,
    extra_body={"thinking": {"type": "disabled"}},
)

structured_llm = llm.with_structured_output(MySchema, method="function_calling")
```

适合：抽取、分类、路由、写库、工作流状态机、交易/风控/自动化任务。  
不适合：你特别需要长推理质量的复杂规划任务。

#### 2. 需要 thinking mode 的推理质量：不要依赖 ToolStrategy 强制结构化

可以让 thinking mode 做推理，但最终输出用 **JSON mode / prompt JSON / 后处理校验重试**。DeepSeek 官方还提供了 `strict` function schema beta，且文档说 strict mode 同时支持 thinking 和 non-thinking mode；不过它依然是工具 schema 严格性，不等于支持 `tool_choice` 强制调用。([DeepSeek API Docs](https://api-docs.deepseek.com/guides/tool_calls "Tool Calls | DeepSeek API Docs"))

工程上可以这样做：

```python
# 第一段：thinking mode 做复杂推理，输出自然语言或半结构化结论
reasoning_result = reasoning_llm.invoke(...)

# 第二段：关闭 thinking，用便宜/快模型做结构化整理
json_result = extractor_llm.with_structured_output(MySchema).invoke(
    f"把下面内容转成严格结构化对象：\n{reasoning_result.content}"
)
```

这其实是我更推荐的生产模式：**推理模型负责想清楚，非 thinking / extractor 模型负责按 schema 落库**。

#### 3. 勉强保留 thinking + tools：禁用 `tool_choice`，但要接受不稳定

社区 workaround 是不要发送 `tool_choice`，例如 LangChain/OpenAI 兼容层里禁用这个参数。但这样模型只是“有工具可用”，不是“必须调用结构化输出工具”。结果就是：有时能得到 tool call，有时会返回普通文本。最新 issue 里的实测也提到这会导致结构化输出 fallback。([GitHub](https://github.com/deepseek-ai/DeepSeek-V3/issues/1376 "[BUG] DeepSeek V4 rejects tool_choice=\"required\" and specific function tool_choice — breaks structured output in all agent frameworks · Issue #1376 · deepseek-ai/DeepSeek-V3 · GitHub"))

这个方案适合实验，不适合关键生产路径。

### thinking mode 的输出质量一定更高吗？

不一定。DeepSeek 官方说 thinking mode 会先输出推理内容以提升最终回答准确性。([DeepSeek API Docs](https://api-docs.deepseek.com/guides/thinking_mode "Thinking Mode | DeepSeek API Docs")) 但工程上要分任务：

复杂推理、代码分析、多步规划、数学题：thinking mode 通常更有价值。  
简单抽取、固定 schema 分类、信息搬运、数据库字段填充：thinking mode 不一定更好，反而更慢、更贵、更容易和工具协议冲突。  
结构化输出任务：**稳定性通常比“更会想”更重要**。一个 95% 正确但偶尔不按 schema 的模型，在自动化系统里可能不如一个 90% 正确但 99.9% 遵守 schema 的模型。

### 我给你的实践建议

当前 DeepSeek V4 Flash + LangChain 结构化输出，我会这样选：

|场景|推荐|
|---|---|
|严格结构化抽取|关闭 thinking，用 `with_structured_output` / `ToolStrategy`|
|agent 里有真实 tools + 最终要结构化|尽量拆成两步：agent thinking 处理任务，最后单独 extractor 结构化|
|必须 thinking + tools|不传 `tool_choice`，并实现重试、parser fallback、`reasoning_content` roundtrip|
|对 schema 合规要求极高|用 provider-native structured output 的模型，或 DeepSeek 非 thinking + 严格校验|
|只是想拿 JSON|JSON mode / prompt JSON + Pydantic 校验重试也可用，但不要当成强保证|

### 最关键的一句话

**LangChain 的 ToolStrategy 仍然适用；不适用的是“DeepSeek V4 thinking mode + 强制 tool_choice + LangChain ToolStrategy”这个组合。**

你禁用 thinking mode 后生效，正好说明你的 schema、LangChain 用法大概率没问题；问题在 DeepSeek V4 thinking mode 当前不接受 LangChain 为了结构化输出而发送的强制 `tool_choice`。
