
## 简答

MCP 工具过多时，问题通常不在 MCP Server，而在 Host 把所有 `tools/list` 返回的完整工具定义一次性塞给模型，导致工具 Schema 占用大量上下文，并降低模型选工具的准确率。生产环境通常在 Host 或 Agent Harness 层实现 progressive tool discovery：模型一开始只看到一个轻量的工具搜索入口，需要时再加载候选工具的完整 Schema，最后只执行真正需要的工具。MCP 协议提供工具发现，但不会强制规定这套渐进式加载策略，因此它属于 Host/Harness 的职责。[MCP 官方客户端最佳实践](https://modelcontextprotocol.io/docs/develop/clients/client-best-practices)

## anthropic的一种新思路 跟deepagent的代码解释器很像
https://www.anthropic.com/engineering/code-execution-with-mcp

Anthropic 的解决方案不是让模型一次性理解更多 MCP 工具，而是把 MCP 工具转换成模型可以按需探索和调用的代码 API。

传统方式：

```
所有 MCP 工具定义
    ↓
一次性塞进模型上下文

模型调用工具
    ↓
完整结果返回模型上下文

模型再调用下一个工具
```

这会产生两个问题：

1. 几百、几千个工具的定义占满上下文。
2. 工具之间传递的大量中间结果反复经过模型。

Anthropic 提出的方式是：

```
MCP Server
    ↓
转换成代码文件和目录
    ↓
Agent 先探索目录
    ↓
只读取需要的工具定义
    ↓
在沙箱中编写代码调用多个 MCP 工具 ## deepagent的code interpreter?
    ↓
只把最终结果返回模型
```

例如，传统 Agent 处理“从 Google Drive 读取会议记录，再写入 Salesforce”：

```
调用 getDocument
    ↓
完整会议记录进入模型上下文
    ↓
模型把会议记录重新写进 updateRecord 参数
    ↓
完整会议记录再次进入上下文
```

Anthropic 的代码执行方式：

```
import * as gdrive from "./servers/google-drive";
import * as salesforce from "./servers/salesforce";

const transcript = (
  await gdrive.getDocument({ documentId: "abc123" })
).content;

await salesforce.updateRecord({
  objectType: "SalesMeeting",
  recordId: "00Q5f000001abcXYZ",
  data: {
    Notes: transcript,
  },
});

console.log("Salesforce 更新完成");
```

这里 `transcript` 在代码执行环境中从 Google Drive 流向 Salesforce，**不会完整进入模型上下文**。模型最终只看到：

```
Salesforce 更新完成
```

所以 Anthropic 这篇文章里的核心方案不是单纯的 `search_tools`，而是：

> **Progressive disclosure + Code Execution**

两者解决的问题不同：

- `search_tools`：减少工具定义进入上下文的数量。
- Code Execution：同时减少工具定义、中间结果和多轮工具调用过程进入上下文的数量。

Anthropic 还强调，代码执行带来的收益不只是节省 Token：

- 可以在沙箱里过滤一万行数据，只返回五行结果。
- 可以用循环、条件、异常处理组织复杂流程。
- 敏感数据可以只在执行环境中流转，不进入模型上下文。
- 可以把中间结果写入文件，后续继续使用。
- 成功的代码还可以沉淀为可复用的 Skill。

但代价是必须建设可靠的执行环境，包括：

```
沙箱隔离
资源限制
权限控制
超时
日志与监控
代码执行审计
```

因此，Anthropic 的结论可以概括为：

> MCP 负责连接外部系统，代码执行环境负责承载工具调用和数据处理，Agent 只负责生成代码和查看最终结果。

这篇文章的核心不是“如何定义更多 MCP Tool”，而是“不要让模型亲自搬运所有工具结果，让代码执行环境替模型完成数据搬运和流程控制”。[Anthropic: Code execution with MCP](https://www.anthropic.com/engineering/code-execution-with-mcp)


## deepagent的代码解释器只是方案的一步

**DeepAgents 最新的 Interpreter，本质上就是同一类“Programmatic Tool Calling / Code Mode”思想**。

两者的共同点是：

```
模型不直接一轮一轮调用工具
    ↓
模型编写一段代码
    ↓
代码在受限运行时中调用多个工具
    ↓
中间结果留在运行时
    ↓
模型只看到最终输出
```

**对应关系**

|Anthropic 文章|DeepAgents|
|---|---|
|Code execution environment|`Interpreter`|
|MCP 工具变成代码 API|MCP 工具经适配后成为 LangChain tools|
|沙箱中执行代码|`langchain_quickjs` 的 QuickJS 运行时|
|代码调用多个工具|`CodeInterpreterMiddleware` + PTC|
|中间结果不回模型|Interpreter 内存中处理，只返回最终结果|

DeepAgents 的典型形式是：

```
from deepagents import create_deep_agent
from langchain_quickjs import CodeInterpreterMiddleware

agent = create_deep_agent(
    model="your-model",
    tools=[
        get_document,
        update_salesforce,
    ],
    middleware=[
        CodeInterpreterMiddleware(
            ptc=["get_document", "update_salesforce"]
        )
    ],
)
```

模型在 Interpreter 中执行的逻辑类似：

```
const document = await tools.getDocument({
  documentId: "abc123",
});

await tools.updateSalesforce({
  recordId: "C1001",
  notes: document.content,
});

console.log("更新完成");
```

关键收益是 `document.content` 不需要先完整返回给模型，再由模型复制到第二个 Tool 中，而是在 Interpreter 内部直接传递。

但两者还有一个重要区别：

- Anthropic 文章重点讨论的是完整的 Agent Harness：工具目录、按需发现、代码执行、沙箱、数据流控制、状态持久化。
- DeepAgents 的 Interpreter 重点是“在 Agent 循环中用代码编排工具和处理中间结果”。

因此，DeepAgents Interpreter **主要解决工具结果和多步调用造成的上下文膨胀**。如果你把 1000 个 MCP 工具全部通过 `client.get_tools()` 交给 Agent，它本身不会自动解决所有工具定义的上下文问题。工具发现、筛选、按领域加载，仍然需要 Harness 层设计。

另外，DeepAgents 当前官方文档明确区分：

- `Interpreter`：内存中的轻量代码执行，默认不能访问文件系统、网络或 Shell。
- `Sandbox`：面向命令执行、依赖安装、文件编辑等完整环境操作。

所以更准确地说：

> Anthropic 提出的是整体架构思想；DeepAgents Interpreter 是这个思想在 LangChain/DeepAgents 体系中的一个具体实现，尤其对应其中的 PTC 和上下文管理部分。

[DeepAgents Interpreters](https://docs.langchain.com/oss/python/deepagents/interpreters)  
[LangChain MCP 集成](https://docs.langchain.com/oss/python/langchain/mcp)


## 常规回答

MCP 最初的设计哲学确实是“全量预加载（Eager Loading）”，即把所有工具定义一次性塞进上下文，这在工具数量少时很高效，但一旦工具变多，就会引发严重的“上下文爆炸”和模型“认知过载”。

不过，在 2026 年的工程实践中，业界已经针对 MCP 的上下文问题总结出了一套成熟的解决方案。虽然它不像 Skills 那样自带完美的“渐进式披露”机制，但通过以下四种核心策略，依然能有效解决工具过多的问题：

### 1. 引入 Tool Search（工具搜索）机制

这是目前最直接的“补丁式”解决方案。以 Anthropic 在 Claude Code 中引入的 Tool Search 为例，它改变了 MCP 工具必须预加载的规则：

- **按需加载**：不再将所有工具定义一次性注入上下文。当工具定义占用超过上下文窗口的 10% 时，系统会自动启用搜索机制。
- **先搜后调**：AI 需要使用工具时，先通过搜索找到对应的工具定义，再将其加载到上下文中。实测显示，这种机制能将上下文 Token 消耗减少 85% 以上。

### 2. 按领域拆分 MCP Server（工具分组）

不要把几十个工具塞进一个“大而全”的单体 MCP Server 中。工程上的最佳实践是按业务领域进行拆分：

- **拆分策略**：将原本包含 30 个工具的单体 Server，拆分为 3-4 个独立的 Server（如 `dev-tools`、`data-tools`、`search-tools`），每个 Server 仅暴露 8-12 个工具。
- **降维打击**：这种拆分让模型的决策变成了“先选领域，再选工具”。实测证明，将 20 个工具拆成 3 个 Server 后，工具调用的准确率可以从 71.7% 提升到 83.5%。

### 3. 两阶段精简描述法（粗筛与精调）

为了减少工具定义对上下文的挤占，可以在架构上采用两阶段过滤：

- **第一阶段（粗筛）**：只给 LLM 提供所有工具的“无参数 + 一行描述”版本。LLM 基于这个极轻量的信息快速过滤，输出 2-3 个候选工具。
- **第二阶段（精调）**：仅对被选中的候选工具，提供完整的参数定义和详细说明，让 LLM 做最终决策并精确调用。
- **极致压缩**：此外，还要将工具描述缩减到极致（例如从平均 87 个 Token 压到 20 个），描述只说明“做什么”，不教“怎么用”。

### 4. 结合 Skills 实现外部工具的渐进式披露

这正是 Skills 和 MCP 协同作战的最佳场景。MCP 工具可以绑定到 Skill 的作用域中，从而“借用” Skills 的渐进式披露机制：

- **平时隐藏**：未激活 Skill 时，其绑定的 MCP 工具不会暴露给 LLM。
- **按需激活**：当 LLM 识别到相关任务并激活某个 Skill 时，对应的 MCP 工具才会被动态加载到上下文中。
- **效果**：如果你有 10 个 MCP 服务，每个提供 10 个工具，一次性暴露会消耗大量 Token；但如果打包成 10 个 Skill，LLM 初始只看到 10 个 Skill 的元数据，需要时才激活对应的 10 个工具，完美解决了多 MCP 服务的工具膨胀问题。

**总结来说：** MCP 解决上下文问题的核心思路是**“从全量预加载向按需演进”**。你可以通过 Tool Search 实现动态发现，通过拆分 Server 和精简描述降低基础消耗，或者更优雅地——**将 MCP 工具封装进 Skills 中**，让 AI 像查阅手册一样，用到哪本手册，再加载里面的工具。
