
# MCP 工具、资源与 Code Execution 复习

## 一、面试回答

MCP Server 可以暴露 Tools、Resources 和 Prompts 三类能力。Tool 是让模型决定执行一个动作，Resource 是让 Host 或应用按地址读取一份上下文资料，Prompt 是可被用户选择的可复用提示模板。

MCP 工具过多时，主要有两类上下文问题：一是所有工具定义和 Schema 一次性注入模型，二是多个工具之间的大量中间结果反复经过模型。生产环境应该由 Host 或 Agent Harness 管理工具发现、筛选和执行，而不是把所有 MCP 工具直接绑定给模型。可以使用渐进式工具发现，只暴露工具目录和搜索入口，按需加载完整定义；复杂流程则可以让模型编写代码，在受限运行时中调用多个工具，只把最终结果返回模型。

Anthropic 的 Code Execution with MCP 进一步把 MCP 工具转换成代码 API，让 Agent 在代码执行环境中探索、组合和调用工具。DeepAgents 当前的 Interpreter 和 Programmatic Tool Calling 采用了相同的核心思想：中间数据留在运行时内，避免每一步都回到模型上下文。但 DeepAgents Interpreter 主要解决工具编排和中间结果膨胀，工具定义的筛选和按需加载仍然属于 Harness 层的职责。

## 二、Tools、Resources、Prompts

|类型|本质|协议入口|通常由谁控制|例子|
|---|---|---|---|---|
|Tool|动作、函数、能力|`tools/call`|模型决定是否调用|搜索订单、取消订单、更新 CRM|
|Resource|可寻址的数据或上下文|`resources/read`|Host 或应用决定是否读取|订单详情、规章制度、配置文件、项目文档|
|Prompt|可复用的消息模板|`prompts/get`|用户选择|订单分析模板、代码审查模板|

关键记忆方式：

Tool     = 做一件事  
Resource = 提供一份材料  
Prompt   = 组织一次任务

### 为什么 Resource 不一定要单独存在

如果只是搜索订单并返回少量订单数据，一个 `search_orders` Tool 完全够用：

@mcp.tool()  
def search_orders(status: str) -> list[dict]:  
    return [  
        {"order_id": "A1001", "status": "shipping"},  
    ]

Resource 的额外价值出现在资料较大、需要反复复用，或者应用希望控制何时把内容放进模型上下文的场景：

@mcp.tool()  
def search_orders(status: str) -> list[dict]:  
    # 只返回摘要和资源地址  
    return [  
        {  
            "order_id": "A1001",  
            "status": "shipping",  
            "uri": "order://A1001",  
        }  
    ]  
​  
​  
@mcp.resource("order://{order_id}")  
def get_order_detail(order_id: str) -> str:  
    # Host 或应用需要时再读取完整内容  
    return f"订单 {order_id} 的完整详情"

同一份资料可以被多个 Agent、编辑器或 UI 按 URI 读取，不必每次都通过 Tool 临时执行查询。Resource 不是必须的，也不一定是静态文件；它可以在读取时实时访问数据库。区别在于访问语义和控制权，而不是函数体是否都能查询数据库。

## 三、工具过多的上下文问题

### 1. 工具定义膨胀

如果 Host 把所有 `tools/list` 返回的完整工具定义直接放入模型的工具参数，工具数量越多，名称、描述、输入 Schema 和输出 Schema 占用的上下文越多，也会增加模型选错工具的概率。

### 2. 工具结果膨胀

传统流程中：

Tool A 返回完整数据  
    -> 数据进入模型上下文  
模型复制数据  
    -> 调用 Tool B  
    -> 数据再次进入上下文

解决工具定义和解决工具结果是两件事：

- 工具定义问题：工具目录、搜索、按需加载、领域拆分、动态启用。
    
- 工具结果问题：摘要、结构化输出、Resource URI、文件落盘、代码执行和中间结果过滤。
    

## 四、渐进式工具发现

MCP 协议本身提供工具发现，但渐进式加载主要由 Host 或 Harness 实现。常见的三层模式是：

Catalog：搜索工具名称和简短描述  
    -> Inspect：加载候选工具的完整 Schema  
        -> Execute：调用最终选中的工具

Host 一开始只给模型暴露轻量入口，例如：

search_tools(query)  
get_tool_details(name)

模型先搜索相关工具，Host 再加载候选工具的完整定义，最后只把当前任务需要的工具交给模型执行。

除此之外，还可以：

- 按业务边界拆分 MCP Server
    
- 只连接当前任务需要的 Server
    
- 给不同 Agent 或子 Agent 配置不同的工具白名单
    
- 缓存工具 Schema，并监听 `notifications/tools/list_changed`
    
- 使用 Skill 作为领域路由，决定应该加载哪个 Server 或工具集合
    

## 五、Anthropic 的 Code Execution with MCP

Anthropic 文章的核心观点是：不要让模型通过多轮直接 Tool Call 搬运所有数据，而是把 MCP Server 暴露成代码 API，让 Agent 编写代码在执行环境中组合工具。

传统方式：

gdrive.getDocument  
    -> 完整会议记录进入模型上下文  
salesforce.updateRecord  
    -> 模型把会议记录复制进参数  
    -> 会议记录再次经过模型上下文

Code Execution 方式：

const transcript = (  
  await gdrive.getDocument({ documentId: "abc123" })  
).content;  
​  
await salesforce.updateRecord({  
  objectType: "SalesMeeting",  
  recordId: "00Q5f000001abcXYZ",  
  data: { Notes: transcript },  
});  
​  
console.log("Salesforce 更新完成");

会议记录在执行环境中直接从 Google Drive 流向 Salesforce，模型只看到最终输出。

Anthropic 文章强调的收益包括：

- 按需探索工具定义，而不是一次性加载全部工具
    
- 在执行环境中过滤、聚合和转换大数据集
    
- 用代码表达循环、条件、重试和错误处理
    
- 让敏感中间数据留在执行环境中
    
- 把中间结果写入文件，支持状态持久化
    
- 把成功的代码沉淀为可复用的 Skill
    

代价是需要可靠的执行基础设施：沙箱隔离、权限、资源限制、超时、监控和审计。

## 六、与 DeepAgents Interpreter 的关系

DeepAgents 当前官方文档中的 `Interpreter` 和 `Programmatic Tool Calling` 与 Anthropic 的核心模式高度相似：模型编写代码，运行时调用允许的工具，中间结果保留在运行时，最终只把必要信息返回模型。

示意写法：

from deepagents import create_deep_agent  
from langchain_quickjs import CodeInterpreterMiddleware  
  
agent = create_deep_agent(  
    model="your-model",  
    tools=[get_document, update_salesforce],  
    middleware=[  
        CodeInterpreterMiddleware(  
            ptc=["get_document", "update_salesforce"]  
        )  
    ],  
)

Interpreter 中的代码可以类似这样调用：

const document = await tools.getDocument({  
  documentId: "abc123",  
});  
  
await tools.updateSalesforce({  
  recordId: "C1001",  
  notes: document.content,  
});  
  
console.log("更新完成");

需要区分两者：

- Anthropic 文章讨论的是更完整的 Code Execution with MCP 架构，包括工具代码目录、按需发现、数据流控制、沙箱和状态管理。
    
- DeepAgents Interpreter 主要解决 Agent 循环中的工具编排、中间结果处理和状态保留。
    
- DeepAgents Interpreter 当前是基于 QuickJS 的内存运行时，默认不能访问宿主机文件系统、网络、Shell 或包管理器。
    
- MCP 工具需要先通过 LangChain MCP Adapter 转换成 LangChain tools，再按 PTC 白名单开放给 Interpreter。
    
- 如果把所有 MCP 工具一次性通过 `get_tools()` 直接交给 Agent，Interpreter 不会自动替你完成工具目录筛选；工具发现和动态加载仍需要 Harness 设计。
    

可以记成：

MCP       = 连接外部系统  
LangChain = 适配 MCP 工具  
DeepAgents Interpreter = 在 Agent 循环中编排工具和处理中间数据  
Harness   = 负责发现、筛选、权限、生命周期和上下文治理

## 七、易错点

1. 不要认为 Resource 一定比 Tool 更高级。简单场景只用一个搜索 Tool 就够了。
    
2. 不要把“工具定义太多”和“工具结果太大”当成同一个问题。
    
3. `search_tools` 可以解决工具定义的渐进式发现，但不自动解决多轮调用的中间结果膨胀。
    
4. Interpreter/PTC 可以减少中间结果回到模型上下文，但不等于自动完成所有 MCP 工具的动态发现。
    
5. Interpreter 不是天然的完整安全沙箱；工具桥接、权限、超时和审计仍需要工程治理。
