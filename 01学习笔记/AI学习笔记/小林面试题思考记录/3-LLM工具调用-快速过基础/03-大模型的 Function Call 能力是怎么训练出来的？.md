
## 简答

FunctionCall的能力主要靠两个训l练阶段来培养，这两个阶段解决的是不同的问题。

第一个是SFT，就是给模型喂大量「包含工具调用的完整对话样本」，每条样本覆盖工具定义、用户问题、模型应该输出的结构化JSON调用、工具执行结果、最终答案，让模型通过模仿学会整套流程。但光有SFT不够，模型可能学得过激，遇到什么问题都想调工具。

第二个阶段是RLHF，通过人类标注「哪种回答更好」来训练奖励模型，再用强化学习调整主模型，让它学会「能直接回答的就直接回答，需要实时数据才去调工具」这个边界感。

一句话总结：SFT教会怎么调，RLHF教会什么时候调。

---

为什么预训练学不会这件事？关键原因是预训练语料里压根没有「标准工具调用JSON」这种模式。预训练喂给模型的是互联网上的海量文本，里面有代码、有文档、有对话，但几乎没有「给定一组工具schema，该在什么场景输出什么JSON」这样的成对样本。

训练数据来源也要提到：人工标注质量高但成本高用于种子数据，模型自动生成（Self-Instruct/
Distillation）成本低量大但要注意幻觉传递的风险。

---

## 详细

大模型的 Function Call（工具调用）能力并非在预训练阶段自然学会的，而是通过一套完整的**“专项训练 + 评估 + 工程约束”**体系构建出来的。

结合当前的技术现实与工程实践，Function Call 的训练主要包含以下几个核心环节：

### 1. 核心训练路径：SFT 与 RLHF/DPO 的协同

业界有一句高度概括的总结：**SFT 解决“会不会调”，RLHF/DPO 解决“该不该调”。**

- **监督微调（SFT）—— 解决“会不会调”：** 预训练只能让模型理解 API 文档和代码片段，但无法让其稳定输出可执行的 `tool_calls` JSON。SFT 阶段通过向模型喂入大量完整的工具调用对话样本，让模型学会工具定义、参数抽取、结构化输出以及工具结果回填。模型本质上学习的是“遇到特定意图时，按特定格式输出指令”的模式匹配。
- **强化学习与偏好对齐（RLHF / DPO）—— 解决“该不该调”：** 模型不仅要会调工具，还要具备边界意识。通过 RLHF 或 DPO 进行偏好对齐，教会模型“该调才调，不该调就直接回答”，避免过度触发工具调用。例如 Llama 3 在 Post-training 阶段就包含了多个 SFT 和 DPO 的迭代过程，且 DPO 的表现优于 PPO。

### 2. 训练数据的构建与组织

高质量的数据是 Function Call 能力的基石。训练数据不仅需要覆盖单工具、多工具并行、多轮上下文等场景，还必须包含异常样本。

- **负样本与失败样本至关重要：** 不要只收集成功样本，更要收集失败样本（如无效参数、工具报错等）。因为线上真正决定系统稳定性的，往往是“异常路径能不能兜住”。
- **数据组织格式：** 工具的名称、描述和参数结构（JSON Schema）会被翻译成文本，像 System Prompt 一样注入到模型的输入上下文中。模型依靠这些语义信息进行意图匹配。
- **难度与质量建模：** 在数据清洗阶段，业界会采用联合筛选机制（如意图密度分析、难度分级、语义去重等），保留最具代表性的样本，从而显著提升模型在复杂推理任务上的准确率。

### 3. 工程层面的约束与护栏

Function Call 的可靠性最终是“模型训练”与“工程护栏”共同决定的。训练完并不等于结束，线上还需要一系列工程机制来保障：

- **解码约束（Decoder Constraints）：** 通过语法约束或 JSON Schema 约束采样，从 Token 层面限制模型只能输出合法的 JSON 结构，杜绝格式错误。
- **工程护栏：** 上线前和运行中需要做 Schema 校验、工具白名单、权限控制、调用日志、失败回流和离线评估等，确保模型输出的安全性与合规性。

**总结来说：** 真正靠谱的 Function Call，不是“给模型几个工具就完事”，而是让模型在大量高质量样本里学会动作格式（SFT），在偏好对齐里学会决策边界（RLHF/DPO），并在工程系统里受到严格的校验和保护。
