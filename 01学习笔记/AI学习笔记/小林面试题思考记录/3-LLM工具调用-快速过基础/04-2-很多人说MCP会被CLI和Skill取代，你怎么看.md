
## 简答

首先得说澄清 是cli调用工具还是mcp来调用工具，而不是整个cli

我认为这句话有一半对：MCP 不会被 CLI 和 Skill “整体取代”，但会被重新定位。

可以把三者看成不同层次：

- **CLI**：执行层，适合本机、确定性强、可脚本化的操作。
- **Skill**：知识与流程层，告诉模型“什么时候做、怎么判断、按什么步骤做”。
- **MCP**：连接层，把外部系统的资源和操作以统一协议暴露给模型。

所以更可能出现的组合是：

```
Skill 负责决策和流程
  -> CLI 负责本地执行
  -> MCP 负责访问远程服务、权限系统和结构化资源
```

CLI 确实会替代一部分 MCP：例如 Git、数据库迁移、测试、文件处理等，本来就有成熟 CLI，直接调用更简单、性能更好、调试也更透明。Skill 也会减少“把简单脚本包装成 MCP Server”的需求。

但对于 SaaS、企业内部系统、需要 OAuth/权限控制/资源发现/多轮交互的场景，MCP 仍然有价值。它解决的不是“如何执行一个命令”，而是“模型如何以统一方式理解并操作一个外部系统”。直接让模型拼 CLI 命令，在权限隔离、参数约束、返回结构和安全审计上往往更脆弱。

真正会被淘汰的，可能是“为了任何小功能都写一个 MCP Server”的做法。未来比较合理的判断标准是：

- 本地工具、批处理、开发工具：优先 CLI。
- 固定领域知识、复杂工作流：优先 Skill。
- 远程服务、结构化资源、身份权限和多客户端接入：优先 MCP。
- 三者结合：通常是最完整的工程方案。

因此，MCP 更像是从“万能插件机制”回归为“标准化外部系统接口”，而不是消失。

## 详细

关于“MCP 会被 CLI（命令行接口）和 Skills（技能）取代”的说法，在 2025 到 2026 年的 AI 社区中确实引发了激烈的讨论。但基于当前的工程实践和技术架构，**这三者并非“谁取代谁”的零和博弈，而是处于不同层级、各司其职的互补关系**。

未来的智能体架构必然是这三者的融合。我们可以从它们的本质区别和协同机制来看待这个问题：

### 1. 为什么会有“CLI 取代 MCP”的争论？

CLI 的复兴主要源于其在执行复杂任务时的效率和成本优势：

- **Token 消耗极低**：MCP 需要将大量工具的定义（名称、参数描述）塞入上下文窗口，导致 Token 消耗极高（有实测显示 MCP 的消耗量可达 CLI 模式的 20 倍）。而 CLI 依赖大模型本身训练时掌握的知识，Token 消耗极低。
- **组合能力强**：CLI 可以通过管道符（如 `jq`、`grep`）像搭乐高一样组合指令，一次性解决复杂的工作流任务，而 MCP 往往需要模型进行多轮次的往返调用。

**但 CLI 无法取代 MCP 的核心原因在于“安全性”与“企业级治理”。** CLI 具有天然的权限劣势，复杂的 CLI 指令背后容易隐藏误删文件等潜在风险，且高权限极易被注入漏洞。因此，CLI 更适合灵活度高、安全要求低的个人开发者场景；而在企业级应用中，MCP 凭借其中转机制、OAuth 授权、沙箱隔离和审计日志，依然是保障数据边界和控制安全的首选。

所以这是一个工程商的trade-off 适合的才是最好的

### 2. Skills 扮演了什么角色？

Skills 并不是用来替代底层通信协议的，它是**面向任务流程的“编排层”**。

- Skills 将高频、标准化的业务流程（如生成周报、撰写 PRD、数据分析）封装为结构化的文档（如 YAML+Markdown 格式的 SKILL.md）。
- 它解决了大模型在长上下文中注意力涣散的问题，通过“渐进式披露”让 AI 按需加载执行指令，从而把“个人经验”转化为组织可复用的生产力资产。

### 3. 未来的终极形态：三层混合架构

在成熟的企业级智能体架构中，MCP、CLI 和 Skills 构成了完美的“黄金三件套”：

- **MCP（通信管道）**：作为底层通信标准，解决“标准化连接”问题，确保 AI 跨系统调用时的安全与合规。
- **CLI（原子执行）**：作为底层工具，解决“深度原子执行”问题，将系统核心业务动作剥离为机器可直接下达的底层指令。
- **Skills（业务编排）**：作为上层能力封装，解决“复杂业务编排”问题。Skills 引擎会将底层的 CLI 命令或 MCP 接口进行组合与封装，转化为智能体可以直接调用的宏观“业务技能”。

**总结来说：** MCP 并没有死，它正在成为 AI 世界的“USB 接口”；CLI 也没有完全替代 MCP，而是接管了对成本和灵活性要求极高的原子操作；Skills 则是在这两者之上，让 AI 真正具备“办成复杂业务”的统筹能力。未来的趋势不是相互取代，而是**MCP 保安全底线，CLI 提执行效率，Skills 做高阶编排**的深度融合。


## 怎么看待飞书CLI？

结合我们刚才探讨的“MCP、CLI 与 Skills 的三层混合架构”，飞书 CLI 可以说是这一理论在 2026 年企业级 AI 协作场景中最具代表性的落地实践。

基于 2026 年 7 月的最新技术现实，我对飞书 CLI 的看法可以总结为：**它表面上是一个开源的命令行工具，本质上是飞书向 AI Agent 开放底座的“特洛伊木马”，它标志着企业协作软件正在从“人与人的协同工具”向“人与 Agent 的协同基础设施”发生质变。**

我们可以从以下四个维度来深度拆解：

### 1. 架构定位：它是 AI 的“原生键盘”，而非人类的效率工具

飞书 CLI 覆盖了消息、文档、日历、多维表格等 17 个核心业务域，提供超过 200 条子命令。但正如业内评价所言，**“CLI 不是给人类的工具，是给 AI 的工具”**。

- **对人类而言**：在终端里发消息或查日程，效率提升接近于零，甚至不如直接用手机或客户端。
- **对 AI Agent 而言**：大模型对 CLI 命令的训练语料极度熟悉，这使得 AI 能够以极低的 Token 消耗和极高的准确率直接操作飞书全家桶。它完美契合了我们之前提到的“原子执行层”，让 AI 真正长出了“手脚”。

### 2. 生态策略：Skills 机制与“零配置”体验

飞书 CLI 并没有走纯粹的 MCP 路线，而是巧妙地引入了 **Skills 机制**。它内置了 26 个 Agent Skills（如 `lark-calendar`、`lark-im`），在通过 npm 安装时，会自动写入 Claude Code、Cursor、Codex 等主流 AI 编程工具的 Skill 目录。 这种“装完就能用”的零门槛体验，极大降低了开发者接入国内办公系统的成本。飞书借此在短短几个月内，就在国内 IM 工具中拿下了高达 65.2% 的 Agent 接入率，成功抢占了 AI 时代的生态入口。

### 3. 商业逻辑：“开源客户端，锁死云端价值”

飞书采用 MIT 协议开源 CLI，看似激进，实则是极其高明的商业防守与进攻。

- **防守**：面对 OpenClaw 等开源社区强行接入的威胁，飞书通过官方开源 CLI 抢回了标准定义权。
- **进攻**：它开源的仅仅是“操作入口”，而 API 调用量、数据存储、权限校验等核心资产依然牢牢掌控在飞书服务端。随着 AI 在飞书上的操作越来越频繁，飞书沉淀的企业上下文（Context）价值就会被无限放大。未来，飞书甚至可能通过“生成式 UI”将飞书视觉风格的微组件反向寄生到第三方 AI 前端中，维持品牌存在感。

### 4. 安全与边界：警惕“手脑分离”与“交出操控权”

尽管飞书 CLI 极大地提升了自动化效率，但在工程实践中必须保持清醒的认知：

- **数据出域风险**：当外网大模型（脑）通过开源 CLI 操作公司系统（手脚）时，核心数据可能会在云端跑一圈。对于中大型企业而言，数据不出域依然是红线，这也是钉钉选择走封闭底层重构路线的原因。
- **人类角色的退化**：在使用飞书 CLI 时，人类的角色正在从“操作者”退化为“审批 NPC”。AI 以你的名义发消息、建日程，你只需点击“同意”。长期来看，这会导致人类失去对工作信息的“手感”和体感。因此，**“日常刷怪让 AI 代劳，Boss 战自己亲手上”**是当下最合理的工程实践准则。

**总结来说**，飞书 CLI 是 2026 年连接 AI 与办公系统最短、最有效的一条线。它完美诠释了“CLI 做原子执行”的架构理念。但作为使用者，我们需要明确边界：把它当作强大的自动化外挂，而不是彻底交出工作掌控权的依赖。
