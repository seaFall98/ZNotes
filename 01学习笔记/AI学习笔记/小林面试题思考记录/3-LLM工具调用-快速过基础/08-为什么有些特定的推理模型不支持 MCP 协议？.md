
面试回答这道题，必须讲清楚三个层次：

第一，推理模型的思考链是一次性连续生成的，不能中途打断；

第二，工具调用天然需要在生成过程中暂停等待外部执行，这和连续生成的范式冲突；第三，MCP底层依赖FunctionCalling，推理模型连FunctionCalling都支持不好，MCP自然也用不了，这是一个传导关系。

如果能再补充后续的解决方案就更加分了：
o3和ClaudeExtendedThinking采用的折中方案是让工具调用发生在思考阶段结束之后，保证思考过程仍然完整连续。
Claude还进一步推出了interleavedthinking模式，允许在多次工具调用之间穿插思考，部分缓解了这个局限。

同时也要点出折中方案的本质权衡：各家的解法都是在「保证推理质量」和「支持工具调用」之间找平衡点。

## 补充：防止过时信息

早期部分推理模型确实不支持 Function Calling，或者不支持在 thinking 模式下调用工具，所以当时有人认为“推理过程和工具调用无法兼容”。但现在这个说法已经不能作为普遍结论。现代推理模型已经支持在多轮工具调用之间继续推理，这叫 interleaved thinking。工具调用并不是强行打断一条不可恢复的思维链，而是模型先生成一次工具调用，工具返回结果后，模型基于保留的上下文继续推理。

另外，MCP 也不等于 Function Calling。MCP 是 Host、Client 和 Server 之间的连接协议；Function Calling 只是模型向 Host 请求执行工具的一种方式。即使模型不支持 Function Calling，Host 仍然可以通过代码或工作流调用 MCP Server，只是模型不能自主选择工具。

截至 2026 年，这个问题更准确的表述应该是：**某个模型、API 或推理模式是否支持 Tool Calling，以及当前 Host 是否支持 MCP，而不是简单说“推理模型不支持 MCP”。** 

仍然可能不支持的，主要是旧版 Preview 模型、普通文本补全模型、没有进行 Tool Calling 对齐的本地模型，或者某个 API 端点没有开放工具调用能力。
