
## 简答1

我用过LiteLLM，它是目前最活跃的开源LLM网关。

我理解网关本质上是架在应用和模型API之间的中间层，主要解决几个实际问题。

第一是多模型统一接口，业务代码只调一个地方，想换模型只改网关配置，不用动应用代码；

第二是APIKey集中管理，不用每个服务都存一份，降低泄漏风险。

第三是限流和配额，可以给不同团队分别设token预算，防止某个团队把整个公司的额度用光；

第四是成本追踪，所有请求的token用量都在网关记录，方便统计哪个服务最烧钱。

还有2个我觉得挺实用的LLM网关很实用的能力，

一是语义缓存，两个用户问了语义相近的问题，直接命中缓存返回上次的结果，根本不打底层模型，省钱还降延迟。阿里云有专门的语义缓存网关

二是prompt 安全过滤，这个sub2api里就有，后台可以看到详细的拦截记录，在网关层可以统一做输入输出的安全校验，包括检测prompt注入攻击（用户试图通过特殊指令劫持模型行为）、过滤个人隐私信息（比如身份证号、手机号不应该被发送到外部API）、内容安全审核等。集中在网关做的好处是：不需要每个业务服务各自实现，安全策略统一更新，任何一个接入点都自动获得保护。有一个服务漏了检测这种事情不会再发生。

### 30～90 秒面试回答

我接触过 sub2api / newapi / LiteLLM Proxy 这类大模型网关，主要把不同厂商、不同模型统一封装成一个类似 OpenAI API 的访问入口。业务服务不需要直接维护多套厂商 SDK，也不需要感知模型供应商的差异。

网关层主要解决四类问题：第一是统一接入和模型路由，比如按模型能力、价格、延迟或供应商状态进行选择；第二是可靠性治理，包括超时、限流、重试、熔断和故障降级；第三是安全和多租户管理，包括 API Key 管理、权限、配额和成本控制；第四是可观测性，统一记录请求量、Token、延迟、错误率和模型成本。

不过网关主要负责模型访问治理，不等于 Agent 编排层。RAG、工作流、工具调用、会话状态和业务权限，通常仍然由上层应用或 Agent 服务负责。


我们让业务统一调用网关地址，网关根据模型和渠道配置选择上游；遇到超时或限流时进行有限重试和故障转移，同时记录 Token、延迟、错误率和租户用量。难点主要在流式响应、工具调用格式兼容，以及多实例部署时 Redis、数据库和限流状态的一致性。

### 网关层解决的问题

- **统一协议**：屏蔽不同模型的接口、参数和错误格式差异。
- **模型路由**：支持主备模型、按权重分流、按成本或延迟选择模型。
- **故障处理**：处理超时、限流、重试、熔断和供应商降级。
- **访问控制**：管理密钥、租户、模型权限、调用额度和并发数。
- **成本治理**：统计 Token 和费用，限制单租户或单应用预算。
- **统一观测**：采集请求日志、Trace ID、延迟、错误率、Token 使用量。
- **流式支持**：透传 SSE 流式响应，并处理客户端断开、超时和连接释放。
- **供应商解耦**：后端从一家模型供应商切换到另一家时，减少业务代码改动。

### 需要注意的边界

网关不能解决模型本身的幻觉、Prompt 质量或业务流程设计问题，也不应把所有 Agent 编排逻辑都堆进去。

另外，重试并不总是安全。流式响应已经返回一部分，或者请求触发了外部工具和副作用时，盲目重试可能造成重复执行。因此需要结合幂等键、请求阶段和错误类型判断是否重试。

### 追问方向

- 多模型路由应该依据哪些指标？
- 网关如何统计 Token 和成本？
- 流式请求断开后如何取消上游请求？
- 大模型网关和 API Gateway、Agent 编排层有什么区别？

---
例1 "今天天气如何"和"今天天气怎么样" 缓存如何处理？

属于同一问题 缓存结果共享。

需要做语义识别 内容分类 选择不同的缓存策略

User Request

↓

Intent Classification

↓

Cache Policy

但是对于这种个性化或者强实时性的问答，甚至可以直接绕过缓存走 LLM。

---

例2：

```
公司的报销流程是什么？
```

判断：

```
Static Knowledge
```

策略：

```
TTL = 7天
```

甚至：

```
永久缓存
+
版本控制
```

---

企业不会只缓存：

```
question → answer
```

而是：

```
cache key =
{
 query embedding,

 model,

 system prompt version,

 user permission,

 tenant,

 knowledge version  #版本管理
}
```

AI Gateway 的语义缓存不是“相似问题返回相同答案”，而是“在语义相似、上下文一致、权限一致、数据仍有效的情况下复用模型结果”。实际非常复杂。。。
