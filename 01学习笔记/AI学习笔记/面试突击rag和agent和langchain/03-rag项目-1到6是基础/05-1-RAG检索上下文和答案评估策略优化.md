## 简答

**上线前，项目整体评估**  
**答案质量：**

- **忠实度：** 将答案分解为多个陈述，计算能被上下文支持的陈述比例
- **答案相关度：** 根据答案反向生成N个问题，计算它们与原始问题的平均语义相似度。

**检索质量：**

- **上下文精度：** 检索到的信息中，与问题真正相关的部分所占的比例及排序质量
- **上下文召回率：** 检索到的上下文是否包含了回答问题所需的全部关键信息，对比检索到的上下文与标准答案，计算覆盖了多少标准答案中的关键信息点

RAGAS提供了TestsetGenerator工具，可以从你的知识库文档中自动生成问题和对答案。1000 ~ 10000行的评估样本测试集数据。

python

编辑

```
# 示例：手动构建数据集字典
data = {
    "user_input": ["客户经理被投诉一次扣多少分？", "每年的评聘申报时间？"],
    "response": ["每投诉一次扣2分", "每年一月份为客户经理评聘的申报时间"],
    "retrieved_contexts": [["...扣分细则规定，每收到一次有效投诉扣2分..."], ["...申报工作于每年一月份启动..."]],
    "reference": ["每投诉一次扣2分", "每年一月份为客户经理评聘的申报时间"]
}
```

**上线后：实时评估**

第一层：从知识库中检索出来的doc进行Context Precision指标+Context Recall指标评估。第二层：在输出答案之前，基于Response Relevancy指标+Faithfulness指标评估。每层都设置阈值，极大提高响应的准确率。  
最终还需要人工介入来审核多次评估但是都低于配置阈值的结果，人工介入采用LangGraph的Human-in-the-loop。人工介入时 interrupt节点触发中断，并在人工介入之后从中断处继续恢复工作流

实时评估只选少数指标，因为指标过多会影响响应速度，影响用户体验，这是一个trade-off，离线ragas做大而全的评估，线上要同时考虑准确率和响应速度。

这个问题提得非常到位。作为面试官，当抛出“RAG检索上下文和答案评估策略优化”这种“大而全”的问题时，本质上是在考察候选人的**系统架构思维**和**端到端调优经验**。

如果候选人上来就背八股文（比如直接说“用Reranker”或“调chunk_size”），通常会被判定为缺乏实战深度。**面试官真正想听的，是你能否把这个问题拆解成有逻辑层次的子模块，并针对每个模块讲清楚“痛点”和“权衡（Trade-off）”。**

基于企业级RAG落地的实际踩坑经验，我作为面试官，会把这个问题拆解为**五个核心维度**。你可以对照你的思考，看看是否覆盖全面：

### 第一维度：数据“供给侧”优化（索引前的质量治理）
**这是企业级最容易被忽视、但回报率最高的一环。** 面试官想听的不是“调分块大小”，而是：

-   **分块（Chunking）策略的语义完整性**：你是如何解决“一句话被切碎导致语义反转”的？是否引入了**基于语义的递归切分**或**Chunk重叠（Overlap）**的滑动窗口？
-   **元数据（Metadata）增强**：不仅仅是切文本，是否将**标题、段落结构、时间戳、权限标签**作为元数据注入向量？这在企业权限隔离和带时间维度的检索（比如“查去年的财报”）中至关重要。
-   **数据清洗颗粒度**：PDF中的页眉页脚、表格OCR错乱、特殊符号，这些脏数据在离线阶段是如何治理的？（如果这块不提，面试官会认为你没经历过真实脏数据毒打）。

### 第二维度：“召回”策略优化（多路召回与粗排）
**这是你提到的“混合检索+RRF”所在层。** 面试官会关注你处理**语义漂移**和**关键词依赖**的能力：

-   **多路召回的组合策略**：向量召回（语义）和BM25（关键词）如何加权？是否引入了**HyDE（假设性文档）**或**Query Rewrite（查询改写）**来弥补用户输入短、意图不明的缺陷？
-   **RRF（倒数排名融合）的陷阱**：RRF虽然简单，但会**抹平高分文档的领先优势**。在面试中，如果能指出“当一路召回极度精准时，RRF反而不如加权求和”，会是一个亮眼的加分项。
-   **粗排的截断（Top-K）**：你是取100还是取50？这一步决定了后续精排的延迟天花板。


### 第三维度：“精排”与上下文重组（生成前的最后防线）
**这是你提到的“Reranker精排”所在层，但远不止于此。** 面试官想看的是**“检索到了，但怎么喂给大模型”**：

-   **精排模型的选型权衡**：使用Cross-Encoder（如BGE-reranker）精度高但慢，如何通过**级联架构**（粗排用双塔，精排用交叉）来平衡延迟？是否做了**延迟敏感型的降级策略**？
-   **“Lost in the Middle”问题处理**：大模型对长上下文的中间部分“失忆”。检索回来的Top-10，你是按相似度排序直接塞进去，还是**重排序（Reorder）**把最相关的放开头和结尾？
-   **上下文压缩与去重**：检索回来的多路结果往往**信息冗余**。是否引入了**LLMLingua**等上下文压缩技术，或者通过**最大边际相关性（MMR）**去重，来为生成腾出宝贵的窗口（Context Window）空间？

### 第四维度：离线评估体系（RAGAS 与 自动化门禁）
**这是你提到的“RAGAS离线评估”。** 面试官不在乎你会不会调用RAGAS库，而在乎**评估结果如何指导线上优化**：

-   **指标拆解**：RAGAS的三个核心指标（**Context Relevancy**、**Faithfulness**、**Answer Relevancy**）分别卡的是哪个环节？例如，Context Relevancy低，问题出在检索召回；Faithfulness低，问题出在生成幻觉。
-   **评估数据集（Golden Dataset）的构建**：企业里谁标注这1000条标准问答对？如何**避免评估集过拟合**？是否引入了**LLM-as-a-Judge**的对抗性评估？
-   **回归门禁**：每次调整分块或Reranker后，跑一次全量评估耗时多久？你们如何将这个评估流程**CI/CD化**（即作为上线的前置条件）？

### 第五维度：线上实时动态评分（在线推理的“护栏”）
**这是你提到的“实时打分优化”，也是面试官判断你是否具备生产经验的关键：**

-   **检索置信度打分**：当向量检索的最大相似度低于0.7时，你是强制回复“我不知道”，还是强行拼接导致幻觉？**拒答阈值（Rejection Threshold）**如何动态调整？
-   **生成过程中的事实核查**：是否引入了**溯源（Citation）**机制？即大模型生成的每一句，是否都能追溯到具体检索片段的引用？这比最终的End-to-End打分更重要。
-   **用户隐式反馈闭环**：线上的“点赞/点踩”或“复制/重新生成”行为，如何转化为**强化学习信号**，反哺到Reranker或Embedding模型的微调中？

---

**面试官视角的“终局追问”：**
当候选人把这五层拆解完，面试官通常会追加一个**极端场景题**来定级（P6/P7分水岭）：
*“如果老板要求**首Token延迟必须控制在300ms以内**，但我们的精排Reranker需要500ms，现在你必须在‘检索质量’和‘延迟’之间做取舍，你会砍掉上述哪一层的优化？并用数据说话。”*

**建议你的回答策略：**
不要试图面面俱到，而是**先抛出这5层框架**，然后挑**1-2个你实际踩过坑的环节**（比如：“我们在RRF转加权求和时解决了某个业务问题”）进行深挖。这样既展示了全局视野，又体现了实战深度，是面试官最想听到的“降维打击”式回答。你现在可以基于这个拆解，重新组织你的答案了。
