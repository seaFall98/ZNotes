
简答

1、上下文分层策略：  
整个项目的上下文分为短期和长期两层。短期上下文中包含：当前会话的历史对话记录、工具结果、当前工作流阶段的关键中间状态。基于PyDantic来定制和结构化State设计。通过RedisSaver实现数据的自动持久化操作。针对短期上下文，我们在langgraph工作流中的第1个节点自定义为summarize_node。在对话轮次或Token数量达到阈值时，异步生成摘要替换原始长历史，保留叙事连贯性。同时也注入会话ID、用户身份，用户偏好、少量核心业务规则等数据，到State中，增强LLM对上下文的理解。  
长期上下文中包含：所有会话历史对话记录、用户偏好，详细业务规则。我们是在最后End节点之后，在项目中，通过自定义的异步多线程操作，把当每一轮对话记录保存到Milvus数据库中。注意：对话记录，我们只保留用户的HumanMessage和最后的AIMessage组成一条记录。这样的话，在需要时还可以检索注入，避免无限延长上下文。而且，我们的项目还对检索结果进行重排和RAGAS评估。评估相关性指标到达阈值后，才会注入到短期上下文的State中。

2、监控与安全策略  
在LangGraph检查点（Checkpoint）中记录短期上下文长度、Token消耗数量等；并且，在短期上下文中，最后一个节点生成的AIMessage，在写入State之前，通过自定义RunnableLambda调用脱敏服务。保证最后传给客户的消息完全脱敏。

最新的langchain deepagent有摘要中间件SummarizationMidlleware
LangChain 的 SummarizationMiddleware 主要解决 Agent 短期上下文窗口限制，当 token 达到阈值时，通过 LLM 对旧消息进行摘要，用 summary 替代历史消息。

DeepAgent 的 SummarizationMiddleware 是基于 LangChain 实现的增强版本，针对长周期任务 Agent 做了 Context Engineering。除了摘要，还会把被压缩的完整历史保存到 backend，避免信息永久丢失，同时支持大 tool 参数提前截断，以及主动 compact。它更适合类似 Claude Code 的长时间 coding agent 场景。

短期记忆 长期记忆 的混淆
从作用域上 短期记忆是单会话 长期记忆是跨会话 最终都是要持久化磁盘里的
从持久化上 短期记忆是内存里的摘要+完整最近N轮message 长期记忆是持久化到数据库磁盘
所以这个说法其实有混淆的地方 得分开说

其实翻译也有问题

working memory 本次会话最近N轮最相关的上下文  工作记忆 但是也被叫短期记忆
 
short-term memory 本轮会话内有效 所谓的短期记忆

long-term memory 跨会话有效的记忆  或者 持久化到数据库磁盘里的记忆 有2种含义

ragas是离线评估
在生产环境中，只对 部分流量 进行评估（通常5%-10%）：
- 异步执行 ：不阻塞用户请求 (消息队列异步处理 后台worker处理评估任务)
- 采样率控制 ：平衡评估覆盖率和成本
- 数据回流 ：生产中的Bad Case自动加入离线评估数据集
- 对所有请求进行 轻量级、非LLM的监控-检索延迟/检索Top-K命中率/Embedding相似度分数/答案长度/Token消耗/用户反馈点赞点踩评分
