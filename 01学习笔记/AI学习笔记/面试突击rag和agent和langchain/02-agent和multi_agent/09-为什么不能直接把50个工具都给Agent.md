**上下文窗口爆炸：** Token 消耗激增，上下文窗口溢出：每个工具的描述（名称、功能、参数）通常需要数百个 Token，50 个工具的描述会占用上万 Token，直接耗尽模型的上下文窗口，导致关键信息无法被模型捕捉。并且也挤占本应用于理解用户请求和生成答案的上下文空间。

**性能下降：** 过多的工具会增加模型决策的复杂度，当工具描述存在重叠或模糊时，模型容易选错工具或产生幻觉（如调用不存在的工具）。我看过网络上的播客和论文，当工具数量超过 30 个时，选择准确率会明显下降。

**响应延迟增加，成本飙升：** 处理大量工具描述需要更多计算资源，导致模型响应时间延长，更多的 Tokens 意味着更高的 API 调用成本。

**指令遵循困难：** 模型可能被大量不相关的工具描述所干扰，无法专注于当前任务。

**解决办法：**

1. **Multi-agent 方案：** 根据用户意图或者业务需求将工具范围从上百个缩小到十几个相关的类别。然后通过工作流的 Graph 开发，把用户的意图拆分成多个节点，按照一定的逻辑链去执行。每个节点分配一部分工具。再或者，通过 Multi-Agent 架构设计。把单个任务拆解成多个子任务，对应不同的子 Agent，每个子 Agent 管理一部分工具列表。
2. **Agent Skills 方案：** Skills 的核心思想是“按需加载工具”，即仅在用户意图匹配时加载相关工具，而非一次性暴露所有工具。这种机制避免了工具过载，显著提升了模型的推理效率。
3. **langgraph-bigtool 库，LangGraph BigTool 通过语义相似性搜索 (RAG) 从工具库中检索最相关的工具（如 Top-5~10 个），而非全部加载。**

补充

在基于LangChain、LangGraph及DeepAgent构建Agent系统的实践中，**不能直接将50个工具全部给一个Agent**，这并非简单的建议，而是由模型能力上限和工程架构成本共同决定的硬约束。

### 一、为什么不能直接给50个工具？

核心问题是**上下文膨胀（Context Bloat）**。当Agent拥有30个以上工具时，每次模型调用都需将所有工具的定义（名称、描述、参数Schema）传入上下文。这会带来三个严重后果：

1.  **模型准确率下降**：模型从海量工具中做选择的决策噪声显著增加，表现远不如从一个更小、更相关的工具集中做选择。
2.  **响应延迟飙升**：上下文越大，模型推理越慢。有团队实测，工具过多会导致延迟显著增加。
3.  **Token成本浪费**：工具定义本身消耗大量输入Token，直接推高成本。

工程实践共识是：给单个Agent **5-8个精选工具**效果最好。工具一旦超过20个，Agent就会陷入“选择困难”，准确率下降。

### 二、如何解决？—— LangChain 2026年的三大方案

LangChain生态在2026年提供了三个层次递进的解决方案，均集成在**Deep Agents**框架中。

#### 方案一：渐进式工具披露（Progressive Tool Disclosure）

这是最直接的优化方案，核心思想是“注册全量，推理时按需过滤”**。

*   **运作机制**：Agent初始化时注册全部50个工具（保证执行器能调用任意一个），然后通过中间件（Middleware）控制模型在每次推理时“看到”哪些工具。模型始终只看到一个核心工具集（如5-8个）加上一个  **search_tools工具** 。当需要其他能力时，模型调用`search_tools`，中间件扫描历史，动态将匹配的工具加入后续模型调用。
*   **技术实现**：LangChain通过`@wrap_model_call`装饰器和`request.override()`API实现。`create_deep_agent`接受`middleware=`参数，可注入自定义中间件。

#### 方案二：Multi-Agent（Subagents）—— 上下文隔离

核心思想是**“分而治之”**，将一个“什么都会”的巨型Agent拆解为多个各司其职的“专家”Agent。

*   **运作机制**：主Agent（Supervisor）将每个子Agent（Subagent）包装成一个“工具”。子Agent拥有独立的上下文窗口、工具集，甚至可以使用不同的模型。子Agent独立完成多步骤任务后，**只将最终结果**返回给主Agent。
*   **核心价值**：通过**上下文隔离**从根本上解决问题。子Agent内部的所有工具调用和中间推理都不会污染主Agent的上下文。
*   **2026年技术事实**：
    *   **官方推荐模式**：LangGraph已将Subagents模式标准化。
    *   **Deep Agents内置支持**：在`create_deep_agent`中传入`subagents`列表即可。Deep Agents还自动内置了一个通用型subagent。
    *   **异步子Agent（v0.5）**：2026年4月发布的Deep Agents v0.5引入了**异步子Agent**。主Agent可以“火并发射”（fire-and-forget）多个子Agent并行工作，并支持异构部署（不同模型、不同硬件）。

#### 方案三：Agent Skills —— 能力的渐进式披露

如果说Multi-Agent是“重模式”（创建新Agent），那么Agent Skills就是“轻模式”——**不创建新Agent，而是让单个Agent按需加载**专门化的提示词（Prompt）和知识。

预设Skill后，Agent的决策退化为**简单的意图匹配**（选1个Skill）。Skill内部预设了硬编码的最佳路径（如“步骤A用Tool_X，步骤B若失败则重试Tool_Y”），将**运行时的试错成本**转移到了**设计时的编排成本**，大幅削减了不必要的ReAct循环步数，也规避了“选错工具”的风险。

Skill本质上是一种**“软编码的SOP（标准作业程序）”**——它把“Agent临场发挥”变成了“执行既定预案”，既省了Token，又省了思考步数，是解决大规模工具调用最彻底的工程化降维手段。


*   **运作机制**：Agent Skills遵循**渐进式披露**原则。Agent启动时只知道技能的名称和描述，只有当它决定需要某个技能时，才会通过`load_skill`等机制读取完整的指令。
*   **核心价值**：极轻量的专业化手段，无需维护多个Agent实例，非常适合为单个Agent添加大量可选的专业知识。
*   **2026年技术事实**：
    *   **官方支持**：LangChain官方发布了LangChain Skills，并提供`SkillsMiddleware`中间件。
    *   **显著效果**：在LangChain官方评测中，为Claude Code添加Skills后，其在相关任务上的通过率从**25%提升至95%**。
    *   **规范与实现**：Skills遵循Anthropic提出的`agentskills.io`规范。

### 三、方案对比与选择

| 维度 | **渐进式工具披露** | **Multi-Agent (Subagents)** | **Agent Skills** |
| :--- | :--- | :--- | :--- |
| **核心思想** | 过滤模型看到的工具 | 上下文隔离，专业分工 | 能力按需加载 |
| **上下文隔离** | 弱 | **强**（子Agent独立上下文） | 弱 |
| **并行处理** | 不支持 | **强**（支持异步并行） | 不支持 |
| **复杂度** | 低 | **高**（需管理多Agent实例、状态） | 低 |
| **适用场景** | 工具数量较多但相关性可动态判断 | 任务可明确拆分为独立、复杂的子领域 | 为Agent添加大量可选的专业知识/指令 |

### 四、总结

在实际生产中，这三个方案并非互斥，而是可以**组合使用**。一个典型的**Deep Agent**可以同时配置**渐进式工具披露**（控制模型视野）、**Subagents**（处理复杂独立子任务）和**Skills**（处理轻量级专业知识）。

> **选择建议**：
> *   如果只是**工具数量过多**，优先用**渐进式工具披露**。
> *   如果工具属于**独立、复杂的领域**（如“财务分析”、“代码审查”），用 **Multi-Agent**。
> *   如果工具是**相关、轻量的专长**（如“写SQL”、“生成报告”），用 **Agent Skills**。

这种分层、按需的设计哲学，正是LangChain Deep Agents框架在2026年解决大规模工具调用的核心思路。
