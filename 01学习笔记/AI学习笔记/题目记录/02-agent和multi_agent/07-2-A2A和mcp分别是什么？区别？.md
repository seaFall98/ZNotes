

1. A2A 是智能体间通信的开放标准协议，它基于 JSON-RPC 2.0 通信机制：使用 HTTP(S) 传输，支持同步请求/响应和 SSE 流式传输。是为了解决不同框架、不同厂商构建的智能体之间的互操作性问题，让它们能像“说同一种语言”一样协作。其核心目标是实现智能体的无缝协调、任务委派与信息交换。
2. 客户端 agent 负责制定和传达任务，而远程 Agent 负责执行这些任务，以提供正确的信息或者采取正确的行动。交涉需要这几种能力
3. **发现能力**：每个 agent 用 json 写一个 agent card，宣传自己的能力，从而允许客户端 agent 识别能够执行任务的最佳 agent，客户端通过“URI”或“策展注册表”找到智能体的 Agent Card，实现能力的自动发现。
4. **任务管理**：简单的任务会通过很快的执行完毕，但是复杂的任务会执行很久，agent 之间会不断地沟通同步执行的进度，任务完成后会生成一个"artifact"(读阿踢法克特)代表任务结束。
5. **协作**：不同的 agent 之间需要相互发消息，传递上下文。
6. A2A 主要用于多智能体协作，解决企业级 AI 项目中的“集成痛点”，尤其是：不同框架的智能体协同和跨厂商智能体的信息共享。

7. MCP (Model Context Protocol) 是模型上下文协议，目的是标准化大语言模型 (LLM) 与外部工具、数据源的交互，解决“LLM 无法访问实时数据或工具”的问题。让 Agent 能无缝调用外部资源（如数据库、API、文件），扩展 Agent 的功能边界。MCP 的设计围绕“工具集成”展开，也是采用客户端-服务器架构：
8. MCP 服务器 (Server) ：功能提供层，标准化封装资源 (Resources)、工具 (Tools)、提示 (Prompts)。
9. MCP 客户端 (Client) ：一般是 Agent 程序或者其他运行 LLM 的应用程序 (dify, Cursor 等)，负责维护与服务器的持久连接，负责请求与响应的处理。
10. MCP 的传输机制：支持本地调用 (stdio)、SSE 和远程调用 (Streamable HTTP)。

---

A2A 与 MCP 是互补而非竞争的协议，二者的差异主要体现在目标定位、应用场景、技术架构三个方面：

|维度|A2A (Agent-to-Agent)|MCP (Model Context Protocol)|
|:--|:--|:--|
|**核心目标**|实现智能体间的通信与协作|实现 LLM 与外部工具/数据源的交互|
|**应用场景**|多智能体系统（如跨框架智能体协同、任务委派）|LLM 工具集成（如实时数据查询、工具调用）|
|**技术架构**|基于 JSON-RPC 2.0，支持异步任务与多模态传输|客户端-服务器架构，支持资源/工具/提示的标准化|
|**关键组件**|Agent Card、任务 (Task)、消息 (Message)|MCP 主机、客户端、服务器 (资源/工具/提示)|
|**互补性**|MCP 为智能体提供工具调用能力，A2A 实现智能体间的协同|A2A 是“通信层”，MCP 是“工具层”，共同构建复杂 AI 系统|
