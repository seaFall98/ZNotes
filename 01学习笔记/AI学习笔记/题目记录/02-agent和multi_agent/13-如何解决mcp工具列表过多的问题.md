[[09-为什么不能直接把50个工具都给Agent]]


面对海量工具，首要任务是避免将所有工具描述一次性塞入模型的上下文窗口。否则的话就会出现大模型的上下文"爆炸"，结合我简历中所说的那个项目，我认为最直接有效的办法就是"减法"策略。整体的思路就是将工具按照业务场景来分类。根据上下文和用户的指令，只激活需要使用的相关工具组。我觉得有两种不同的解决方案。我项目中采用的是第 1 种。

第一种（强调这是老方案）：采用基于 Supervisor 的 Multi-Agent 架构(supervisor已经不维护了，看deepagent的多agent体系)。将庞大的工具集按功能或领域分配给不同的专家 Agent（技能），并通过一个主管 Agent（Supervisor）来动态调度，实现工具的"按需分配"和"专业分工"。主管 Agent 分析用户请求的意图，将复杂的任务分解成多个子任务。根据子任务的性质，将其分配给最专业的"专家 Agent"去处理。每一个专家 Agent，只负责管理一小部分的工具。

第二种：采用渐进式披露的 Agent skills 架构。首先根据不同的业务场景来划分多个 Skills 技能包。比如专门负责数据分析的 skill，专门负责代码执行的 skill 等等。每个不同的 skill 管理各自一小部分工具列表。采用新版的 langChain1.0 中的中间件 Middleware 和工具动态加载机制。由一个智能体负责，基于上下文和用户指令动态决策的加载某一个专门的 Skill 技能包。再去动态加载这个 Skill 包中所管理的提示词和工具列表。这些工具就来自于 MCP 服务器中的一部分工具。

这种分工的直接好处就是解决了在海量 MCP 工具造成的上下文膨胀：每个专家 Agent 或者 Skill 技能包在工作时，其系统提示词中只包含与它专业相关的少数几个工具的精确的定义。这极大减少了上下文长度，降低了模型的认知负荷，使工具选择更加精准和快速。
