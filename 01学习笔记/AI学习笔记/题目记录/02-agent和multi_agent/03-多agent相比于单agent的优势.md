




## 一、多Agent vs 单Agent：优势与劣势

### 多Agent优势

**1. 专业分工，各司其职**
每个Agent承担特定角色（数据采集、分析、验证等），通过协作完成复杂任务，避免单个Agent被迫同时扮演多种角色导致的“角色迷失”。

**2. 上下文隔离，防止污染**
子Agent在各自独立、干净的上下文中运行，避免一个子任务积累的大量无关信息污染后续子任务的推理能力。

**3. 并行处理，提升吞吐**
多个Agent可并行执行子任务，相比单Agent顺序处理显著提升效率。

**4. 故障隔离，局部容错**
单个Agent故障不影响整体服务——某个摄像头失效不影响整体决策，某个感知Agent出问题不导致系统瘫痪。

**5. 独立演进，按需升级**
可单独优化某个Agent（如升级知识检索Agent）而不影响其他模块，团队可独立开发维护各自负责的Agent。

**6. 横向扩展，突破上下文窗口限制**
通过增加Agent类型或数量扩展功能，不受单模型上下文窗口限制。

### 多Agent劣势

**1. Token消耗剧增（3-10倍）**
多Agent实现相比单Agent方案，同等任务通常消耗3-10倍的Token——来自跨Agent重复上下文、协调消息和交接结果汇总。

**2. 协调复杂，引入额外延迟**
需维护智能体间通信协议、数据一致性机制，跨Agent协作可能引入消息队列排队等延迟。

**3. 设计复杂，初期投入高**
初始架构设计需精心规划职责划分与协作规则，远复杂于单Agent。

**4. 运维成本高**
需维护多个Agent的代码库、依赖项与版本兼容性，资源成本随Agent数量增加而上升。

**5. 可能边际收益递减甚至为负**
当单Agent准确率超过45%时，引入多Agent协作常导致收益递减甚至负收益——多Agent本质是将问题从“如何让AI更聪明”变成了“如何管理一个聪明的团队”。

### 单Agent优势

**1. 简单直接，易于调试**
逻辑简单，结构清晰，适合处理线性、确定性任务。

**2. 资源消耗低**
运行单个Agent所需计算资源和编排开销远少于管理多个Agent。

**3. 运维简单**
模型训练与调优虽需专业团队，但日常运维相对简单。

### 单Agent劣势

**1. 上下文爆炸**
长周期任务中，Token消耗与上下文窗口压力呈指数增长。

**2. 工具选择困难**
为单个Agent配备超过15个工具后，会陷入“选择恐惧症”，任务完成率可能下降37%以上。

**3. 单点故障**
单一节点故障导致系统完全瘫痪。

**4. 功能扩展成本高**
功能扩展依赖模型升级（如增加参数），需重新训练整个模型。


## 二、企业级场景与选型

### 何时选单Agent

- **任务聚焦且线性**：任务流程清晰、顺序执行
- **中小型项目、快速上线**：资源有限、需要快速迭代
- **短周期任务**：单次对话可完成的工作流
- **验证阶段**：不确定时先做单Agent原型——单Agent可重构为多Agent，反向则更难

### 何时选多Agent

- **上下文污染明显**：单个Agent的上下文因累积无关信息而性能下降
- **任务可并行**：子任务天然可并行执行
- **需要专业化分工**：不同子任务需要不同的工具集和提示词聚焦
- **企业级复杂业务**：涉及多部门协作、跨系统流转和长周期自治运行
- **高并发、大规模**：需要支持大规模并发处理和分布式冗余

### 企业选型核心考量

**1. 安全与治理能力**：金融、政务、能源等强监管行业的首要准入条件

**2. 可编排、可协同、可治理**：企业AI的分水岭已从“有没有Agent”变为“是否具备可编排、可协同、可治理的MAS能力”

**3. 任务架构决定选择，而非预算或野心**：选择单Agent还是多Agent，取决于任务架构本身

**4. 混合架构是趋势**：单Agent与多Agent混合使用——需要可靠性和治理能力的同时又需要扩展能力


## 三、基于LangChain + LangGraph + DeepAgent的Python开发环境

### 框架分工（三层架构）

**LangChain** — 基础组件层：提供工具集成、提示词模板、核心构建块

**LangGraph** — 编排引擎层：状态驱动的图结构工作流编排，负责流程执行与状态管理

**DeepAgent** — 上层Harness层：聚焦多智能体协作，内置子Agent派生、任务规划、虚拟文件系统和上下文管理

### 核心协作模式

LangChain官方文档定义了五种多Agent模式：

| 模式 | 一句话速记 |
|------|-----------|
| **Subagents（子Agent）** | 主Agent将子Agent当作工具调用，集中路由 |
| **Handoffs（交接）** | 基于状态动态切换Agent行为 |
| **Skills（技能）** | 单Agent主导，按需加载专业技能和知识 |
| **Router（路由）** | 路由节点分类输入，分发到专业Agent |
| **Custom Workflow（自定义工作流）** | LangGraph构建定制化执行流，混合确定性逻辑和Agent行为 |

### DeepAgent核心能力

DeepAgent作为LangChain团队推出的专用框架，提供六大内置能力：

- **任务规划**：内置任务分解与规划能力
- **子Agent派生**：主Agent可动态派生专业子Agent，各自运行在独立上下文窗口中
- **虚拟文件系统**：用于上下文管理，文件在对话轮次间持久化
- **长时记忆**：跨会话的记忆管理
- **异步子Agent（v0.5+）** ：启动非阻塞后台任务，用户可继续交互
- **人工介入（HITL）** ：关键决策点支持暂停等待人工审批

### 环境搭建要点

- **Python版本**：3.11或更高版本
- **核心依赖**：`langchain`、`langgraph`、`deepagents`
- **可观测性**：集成LangSmith进行追踪、调试和生产部署
- **生产部署**：Managed Deep Agents可将本地项目转为托管LangGraph部署
- **状态管理**：使用TypedDict或Pydantic定义State，所有节点输入输出需符合State结构

### 开发最佳实践速记

1. **从单Agent开始**：单Agent可重构为多Agent，反向更难——不确定时先做单Agent原型
2. **上下文工程是核心**：多Agent设计的质量取决于确保每个Agent能访问到正确的数据
3. **图结构优于链结构**：LangGraph的有向图支持动态拓扑、分支、循环和并行，远超LangChain链式架构的静态线性流程
4. **用LangSmith全程追踪**：从开发到生产，用LangSmith追踪每个组件的行为
5. **生产就绪模板**：参考社区生产级模板（如`langgraph-agent-stack`），包含成本上限、版本管理、Helm部署和800+测试
