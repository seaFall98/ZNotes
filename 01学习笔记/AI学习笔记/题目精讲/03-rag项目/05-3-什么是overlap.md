
## 什么是Overlap（重叠）

**Overlap（重叠）** 是指在将长文档切分成多个文本块（Chunk）时，**相邻两个块之间共享的部分文本**。

### 具体解释

假设你有一篇完整的文档，需要切分成多段：

```
┌─────────────────────────────────────────────────────┐
│         完整文档（如一篇5000字的文章）                │
└─────────────────────────────────────────────────────┘
```

**不加overlap（overlap=0）**：
```
┌────────────┐ ┌────────────┐ ┌────────────┐
│  Chunk 1   │ │  Chunk 2   │ │  Chunk 3   │
│ (前1000字) │ │ (1001-2000)│ │(2001-3000) │
└────────────┘ └────────────┘ └────────────┘
                 ↑           ↑
           Chunk1的结尾     Chunk2的开头
           ……无衔接，硬切分……
```

**加入overlap后**：
```
┌────────────────────┐
│     Chunk 1         │ ← 前1000字
│  (前1000字)          │
└────────────────────┘
    ┌────────────────────┐
    │      Chunk 2        │ ← 第901字到第1900字
    │  (901-1900字)       │
    └────────────────────┘
        ┌────────────────────┐
        │      Chunk 3        │ ← 第1801字到第2800字
        │  (1801-2800字)      │
        └────────────────────┘

← 灰色重叠区域 → 
   Chunk1的末尾100字 = Chunk2的开头100字
   （overlap = 100字）
```

## 为什么需要overlap？

### 问题场景

文档中一个完整的语义单元（比如一个段落、一个观点或一个实体介绍）恰好被切分在 **Chunk1的结尾** 和 **Chunk2的开头**：

```
……（前文）北京是中国的首都，面积16410平方千米，……
         ↑ 切分点在这里（chunk_size刚好到这里的边界）
……常住人口2189万人，是政治、文化、国际交往中心。……
```

- **不设overlap**：检索时可能只召回包含“北京是中国的首都”的Chunk1，但Chunk2里关于人口和功能的信息被遗漏了，导致Context Recall（召回率）低下
- **设置overlap**：Chunk2会重复包含Chunk1末尾的“北京是中国的首都”，检索时无论命中Chunk1还是Chunk2，都能拿到完整的上下文

## Chunk Size 与 Overlap 的权衡

| 参数 | 含义 | 设置过大 | 设置过小 |
|------|------|---------|---------|
| **Chunk Size**（块大小） | 每个文本块包含的字符/Token数量 | 容易吞入无关噪声（Context Precision下降）；超出模型上下文窗口限制 | 语义被切碎，关键信息散落在多个块中，检索覆盖不全 |
| **Overlap**（重叠长度） | 相邻块之间重叠的字符/Token数量 | 大量冗余信息，增加向量存储和检索开销 | 切分边界处的语义断档无法被覆盖，Context Recall下降 |

---

## 推荐实践

| 场景 | 推荐Chunk Size | 推荐Overlap | 理由 |
|------|:---:|:---:|------|
| 通用中文文档 | 512 tokens | 50-100 tokens | 平衡语义完整性与检索精度，overlap占整体的10%-20% |
| 短句/技术术语密集 | 256 tokens | 30-50 tokens | 块小利于精准匹配，重叠不宜过多避免冗余 |
| 长段落/叙事性文本 | 1000 tokens | 150-200 tokens | 块大保留完整语义，适当加大overlap确保段落连贯 |

**经验法则**：通常将 Overlap 设置为 **Chunk Size 的 10%~20%**，然后根据实际的Context Recall评估分数（参考上一轮对话中的评估方法）做反向调整——召回率过低就适当加大overlap或增加`top_k`。
