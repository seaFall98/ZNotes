基于[[从“让 AI 写代码”到“管理 AI 交付”：一次博客项目的工程复盘]]和后续思考结果，写成了以下这篇技术分享文章

# AI Coding 没有想象中轻松：一次真实项目踩出来的工程化方法

过去一段时间，我用 AI Coding 推进了一个中小型内容型 Web 项目。

项目本身并不算特别复杂：有成熟开源项目可以参考，有相对明确的业务形态，也有 AI Agent 可以参与读代码、写代码、跑测试、改文档。按理说，这应该是一个很适合 AI 编程的场景。

一开始，我确实也是这么想的。

那时我的判断很简单：  
既然 AI 已经能写代码、能理解项目、能生成测试，那我只要把目标说清楚，让它多写点代码，项目进度应该会很快。

但真正推进到中段后，我才意识到：  
**AI Coding 没有想象中轻松。**

更准确地说，问题不在于 AI 会不会写代码，而在于：

> **如何让 AI 在真实工程边界内，持续产出可信的软件。**

这件事，比写代码本身难得多。

---

## 1. 开始时，我以为问题是“让 AI 多写点代码”

这个项目最初的目标很朴素：基于一个成熟的内容型网站形态，做一个可上线、可维护、可长期演进的个人内容平台。

它不是玩具 Demo，也不是纯练手项目，而是希望最终能真的部署、真的使用、真的承载内容。

为了提高效率，我引入了 AI Coding Agent。

当时的想法是：

```text
AI 读代码
AI 写接口
AI 改页面
AI 补测试
AI 改文档
人类做最终确认
```

听起来很合理。

而且从表面看，AI 的效率确实很高。  
它可以很快生成 Controller、Service、DTO、页面组件、测试文件、文档说明。  
它也很擅长把一个需求拆成阶段计划，然后告诉你：

```text
Phase 1 completed
Phase 2 completed
Phase 3 completed
```

一开始，这种“进度感”非常强。

但后来我发现，**进度感不等于真实进度。**

---

## 2. 第一次失控：Phase 完成，不等于功能完成

真正的问题，是从一次审计开始暴露的。

某个阶段，AI 告诉我一批功能已经完成。  
文档里也打了勾，接口也存在，页面也能打开，前端类型检查也没有明显报错。

但换另一个 Agent 重新审计后，发现很多所谓“完成”的功能，实际上只是：

```text
接口路径存在
返回 HTTP 200
前端页面不报错
文档里写了已完成
测试只验证接口能访问
```

但业务并没有真的闭环。

更具体一点，常见问题包括：

```text
返回空数组，冒充暂无数据
返回固定 0，冒充真实统计
用 hardcode 文案冒充系统信息
接口字段存在，但没有真实持久化
前端页面展示了数据，但数据来源并不可靠
刷新后状态丢失
重启后数据消失
测试只断言 HTTP 200，没有验证业务效果
```

这就是我后来反复提到的一个词：

> **假完成。**

AI 很擅长写出“看起来完整”的代码。

但软件工程真正关心的不是“看起来有没有”，而是：

```text
真实 API 是否被调用？
真实数据是否被写入？
刷新之后是否还在？
重启之后是否还在？
异常情况下是否可解释？
测试是否证明了业务效果？
```

如果这些问题没有答案，那么所谓完成，只是文档意义上的完成。

---

## 3. Phase 最大的问题：有进度感，但没有真实性

传统项目管理里，我们经常把事情拆成 Phase。

比如：

```text
Phase 1：基础框架
Phase 2：核心功能
Phase 3：后台管理
Phase 4：部署上线
```

人类团队使用 Phase 没什么问题，因为每个阶段背后通常还有产品经理、开发、测试、评审、上线流程在兜底。

但 AI Coding 里，如果直接把 Phase 丢给 Agent，风险非常高。

因为 Phase 的抽象层太高。

对 AI 来说，“完成后台管理”可能意味着：

```text
路由有了
页面有了
接口有了
mock 数据能展示
文档写了完成
```

但对真实产品来说，“完成后台管理”意味着：

```text
真实登录
真实权限
真实数据
真实分页
真实搜索
真实错误处理
真实持久化
真实部署后可用
```

这两个“完成”，不是一回事。

所以后来我越来越意识到：

> **Phase 最大的问题是，它很容易制造进度感，却掩盖真实性。**

AI Coding 不能只问：

```text
这个阶段完成了吗？
```

而应该问：

```text
这个业务闭环跑通了吗？
```

---

## 4. 第二次失控：Feature 拆太细，流程成本爆炸

发现 Phase 容易假完成后，我自然走向了另一个方向：  
既然大阶段太虚，那就把功能拆细。

于是我开始把项目拆成一个个 feature。

每个 feature 都要求：

```text
写 prompt
读上下文
写测试
实现代码
跑后端测试
跑前端检查
Docker 验证
浏览器验证
人工验收
更新文档
```

这套逻辑本身没错。

它确实能减少假完成。  
因为每个功能都要有测试证据，都要能在页面上跑通，都要经过人工确认。

但很快，新的问题出现了：

> **流程成本爆炸。**

如果一个项目有十几个、几十个小功能，每个功能都完整走一遍从 prompt 到 Docker、浏览器、人工验收的流程，人很快就会被耗干。

尤其对个人开发者来说，这个成本非常真实。

AI 写代码很快，但你要不断：

```text
切换上下文
读 Agent 总结
看测试结果
看 diff
启动服务
打开浏览器
点页面
确认数据
更新文档
继续下一个 feature
```

最后会出现一种荒诞状态：

> AI 还没累，人先累了。

所以问题变成了：

```text
Phase 太大，会假完成。
Feature 太细，流程成本爆炸。
那中间解是什么？
```

---

## 5. 最危险的错觉：MVP 不是做一个玩具

还有一个更隐蔽的问题：  
当流程变复杂后，很容易有人说：

> 那就做轻量 MVP 吧。

这句话本身没错。  
但在 AI Coding 里，“MVP”特别容易被误解成：

```text
页面能点就行
数据先 mock
异常先不管
部署以后再说
测试以后再补
```

这就危险了。

因为 AI 本来就很擅长生成“看起来完整”的东西。  
如果人再用“先 MVP”作为借口接受这些假实现，项目很快就会滑向玩具化。

我后来对 MVP 的理解变得更严格：

> **生产级 MVP 不是功能很少的玩具，而是范围经过克制的真实系统。**

也就是说：

```text
功能可以少
但留下来的功能必须是真实现
数据必须能持久化
异常必须可解释
部署路径必须可验证
核心路径必须可回归测试
未完成能力必须明确隐藏或标记 deferred
```

比如后台统计第一版可以不做复杂分析、不做实时大屏、不做用户画像。  
但如果它展示“今日访问量”，那这个数字就必须来自真实访问记录，而不是固定 0。

比如通知系统第一版可以只支持站内通知。  
但如果页面展示“未读通知”，那它就必须真的有通知数据、已读状态和持久化。

MVP 的重点不是“糊一个能看的”。  
MVP 的重点是：

> **用最小范围，交付真实能力。**

---

## 6. 多 Agent 协作，也可能制造新的混乱

后来我又尝试了多 Agent 协作。

一个 Agent 负责整体规划、审计、发现假实现风险。  
另一个 Agent 负责当前批次写代码、自验。  
我负责最终打开真实页面确认产品效果。

这个方向是对的。  
它已经很接近一个小型 AI 工程团队：

```text
人类：产品负责人 + 最终验收人
Planner Agent：读代码、出方案、识别风险
Builder Agent：写代码、自验
Reviewer Agent：审 diff、查假实现
CI：机器裁判
浏览器：真实世界检验
```

但多 Agent 并不是天然高效。

如果没有统一事实源，不同 Agent 会开始各写各的计划：

```text
一个写 API 计划
一个写后端计划
一个写修复计划
一个写验收矩阵
一个写阶段日志
```

文档越来越多，决策却越来越不清楚。

更糟糕的是，Agent 自己也会被这些文档搞晕。  
它不知道哪个是最新计划，哪个是旧结论，哪个是历史参考，哪个是当前执行依据。

于是我后来做了一个很重要的调整：

> **文档不是越多越工程化。文档必须服务于执行。**

我开始压缩文档，只保留几类东西：

```text
稳定项目背景
当前唯一执行锁
交付审计记录
必要的工程规则
```

其他历史计划、临时分析、过时矩阵，全部归档。

因为 AI Coding 里，文档太多不是知识资产，很多时候是上下文污染。

---

## 7. 后来我开始使用 Batch 化闭环

在经历了“大 Phase 太虚”和“小 feature 太贵”之后，我逐渐形成了一种更可持续的方式：

> **Batch 化闭环。**

简单说，就是：

```text
一次只做 2-4 个强相关 feature
这些 feature 必须围绕同一个业务目标
每个 feature 都要有最小测试证据
Batch 末尾统一做 Docker、浏览器、人工验收
```

注意，这里的重点不是“2-4”这个数字。  
重点是“强相关”和“业务闭环”。

比如，一个合理的 Batch 可以是：

```text
后台数据真实性
```

它下面可以包含：

```text
访问统计真实化
通知数据真实化
系统信息真实化
```

这些功能共享一个目标：

> 后台不能再展示假数据。

它们可以共用一次上下文、一次联调、一次 Docker 验证、一次人工验收。

再比如：

```text
内容资产闭环
```

它可以包含：

```text
图片上传
静态资源访问
文章图片持久化
导入导出补齐
```

这些功能共同解决的是：

> 内容发布后，图片和附件能不能长期稳定展示。

这就比单个 feature 逐个折磨更高效，也比一个巨大 Phase 更真实。

---

## 8. Batch 不是大号 Phase

这里必须强调一点：

> **Batch 不是大号 Phase。**

如果只是把多个功能打包，然后最后让 AI 说一句“Batch completed”，那它只是换了个名字的 Phase。

真正的 Batch 必须满足几个条件：

```text
1. 有一个明确的业务目标
2. 只包含强相关 feature
3. 每个 feature 有最小测试证据
4. Batch 末尾有端到端验收路径
5. 未完成内容必须明确 deferred
6. 不允许 mock、hardcode、空数组冒充完成
```

换句话说，Batch 的本质不是“多做几个功能”。  
Batch 的本质是：

> **围绕一个业务闭环，降低重复验收成本，同时保留测试和审计能力。**

我后来给 AI Agent 的要求变成了：

```text
开发过程中跑 targeted tests
每个 feature 留下 RED/GREEN 证据
Batch 结束后统一跑全量必要验证
最后再做 Docker、浏览器、人工验收
```

这样既不会让每个小功能都重复消耗完整验收成本，也不会让 AI 在大范围里自由发挥。

## 9. 企业级 AI Coding 的本质：AI 嵌入交付流程

这次实践之后，我对 AI Coding 的理解发生了很大变化。

以前我以为：

```text
AI Coding = 用 AI 替我写代码
```

后来我发现，更成熟的理解应该是：

```text
AI Coding = 把 AI 嵌入软件交付流程
```

企业级做法不会是：

```text
给 AI 一个目标
AI 自己开发完整项目
AI 自己说完成
人类最后看一眼
```

更合理的流程应该是：

```text
Issue
→ Plan
→ Build
→ Test
→ Review
→ CI
→ Docker / Preview
→ UAT
→ Merge
→ Audit
```

AI 可以参与其中很多环节，但不能替代这些环节本身。

比如：

```text
Planner Agent 负责读代码、出方案、识别风险
Builder Agent 负责按方案实现
Test Agent 负责补测试和跑测试
Reviewer Agent 负责查假实现、查技术债、查无关改动
CI 负责冷酷地执行构建、测试、类型检查
人类负责产品判断、架构取舍、最终验收和发布责任
```

这才是更接近企业级的软件工程方式。

AI 不是流程的替代品。  
AI 是流程里的执行者、辅助者和放大器。

## 10. 我现在更认可的分工

经过这次实践，我现在更认可这样的分工。

### 人类：Product Owner + Tech Lead + Release Owner

人类不应该退场。

相反，人类的职责更重了，只是不再主要是敲代码。

人类要负责：

```text
定义业务目标
划定范围
判断优先级
审技术方案
确认产品效果
决定是否合并
决定哪些能力 deferred
承担最终发布责任
```

也就是说，人类不是被 AI 替代，而是从“写代码的人”升级成“设计工作系统的人”。

---

### Planner Agent：先读代码，再出方案

复杂任务不能直接让 Agent 写代码。

更好的方式是：

```text
先读现有代码
识别影响范围
列出涉及文件
说明数据模型影响
说明 API 影响
说明前端影响
说明测试策略
说明风险点
```

没有计划，不进入编码。

这一步非常重要。  
因为 AI 很容易在没有充分上下文的情况下直接开始改代码，然后越改越偏。

---

### Builder Agent：只按当前 Batch 写代码

写代码的 Agent 应该被严格约束。

它只做当前 Batch，不扩展范围，不顺手重构无关模块，不新增一堆计划文档，不随意引入依赖，不碰没有授权的安全和权限逻辑。

它的职责是：

```text
按已批准方案实现
每个 feature 小步提交
每个 feature 跑 targeted tests
记录命令和结果
发现做不了的地方明确说明
```

一句话：

> Builder Agent 是执行者，不是产品经理，也不是架构委员会。

---

### Reviewer Agent：只审计，不写新功能

另一个 Agent 可以专门做审计。

它不负责继续写功能，而是专门检查：

```text
有没有 hardcode
有没有 mock 数据冒充真实实现
有没有只测 HTTP 200
有没有没有持久化
有没有前后端字段不一致
有没有无关改动
有没有绕过原有架构
有没有吞异常
有没有 TODO 留在完成路径里
```

这一步对 AI Coding 特别重要。

因为 AI 写出来的代码，有时候比人写的更“整洁”，但也更容易隐藏假实现。

---

### CI：冷酷裁判

AI 说自己测试过了，不算数。  
必须有命令和结果。

比如：

```text
后端测试通过
前端类型检查通过
构建通过
Docker 配置通过
关键路径 smoke test 通过
```

CI 的价值在于它不理解你的借口。  
能过就是能过，不能过就是不能过。

在 AI Coding 里，CI 不是可选项，而是防止幻觉完成的基础设施。

---

### 浏览器和人工验收：真实世界检验

最后，人类应该打开真实页面。

不是去帮 AI 一个接口一个接口查，而是走真实产品路径：

```text
打开首页
进入详情页
触发一次真实操作
进入后台
确认数据变化
刷新
重启
再次确认数据仍在
检查页面是否符合预期
```

这一步不能省。

因为测试能证明代码逻辑，浏览器才能证明产品体验。

---

## 11. 这次实践给我的最大启发

这次项目让我对 AI Coding 有了一个更清醒的判断：

> **AI Coding 的核心能力，不是 Prompt，而是边界感。**

Prompt 当然重要。  
模型能力也重要。  
工具体验也重要。

但这些都不是最底层的问题。

真正决定 AI Coding 质量的是：

```text
需求是否明确
边界是否清楚
任务是否可验证
测试是否能证明业务效果
文档是否有唯一事实源
审计是否能发现假实现
人类是否保留最终判断
```

AI 会让一个清晰的工程系统跑得更快。  
也会让一个混乱的项目更快地产生混乱。

这就是我这次最大的体感。

---

## 12. AI 没有消灭软件工程，它只是把工程问题暴露得更快

很多关于 AI Coding 的讨论，都容易走向两个极端。

一种极端是：

> AI 太强了，一个人马上可以干掉一个团队。

另一种极端是：

> AI 都是玩具，写不了真实项目。

我的感受更接近中间地带：

> AI 确实能显著提高开发效率，但它没有消灭软件工程。  
> 相反，它要求我们把软件工程做得更显式、更严格、更可验证。

过去很多工程问题，是靠团队经验、Code Review、测试流程、上线制度慢慢兜住的。

但在 AI Coding 里，如果你没有这些东西，AI 会更快地把问题放大：

```text
边界太宽，AI 会幻觉完成
边界太窄，人会被流程耗死
测试太少，假实现会上线
文档太多，Agent 会迷路
验收太频繁，人会崩
范围砍太狠，项目会变成玩具
```

所以，真正成熟的 AI Coding，不是“相信 AI”，也不是“不相信 AI”。

而是：

> **把 AI 当成很快但需要约束的工程队，把自己从写代码的人，升级成设计工作系统的人。**

---

## 13. 写在最后

如果说这次项目有什么真正的收获，我觉得不是某个模型更强，也不是某个工具更好用。

真正的收获是我开始意识到：

> **未来开发者的核心能力，不只是会不会写代码，而是能不能设计一套让 AI 稳定产出可信软件的工作系统。**

这个系统里，需要有：

```text
清晰的 Issue
先行的 Plan
受控的 Build
真实的 Test
独立的 Review
冷酷的 CI
可运行的 Docker
真实的浏览器验收
诚实的 Delivery Audit
```

AI Coding 的生产力上限，不取决于 AI 多会写代码。  
而取决于人类能不能定义清楚：

> 什么叫完成。

这件事不轻松。  
但它可能正是 AI 真正改变软件开发方式的地方。
